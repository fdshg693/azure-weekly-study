# ルール

採用言語

- スクリプト言語には、python, nodejs を使用。どちらでもいい場合は、簡単な方を選ぶ。
    - python は仮想環境を利用
        - 重複インストールをさけるために、ルート直下の`.venv`の仮想環境を各プロジェクトで共有する形をとる
        - 依存関係の要求を明確にするために、各プロジェクトには`requirements.txt`を置く
        - 軽い依存のみの場合、何かしらでプロジェクトごとに仮想環境を作った方が都合のいい場合は、プロジェクトごとに仮想環境を作成する可能性もある
- 各プロジェクトにはjustfileをおいて、実行を簡略化する
    - justfileは行をまたいで変数を保持できない、パワーシェルの改行がしにくい等の巨大ワンライナーを作りがちな側面がある
    - そこで、多少なりともプロジェクトが複雑になる場合は、justfileではなくTaskfileを利用すること

コード品質
- 基本的にコメント等を付けて、難しいコードの理解を助ける
- 環境変数からの値取得に加えて、シンプルな`.env`からの値取得もサポートする（当然GITIGNORE済み）
    - 対応する`.env.example`も用意する

ガードレール
- 明示的な指示がない限り、実際のデプロイは行わない（コードの作成とローカルでの動作確認までを行う）
    - ユーザーからの明示的な指示があった場合
    - 読み取り系のコマンドは比較的害が少ないので、原因調査や複雑なプロジェクト作成中の確認としては実行してもよい
- 学習目的プロジェクトの場合は、ただ一度デプロイして終わりではなく、コマンドを叩いて操作すると結果が様々に変化することが体験するところまでやる。
    - 例えば、認証フローのプロジェクトであれば、Azコマンドでユーザーの権限を操作して、それによってAPIのレスポンスが変わることを体験するなど

ファイル・フォルダ構成
- 明示的な指示がない限り以下が理想
```markdown
## 各プロジェクト共通ファイル構成

細かく分ければよいわけではなく、可読性・理解しやすさを最優先に分割する。

- `justfile`：アプリ登録・設定生成・ローカル実行などのコマンドをまとめる
    - プロジェクトが複雑になる場合は、justfileではなくTaskfile(Taskfile.yml等)を利用すること
    - 無理なワンライナーを埋め込まず、ファイル分割・powershellスクリプト分離等を活用して、可読性を保つこと
- `README.md`：このプロジェクトで行う内容・手順・学習の流れ
- `KNOWLEDGE.md`：このプロジェクトで新たに出た用語・概念（前のプロジェクトでカバー済みの語は含めない）
- `PLAN.md`（任意）：そのプロジェクト固有の設計（大きめのプロジェクトのみ、かつユーザーが明示的に指示した場合に作成）
- `MERMAID.md`（任意）：認証フロー／構成を mermaid で表現
- `*.bicep`: 内容に応じて、環境構築のためのBicepファイルを配置。適切なファイル分割を行い、可読性を保つこと
```

ドキュメント管理
- **以下の内容が書かれていることを念頭におき、必要に応じて確認・更新すること**
- `learn/CLAUDE.md`には、各プロジェクトごとの概要を短く書き（あまりに1トピック配下のサブフォルダが多ければ適当に抽出していい。ただし、略記しているむねを必ず書く）、CLAUDEがユーザーのAzureのレベル感を把握しやすいようにする
- 各トピック配下の`learn/{topic}/CLAUDE.md`には、トピックごとのプロジェクトについてより踏み込んで書くき、次のプロジェクト作成の目安とする
    - 学習した概念、学習できていない概念はなにか
    - `learn/CLAUDE.md`以上に詳しい、そのトピックに限定したユーザーのレベル感を把握できる内容を書く
