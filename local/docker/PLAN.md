# docker — ステップアップ学習（PLAN）

Docker の**概念（イメージ／コンテナ／レイヤ）と基本コマンド（`build`/`run`/`exec`/`ps`/`logs`/`rm`）は
すでに分かっている中級者**を対象に、「動かせる」から **「中身を理解して作り込む・壊して原因を切り分ける」**
へ引き上げるための次ステップ候補。

このフォルダは `learn/`（Azure 前提）と違い、**ローカルだけで完結**する（クラウド・課金不要、Docker Engine
さえあれば全ステップ実行可能）。クラウド固有の話には踏み込まず、まず Docker / コンテナ一般を深掘りする。

> **責務分け（重複回避）**: ここは **コンテナ単体をローカルで深掘り**する担当（イメージの作り込み・ランタイム
> 挙動・隔離の仕組み）。**作ったイメージを Azure マネージドに載せて動かす**のは
> [../../learn/container/PLAN.md](../../learn/container/PLAN.md)（ACI / Web App for Containers / Container Apps の App 側 / ACR）、
> **フルオーケストレーション（AKS）**は [../../learn/k8s/PLAN.md](../../learn/k8s/PLAN.md)、
> **起動して終了するバッチ（Container Apps Job 等）**は [../../learn/automate/CLAUDE.md](../../learn/automate/CLAUDE.md) が担当。
> ＝ このフォルダは **Azure には出さない**。

## このフェーズの方針

`learn/network` の学び方を踏襲する。

- **まず一般概念 → 実装 → 設定を出し入れして因果を確かめる。** 「なんとなく動いた」で止めず、
  スイッチを on/off して**挙動が変わることから因果を切り分ける**のを各ステップの核に据える。
  （例：`--memory` を絞る → OOM で kill される → 緩める → 生き残る、を観測）
- **1 ステップ＝1 概念にこだわらない。** 基礎がある前提なので、関連する複数概念を 1 ステップに
  まとめてよい（例：「PID 1・シグナル・ゾンビ・`ENTRYPOINT`/`CMD`」を 1 ステップで扱う）。
- **構築・実行はユーザー自身が行う。** AI はコード・手順を用意するが、コンテナの起動や破壊は手元で叩いて体感する。

各ステップ共通の最低限のファイル構成は `learn` ルールに準じる（`justfile` で操作をまとめ、`README.md` で
手順と学習の流れ、`KNOWLEDGE.md` で新出概念。図が要るものは `MERMAID.md`）。

## 案一覧

各案に **レベル（★＝中級者が入りやすい／★★＝中／★★★＝発展）**、**解く問題（一般）**、**実現（Docker での具体）**、
**1 ステップに込める概念**、**出し入れ検証**を併記する。番号は優先度ではなく識別用。

---

### 案1. イメージを小さく・速く作る ―― レイヤキャッシュとマルチステージビルド

- **レベル**：★
- **解く問題（一般）**：ビルドは毎回フルでやり直すと遅く、成果物に**ビルドにしか要らない物（コンパイラ・
  開発依存）が混入**して肥大化する。「キャッシュが効く粒度」と「実行に必要な物だけを残す」を分けて考える。
- **実現（Docker）**：`Dockerfile` の命令順を変えて**レイヤキャッシュのヒット／ミス**を観測（依存インストールを
  ソースコピーより前に置く定番）。`multi-stage build` でビルド用ステージと実行用ステージを分け、`--from` で
  成果物だけを最終イメージへコピー。`.dockerignore` でビルドコンテキストを絞る。
- **1 ステップに込める概念**：レイヤとキャッシュ無効化の伝播・ビルドコンテキスト・マルチステージ・
  `slim`/`distroless` 等のベース選び。
- **出し入れ検証**：① ソースを 1 行変えて再ビルドし、`COPY` の位置によって**再ビルドされるレイヤ数が変わる**ことを
  確認。② マルチステージの有無で `docker images` の **SIZE が数百 MB → 数十 MB** に変わるのを比較。

### 案2. プロセスとしてのコンテナ ―― PID 1・シグナル・`ENTRYPOINT`/`CMD`

- **レベル**：★★
- **解く問題（一般）**：コンテナは「軽量 VM」ではなく**1 プロセスのラッパ**。だから「`Ctrl-C`／`docker stop` で
  すぐ止まらない」「子プロセスがゾンビになる」といった現象が起きる。**PID 1 の責務**（シグナル処理・子の回収）を理解する。
- **実現（Docker）**：`ENTRYPOINT`/`CMD` の **shell 形式 vs exec 形式**で PID 1 が `/bin/sh` になるか目的プロセスに
  なるかを `docker exec ... ps` で確認。`docker stop`（SIGTERM → 10 秒後 SIGKILL）でアプリが grace に落ちるか観測。
  `--init`（tini）でゾンビ回収を体感。
- **1 ステップに込める概念**：PID 1・SIGTERM/SIGKILL・10 秒の猶予（`--stop-timeout`）・exec 形式の重要性・
  ゾンビプロセスと init。
- **出し入れ検証**：① shell 形式 → exec 形式に変えると `docker stop` の**所要時間が 10 秒 → 即時**に変わる
  （シグナルがアプリに届くようになる）。② `--init` の有無でゾンビプロセスが残る／回収されるを `ps` で比較。

### 案3. リソース制御 ―― cgroups（CPU／メモリ）と OOM

- **レベル**：★★
- **解く問題（一般）**：1 ホストに同居するコンテナが互いの CPU・メモリを食い合わないよう**上限を課す**。
  上限に当たったとき**何が起きるか**（メモリは kill、CPU は絞られる、の非対称）を理解する。
- **実現（Docker）**：`--memory` / `--memory-swap` / `--cpus` / `--cpu-shares` を付けて `stress`（または負荷スクリプト）を
  走らせ、`docker stats` でリアルタイムに使用量を観測。メモリ超過時の **OOM kill**（`docker inspect` の
  `OOMKilled: true`、exit code 137）を確認。
- **1 ステップに込める概念**：cgroups・ハードリミット（memory）vs ベストエフォート（cpu shares）・OOM・
  exit code 137 の読み方。
- **出し入れ検証**：① `--memory 128m` で確保量を増やすプロセスを走らせ **OOM kill**、`--memory 512m` で生存、を切り替え。
  ② `--cpus 0.5` を付け外しして同じ計算負荷の**所要時間が倍**になることを `time` で確認。

### 案4. コンテナのネットワーク ―― bridge / host / custom network・名前解決・publish

- **レベル**：★★
- **解く問題（一般）**：複数コンテナを**どう繋ぎ／どう隔離し**、外部からどう届かせるか。「ポートを公開する」と
  「コンテナ同士が名前で見つける」は別レイヤの話であることを切り分ける。
- **実現（Docker）**：default bridge と**ユーザー定義ネットワーク**の違い（後者は**コンテナ名で DNS 解決**できる）を
  `docker network create` して確認。`-p` の publish（ホスト↔コンテナ）と、ネットワーク内通信を区別。`--network host`、
  別ネットワークに分けて**到達できないこと**を観測。`docker network inspect` でサブネット／所属を確認。
- **1 ステップに込める概念**：bridge/host/none・組み込み DNS とサービス検出・publish と NAT・ネットワーク分離による隔離。
- **出し入れ検証**：① 2 コンテナを同一ユーザー定義ネットワークに置くと**名前で ping/curl 到達**、別ネットワークに
  分けると**名前解決できず不達**、を切り替え。② `-p` を付け外しして**ホストからの到達**だけが変わる（コンテナ間通信は不変）ことを確認。

### 案5. データの永続化 ―― bind mount / named volume / tmpfs

- **レベル**：★
- **解く問題（一般）**：コンテナのファイルシステムは**使い捨て**。消えてほしくないデータ（DB）と、共有したい
  ソース、消えてよい一時データを**置き場所で区別**する。
- **実現（Docker）**：`-v $(pwd):/app`（bind mount＝ホストのパスを直結）と `-v data:/var/lib/...`（named volume＝Docker 管理）と
  `--tmpfs`（メモリ上）の 3 種を比較。コンテナを `rm` → 再作成して**データが残る／消える**を観測。`docker volume ls/inspect`。
- **1 ステップに込める概念**：書き込み可能レイヤの揮発性・bind mount vs named volume の使い分け・
  マウントによる隠蔽（マウント先の元ファイルが見えなくなる）・所有権/権限の落とし穴。
- **出し入れ検証**：① named volume にファイルを書く → `docker rm` → 同じ volume で再 `run` し**残存**を確認。
  volume 無しなら**消える**を対比。② ソースを bind mount すると**ホストの編集が即コンテナへ反映**されることを確認。

### 案6. 複数コンテナをまとめる ―― Docker Compose・依存・ヘルスチェック

- **レベル**：★★
- **解く問題（一般）**：`docker run` を手で何回も打つのは破綻する。**構成を宣言的に記述**し、起動順や
  「相手が本当に受け付け可能になるまで待つ」を扱う。
- **実現（Docker）**：`compose.yaml` に app＋db を定義。`depends_on` と **`HEALTHCHECK` / `condition: service_healthy`** で
  「コンテナが起動した」と「アプリが ready」を区別。`docker compose up/down`、`--scale` で複製、`compose` のネットワーク・
  volume が自動で作られることを確認。案4・案5 の宣言版。
- **1 ステップに込める概念**：宣言的構成・サービス間 DNS（サービス名で解決）・healthcheck と起動順序・
  環境変数／`.env` による設定の外出し（`learn` ルールの `.env`/`.env.example` に準拠）。
- **出し入れ検証**：① db の healthcheck を壊す（無効なコマンドに）と app が**起動を待ち続ける／失敗**、直すと**順に立ち上がる**を切り替え。
  ② `--scale app=3` で app だけ 3 つに増やし、`compose ps` で確認。

### 案7. イメージとレジストリの中身 ―― レイヤ・digest・ローカルレジストリ

- **レベル**：★★
- **解く問題（一般）**：`tag`（人間用の可変ラベル）と **`digest`（内容ハッシュ＝不変の同一性）** の違い、
  イメージが**レイヤの集合＋マニフェスト**であることを理解し、「同じタグなのに中身が違う」事故を避ける。
- **実現（Docker）**：`docker history` でレイヤ構成、`docker image inspect` で digest／`RootFS` レイヤ、`docker save` で
  中身を展開して観察。`registry:2` を**ローカルで起動**して `push`/`pull` し、`tag` を打ち替えても digest は不変なことを確認。
- **1 ステップに込める概念**：イメージマニフェストとレイヤの content-addressable 性・tag vs digest・
  ローカルレジストリ・レイヤ共有によるディスク節約。
- **出し入れ検証**：① 同じイメージを別タグで push しても `digest` が一致することを確認（tag は別名にすぎない）。
  ② 1 レイヤだけ変えて再 push し、**変わっていないレイヤは転送されない**（`Layer already exists`）ことを観測。

### 案8. BuildKit を使いこなす ―― cache mount・secret mount・マルチプラットフォーム

- **レベル**：★★★
- **解く問題（一般）**：素朴なビルドは**依存ダウンロードを毎回やり直し**、ビルド時の秘密情報が**レイヤに焼き込まれ**、
  成果物が**1 アーキテクチャ専用**になりがち。これらをビルドエンジンの機能で解く。
- **実現（Docker）**：`# syntax=docker/dockerfile:1` ＋ BuildKit で、`RUN --mount=type=cache`（パッケージキャッシュを
  レイヤに残さず再利用）、`RUN --mount=type=secret`（ビルド時 secret をイメージに残さず使用）、`docker buildx` で
  `--platform linux/amd64,linux/arm64` のマルチアーキビルド。
- **1 ステップに込める概念**：BuildKit の並列・キャッシュ・cache mount／secret mount／SSH mount・buildx と
  マルチプラットフォーム・「秘密をレイヤに残さない」原則。
- **出し入れ検証**：① cache mount の有無で**2 回目以降のビルド時間**が大きく変わるのを `time` で比較。
  ② `--mount=type=secret` で渡した値が、最終イメージの `docker history`／レイヤ展開に**現れない**ことを確認（`ARG` で渡すと焼き込まれるのと対比）。

### 案9. セキュリティの作り込み ―― 非 root・capability・read-only・スキャン

- **レベル**：★★★
- **解く問題（一般）**：コンテナはデフォルトだと**root で動き、過剰な権限**を持ち、FS も書き換え可能。
  「最小権限」をコンテナにも適用し、侵害時の被害を絞る。
- **実現（Docker）**：`USER` で非 root 化、`--cap-drop ALL`＋必要な `--cap-add` だけ、`--read-only`＋`--tmpfs` で
  書き込み面を限定、`--security-opt no-new-privileges`。`docker scout`（または `trivy`）でイメージの既知脆弱性を
  スキャンし、ベース更新で減ることを確認。
- **1 ステップに込める概念**：root/非 root・Linux capabilities・read-only rootfs・特権の昇格防止・
  イメージ脆弱性スキャンとベース選択。
- **出し入れ検証**：① 非 root ユーザーで `/etc` に書こうとして**拒否**、root だと**書ける**を対比。
  ② `--cap-drop ALL` で `ping`（要 NET_RAW）が**失敗**、`--cap-add NET_RAW` で**成功**を切り替え、必要 capability を切り分ける。

### 案10. 隔離の仕組みを覗く ―― namespace と cgroup を生で観察

- **レベル**：★★★
- **解く問題（一般）**：これまでの全ステップを支える土台。「コンテナ＝隔離されたプロセス」が、Linux の
  **namespace（見える範囲を分ける）** と **cgroup（使える量を分ける）** という 2 つの仕組みでできていることを、
  抽象でなく**実物**で確認する。案2〜5・案3 の裏側を一気に見る総まとめ。
- **実現（Docker）**：コンテナ内 `ps`／`ip addr`／`hostname` がホストと違う＝PID/Net/UTS namespace の分離を確認。
  `--pid host` でホストのプロセスが見える＝namespace 共有を対比。`/sys/fs/cgroup` でメモリ／CPU の上限値を直接読む。
  `unshare` / `nsenter` で namespace を手で作る／入る最小実験（任意）。
- **1 ステップに込める概念**：namespace（pid/net/mnt/uts/ipc/user）・cgroup と階層・「隔離は機能の集合で
  あって魔法ではない」という見方・`--pid host`/`--net host` が隔離を外す意味。
- **出し入れ検証**：① `--pid host` の有無で、コンテナ内 `ps` に**ホストのプロセスが見える／見えない**を切り替え。
  ② 案3 で付けた `--memory` の値が `/sys/fs/cgroup/.../memory.max` に**そのまま反映**されていることを確認（制限の実体を見る）。

## おすすめの進め方

依存関係と入りやすさを踏まえ、3 グループに分けて進めるのがおすすめ。番号順に強い前後依存はないが、
各グループ内は前から積むと理解が滑らか。

1. **ランタイム挙動グループ（★〜★★）**：案5（永続化）→ 案2（プロセスとシグナル）→ 案3（リソース制御）→ 案4（ネットワーク）
   - 「コンテナを 1 つ動かしたときに実際に何が起きているか」を体で覚える土台。`docker stats`／`inspect`／`exec ps` で
     挙動を覗く癖をここで付ける。

2. **ビルドグループ（★〜★★★）**：案1（マルチステージ／キャッシュ）→ 案7（イメージとレジストリの中身）→ 案8（BuildKit）
   - 「良いイメージを速く・安全に作る」一連。案1 で素朴な最適化、案7 で中身の理解、案8 でエンジン機能まで。

3. **統合・発展グループ（★★〜★★★）**：案6（Compose で複数コンテナ）→ 案9（セキュリティの作り込み）→ 案10（namespace/cgroup を覗く）
   - 複数コンテナの宣言的運用で実戦化し、最小権限で締め、最後に案10 で「全部の裏側は namespace と cgroup だった」と
     回収する仕上げ。

> どの案も「まず一般概念 → Docker での実装 → 設定を出し入れして因果を確かめる」という `learn/network` 流の
> 学び方を踏襲する。すべてローカルで完結し、構築・実行はユーザー自身が行う。
