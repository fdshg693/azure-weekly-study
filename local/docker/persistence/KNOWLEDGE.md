# KNOWLEDGE — persistence で新たに出た用語・概念

このステップで初めて出てきた語のみ。Docker の基本（image / container / `run` / `rm` 等）は既知前提。

## 書き込み可能レイヤ（writable / container layer）

イメージは読み取り専用レイヤの積み重ね。コンテナはその一番上に**自分専用の薄い書き込み層**を載せて動く。
コンテナ内での新規作成・変更はこの層に書かれ、**コンテナを削除すると一緒に消える**。「残したいデータは
この層の外（volume / bind mount）に出す」がデータ永続化の出発点。

## bind mount（バインドマウント）

**ホスト上の任意のパス**を、コンテナ内のパスに直結する仕組み（`-v ホスト絶対パス:コンテナパス`）。
ホストのファイルをそのまま見るので、ホスト側の編集が**コンテナ再起動なしに即反映**される。
ソース共有・設定ファイルの流し込みに向く。中身の同一性はホストのパスに依存する。

## named volume（名前付きボリューム）

**Docker が管理する保存領域**（`-v ボリューム名:コンテナパス`）。保存先は Docker の管理下
（`docker volume inspect` の `Mountpoint`）で、ホストの特定パスに縛られない。コンテナを消しても残るので、
**DB など残したいデータ**の標準的な置き場所。空のボリュームを初めて空でないパスにマウントすると、
イメージ側のそのディレクトリの中身が**コピーで初期化**される（bind mount との違い）。

## tmpfs マウント

`--tmpfs コンテナパス` で、そのパスを**メモリ上のファイルシステム**にする。ディスクに書かれず、
コンテナ終了で完全に消える。機密の一時データや、ディスクに残したくない高速スクラッチ用。

## マウントによる隠蔽（mount masking / shadowing）

あるパスに volume や bind mount を被せると、**イメージに元々あったそのパスの中身は隠れて見えなくなる**。
本ステップでは、`./site` を nginx の公開ディレクトリに被せると、イメージ既定のページが隠れることで体感する。
「設定したはずのファイルが見えない」事故の典型原因。

## ボリュームの所有権問題（permission pitfall）

空の named volume を初めてマウントすると、その階層は**コンテナ内の所有者（既定 root）**で作られる。
非 root ユーザーで動くコンテナはここに書き込めず `Permission denied` になる。対処は、初回に root で
`chown` する／イメージ側で所有者を合わせる／ホスト側のディレクトリ権限を整える、など。原因は
**権限**であり、ボリューム破損ではない点を切り分けられることが大事。

## コンテナの寿命とデータの寿命の分離

このステップ全体を貫く考え方。`docker rm` でコンテナを使い捨てても、データを残したいなら
書き込みレイヤの外（volume / bind mount）へ出す。「使い捨てるもの」と「残すもの」を**置き場所で**分ける。
