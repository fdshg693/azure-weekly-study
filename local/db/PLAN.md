# db（ローカル）— ステップアップ学習（PLAN）

**RDB の基本コマンド（`CREATE TABLE` / `INSERT` / `SELECT` / `JOIN` / 単純な `WHERE`・`GROUP BY`）は
すでに書ける中級者**を対象に、「クエリが書ける」から **「本番で壊さず・遅くせず・守って運用できる」**
へ引き上げるための次ステップ候補。基本的な SQL の文法・CRUD はスキップする。

このフォルダは `learn/`（Azure 前提）と違い、**ローカルだけで完結**する（クラウド・課金不要、Docker Engine
さえあれば全ステップ実行可能）。**DB エンジンは PostgreSQL に統一**する（リポジトリ全体が PostgreSQL で
揃っており、k8s `workload-identity` や `learn/db` の話と地続きになる）。

> 関連: **マネージド DB（PaaS）側**は [../../learn/db/PLAN.md](../../learn/db/PLAN.md)。あちらは「運用を Azure に
> 任せると何が制約になるか」を学ぶ。**こちらのローカル db は逆向き**——マネージドが裏で握っていて触れない
> ノブ（`autovacuum` の設定・分離レベル・レプリケーション・WAL・ロール設計）を**自分で回して因果を見る**。
> 「PaaS で隠れているもの」を生で触る場所、という位置づけ。

## このフェーズの方針

`local/docker/PLAN.md` および `learn/network` の学び方を踏襲する。

- **まず一般概念 → 実装 → 設定を出し入れして因果を確かめる。** 「なんとなく速い／動いた」で止めず、
  スイッチを on/off して**挙動が変わることから因果を切り分ける**のを各ステップの核に据える。
  （例：インデックスを外す → `Seq Scan` に落ちて実行時間が跳ねる → 貼る → `Index Scan` に戻る、を `EXPLAIN ANALYZE` で観測）
- **1 ステップ＝1 概念にこだわらない。** 基礎がある前提なので、関連する複数概念を 1 ステップに
  まとめてよい（例：「分離レベル・MVCC・ロック・デッドロック」を 1 ステップで扱う）。
- **構築・実行はユーザー自身が行う。** AI はコード・手順・検証用のデータ投入スクリプトを用意するが、
  コンテナの起動や破壊・負荷の投入は手元で叩いて体感する。
- **本番想定のデータ量を作る。** 性能・運用の話は数十行のテーブルでは何も起きない。各ステップで
  `generate_series` 等で**数十万〜数百万行**を投入してから実験する（用意スクリプトに含める）。

各ステップ共通の最低限のファイル構成は `learn` ルールに準じる（`compose.yaml` ＋ `justfile`/`Taskfile.yml` で
操作をまとめ、`README.md` で手順と学習の流れ、`KNOWLEDGE.md` で新出概念、初期データ投入用の `seed.sql` 等）。
DB 接続情報などは `.env`/`.env.example` で外出しする（`learn` ルール準拠）。

## 案一覧

ユーザー指定の 4 領域 **パフォーマンス / DB 設計 / ロール制御 / 障害対応** に沿って並べる。
各案に **レベル（★＝中級者が入りやすい／★★＝中／★★★＝発展）**、**解く問題（一般）**、**実現（PostgreSQL での具体）**、
**1 ステップに込める概念**、**出し入れ検証**を併記する。番号は優先度ではなく識別用。

## A. パフォーマンス

### 案1. インデックスと実行計画 ―― `EXPLAIN ANALYZE`・Seq Scan vs Index Scan
- **レベル**：★
- **解く問題（一般）**：同じ結果を返すクエリでも、**データの探し方**で速度が桁違いになる。「DB が実際に
  どう探しているか」を実行計画で読み、インデックスが**効く条件／効かない条件**を切り分ける。
- **実現（PostgreSQL）**：数百万行のテーブルに対し `EXPLAIN (ANALYZE, BUFFERS)` でプランと実測時間・読んだ
  ブロック数を観察。B-tree インデックスを `CREATE INDEX` で貼る／`DROP` する。**複合インデックスの列順**、
  **カバリングインデックス（`INCLUDE`）**、`LIKE 'foo%'` は効くが `'%foo'` は効かない、関数を噛ませると効かない、を比較。
- **1 ステップに込める概念**：実行計画の読み方・Seq Scan / Index Scan / Bitmap Scan・選択性（カーディナリティ）・
  複合インデックスの先頭列規則・インデックスは**読みに効き書きに対してはコスト**。
- **出し入れ検証**：① インデックス有無で同一クエリの `EXPLAIN ANALYZE` の **node が Seq Scan → Index Scan に変わり
  実行時間が跳ねる／戻る**。② 複合インデックス `(a, b)` に対し `WHERE a=…` は効くが `WHERE b=…` 単独では効かないことを確認。

### 案2. 統計情報・VACUUM・テーブル肥大化（bloat）
- **レベル**：★★
- **解く問題（一般）**：「昨日まで速かったクエリが急に遅い」。原因は**プランナの統計が古い**か、更新/削除の
  繰り返しで**不要タプルが溜まって肥大化**したか。MVCC を持つ DB 特有の運用課題を理解する。
- **実現（PostgreSQL）**：大量 `UPDATE`/`DELETE` を流し、`pg_stat_user_tables` の `n_dead_tup` と `pg_relation_size` で
  bloat を観測。`autovacuum` を**わざと止め**て肥大化を再現 → `VACUUM` / `VACUUM FULL` / `ANALYZE` で回収・統計更新。
  `ANALYZE` 前後でプランが変わる様子も見る。
- **1 ステップに込める概念**：MVCC と dead tuple・autovacuum の役割と閾値・`VACUUM` と `VACUUM FULL`（ロック差）・
  統計情報とプランナ・**マネージド DB が裏で回しているもの**の正体。
- **出し入れ検証**：① `autovacuum = off` で大量更新 → `n_dead_tup` とサイズが膨張、`VACUUM` で回収を観測。
  ② 大量投入直後（統計が古い）と `ANALYZE` 後でプランが変わる（行数見積りが正される）ことを `EXPLAIN` で比較。

### 案3. 接続数とコネクションプール
- **レベル**：★★
- **解く問題（一般）**：DB の接続は**1 本ごとにメモリ・プロセスを食う有限資源**。アプリが素朴に接続を張ると
  **接続あふれ**で全体が落ちる。「アプリ側で接続を使い回す／前段でプールする」という運用の定番を理解する。
- **実現（PostgreSQL）**：`max_connections` を小さく設定 → 並行クライアントを増やして `FATAL: too many connections` を
  再現。**PgBouncer**（transaction pooling）をコンテナで前段に置き、少ない実接続で多数クライアントを捌けることを確認。
- **1 ステップに込める概念**：接続のコスト・`max_connections` と `superuser_reserved_connections`・
  プーリング（session/transaction/statement）・`pg_stat_activity` で接続状態を見る。
- **出し入れ検証**：① `max_connections` を絞る／緩めると、同時接続を増やしたとき**落ちる本数の閾値が変わる**。
  ② PgBouncer 経由と直結で、同じ並行数を掛けたときの**実接続数（`pg_stat_activity` の件数）が激減**することを比較。

## B. DB 設計

### 案4. 制約とデータ整合性 ―― FK・UNIQUE・CHECK・NOT NULL・正規化
- **レベル**：★
- **解く問題（一般）**：「アプリ側で気をつける」だけでは不整合データは必ず混入する。**DB が最後の砦として
  不正なデータを拒否する**よう設計する。正規化で重複を消し、制約で守る範囲を決める。
- **実現（PostgreSQL）**：注文/顧客/商品のような複数テーブルを設計。`FOREIGN KEY`・`UNIQUE`・`CHECK`・
  `NOT NULL`・`ON DELETE CASCADE/RESTRICT` を付け外しして、不正データの投入が**拒否される／通ってしまう**を対比。
  非正規（1 テーブルに詰め込み）vs 正規化での更新異常も見せる。
- **1 ステップに込める概念**：第 1〜3 正規形・参照整合性・各制約の役割・`ON DELETE` の挙動差・
  「制約はバグを**書き込み時に**止める」。
- **出し入れ検証**：① FK を外すと**親のない子レコード**が入る、付けると拒否される。`ON DELETE` を
  `RESTRICT`↔`CASCADE` で親削除の挙動が変わる。② `CHECK (price >= 0)` の有無で負の価格が通る／弾かれるを切り替え。

### 案5. トランザクション分離レベルと並行制御 ―― MVCC・ロック・デッドロック
- **レベル**：★★★
- **解く問題（一般）**：本番は**複数トランザクションが同時に**走る。「片方の更新がもう片方を上書きする」
  「読んだ値が途中で変わる」などの並行性異常を、**分離レベル**という設定で制御することを理解する。
- **実現（PostgreSQL）**：**2 つの `psql` セッション**を並べ、`READ COMMITTED`（既定）と `REPEATABLE READ` /
  `SERIALIZABLE` で同じ手順を流して結果差を観測。non-repeatable read・lost update・`SERIALIZABLE` の
  `could not serialize` エラーを再現。`SELECT … FOR UPDATE` の行ロック、わざとデッドロックを作って自動検出を見る。
- **1 ステップに込める概念**：ACID・分離レベル 4 種と防げる異常・MVCC とスナップショット・
  行ロック / `FOR UPDATE`・デッドロック検出・楽観 vs 悲観ロック。
- **出し入れ検証**：① 同じ並行手順を `READ COMMITTED`↔`SERIALIZABLE` で実行 → 一方は静かに上書き、もう一方は
  **シリアライズ失敗で abort**、と挙動が変わる。② 2 セッションが互いの行を逆順にロック → **デッドロックを DB が検出して片方を kill** するのを観測。

### 案6. WAL・チェックポイント・耐久性のトレードオフ
- **レベル**：★★★
- **解く問題（一般）**：「コミットした＝ディスクに確実に残った」を保証する仕組み（**WAL**）と、その保証を
  **どこまで緩めると速くなり、何を失うか**を理解する。性能と耐久性は地続きであることを掴む。
- **実現（PostgreSQL）**：`synchronous_commit` を `on`↔`off`、`fsync` を on↔off、`wal_level`・`checkpoint_timeout` を
  変えて、大量 INSERT のスループットを `\timing` で比較。`pg_stat_wal` / チェックポイントログで WAL 量を観察
  （クラッシュ時のデータロスは「理屈として何が危ういか」を言語化、案7 のバックアップで実際に守る）。
- **1 ステップに込める概念**：WAL（先行書き込みログ）・チェックポイント・`synchronous_commit` と
  グループコミット・耐久性 vs スループットのトレードオフ・**この WAL が案8 のレプリケーションと案7 の PITR の土台**。
- **出し入れ検証**：① `synchronous_commit = off` にすると同じ INSERT 負荷の**スループットが上がる**が、引き換えに
  「クラッシュで直近コミットを失いうる」ことを整理。② チェックポイント間隔を変えると WAL 生成・I/O の山が変わるのを観察。

## C. ロール制御

### 案7. ロールと権限設計 ―― 最小権限・GRANT/REVOKE・行レベルセキュリティ
- **レベル**：★★
- **解く問題（一般）**：アプリが**管理者権限で DB に繋ぎっぱなし**は事故の元。役割ごとに**最小権限**の
  ロールを切り、「読むだけ」「特定スキーマだけ」「自分の行だけ」を DB 側で強制する。
- **実現（PostgreSQL）**：`read_only` / `app_rw` / `admin` などのロールを作り、`GRANT`/`REVOKE` を
  スキーマ・テーブル・列レベルで付け外し。読み取り専用ロールで `INSERT` → 拒否、付与 → 通る。
  **Row Level Security（`ENABLE ROW LEVEL SECURITY` + `POLICY`）** でテナント ID により**同じクエリでも見える行が変わる**様子を確認。
- **1 ステップに込める概念**：ロールと継承（`GRANT role TO role`）・オブジェクト権限と `DEFAULT PRIVILEGES`・
  最小権限の原則・Row Level Security とマルチテナント・`SECURITY DEFINER` 関数の注意。
- **出し入れ検証**：① 読み取り専用ロールに `INSERT` 権限を付け外し → 同じ INSERT が**通る／拒否**を切り替え。
  ② RLS の `POLICY` を on/off ＋ `current_setting` のテナント値を変えると、**同一の `SELECT *` が返す行集合が変わる**ことを確認。

## D. 障害対応

### 案8. バックアップとリカバリ ―― 論理ダンプ・PITR（WAL アーカイブ）
- **レベル**：★★
- **解く問題（一般）**：「消してしまった」「壊れた」に**戻せること**が運用の最後の砦。スナップショット的な
  ダンプと、**任意時刻に巻き戻す PITR** の使い分けを理解する。
- **実現（PostgreSQL）**：`pg_dump`/`pg_restore` で論理バックアップ → 一部データを破壊 → 別 DB に復元して差分確認。
  `archive_command` で WAL をアーカイブし、`pg_basebackup` + WAL リプレイで**特定時刻へ PITR**（`recovery_target_time`）。
  「DROP TABLE した 1 分前」に戻す、を実演。
- **1 ステップに込める概念**：論理 vs 物理バックアップ・base backup と WAL アーカイブ・PITR と
  `recovery_target_time`・RPO/RTO の感覚・**案6 の WAL がここで効く**。
- **出し入れ検証**：① テーブルを `DROP` → ダンプから復元して**戻る**ことを確認。② WAL アーカイブを有効化し、
  破壊操作の**直前の時刻**を `recovery_target_time` に指定して復元 → 破壊が「なかったこと」になるのを観測（保持 WAL が無いと戻せない、も対比）。

### 案9. レプリケーションとフェイルオーバー ―― ストリーミングレプリケーション・読み取りスケール・昇格
- **レベル**：★★★
- **解く問題（一般）**：単一 DB は**そのサーバーが死ぬと全停止**。複製を常に追従させ、障害時に**昇格**して
  継続する／読み取りを複製に逃がして**スケール**する、という冗長化の基本を理解する。
- **実現（PostgreSQL）**：primary + standby の 2 コンテナでストリーミングレプリケーション。primary に書く → standby で
  読める（読み取り専用）。`pg_stat_replication` でレプリケーション遅延（lag）を観察。primary を止めて
  `pg_ctl promote` で standby を昇格 → 書けるようになる。**同期 vs 非同期**（`synchronous_standby_names`）の差を見る。
- **1 ステップに込める概念**：物理レプリケーション・primary/standby・hot standby（複製での読み取り）・
  同期 / 非同期レプリケーションと lag・フェイルオーバーと昇格・split-brain の危うさ（触り）。
- **出し入れ検証**：① 同期↔非同期を切り替え、primary のコミット待ち時間とレプリ lag のトレードオフを観測。
  ② primary を停止 → standby を昇格 → 旧 standby に書き込めるようになる（＝フェイルオーバー成立）を実演。

### 案10. 監視と障害の切り分け ―― スロークエリ・ロック待ち・統計ビュー
- **レベル**：★★
- **解く問題（一般）**：「DB が遅い／詰まった」を**勘でなくデータで**切り分ける。何が走っているか・何が誰を
  待たせているか・どのクエリが重いかを、DB 自身が持つ計器で読む。
- **実現（PostgreSQL）**：`log_min_duration_statement` を下げてスロークエリをログに出す。`pg_stat_activity` で
  実行中クエリと `wait_event`、`pg_locks` でロック待ちの連鎖、`pg_stat_statements` 拡張で**重いクエリ TOP N** を集計。
  案5 のロック詰まりや案2 の bloat による遅延を、ここで「計器から逆に診断」する総まとめ。
- **1 ステップに込める概念**：観測する計器（statistics views / 拡張）・スロークエリログ・待機イベント・
  ロックツリーの読み方・「再現 → 計器で観測 → 原因特定」の型。
- **出し入れ検証**：① `log_min_duration_statement` の閾値を下げる／上げると、同じクエリが**ログに出る／出ない**を切り替え。
  ② 案5 のデッドロック／長トランザクションをわざと起こし、`pg_stat_activity`・`pg_locks` で**誰が誰を待っているか**を特定できることを確認。

## おすすめの進め方

ユーザー指定の 4 領域を、依存関係を踏まえて 3 グループで進めるのがおすすめ。番号順に強い前後依存はないが、
WAL（案6）→ バックアップ（案8）→ レプリケーション（案9）は土台が積み上がるので、この順を推奨。

1. **読みの性能と設計の土台グループ（★〜★★）**：案1（インデックスと実行計画）→ 案4（制約と正規化）→ 案2（VACUUM/統計）
   - まず「速いとは何か・正しいデータとは何か」を、`EXPLAIN ANALYZE` を読む癖とともに体に入れる。

2. **並行性と耐久性グループ（★★〜★★★）**：案5（分離レベル・ロック）→ 案6（WAL・耐久性）→ 案3（接続数・プール）
   - 「同時に走る」「コミットを守る」「接続が有限」という本番特有の難所。ここが運用の肝。

3. **守りと運用グループ（★★〜★★★）**：案7（ロール・最小権限・RLS）→ 案8（バックアップ・PITR）→ 案9（レプリケーション・昇格）→ 案10（監視・切り分け）
   - 権限で締め、壊れても戻せるようにし、冗長化し、最後に案10 で「全部の異常を計器から診断する」型で回収する。

> どの案も「まず一般概念 → PostgreSQL での実装 → 設定を出し入れして因果を確かめる」という流儀を踏襲する。
> すべてローカル Docker で完結し、構築・実行・負荷投入はユーザー自身が行う。各プロジェクトは `local/db/{name}/` に置く想定。
> マネージド DB（[../../learn/db/PLAN.md](../../learn/db/PLAN.md)）が「裏で自動でやってくれること」を、ここでは**手で回して中身を見る**——
> 毎回「この設定を PaaS は自分に触らせるか／隠すか」を一言で言語化すると、PaaS と自前運用の対比が立体的になる。
