# KNOWLEDGE — index-explain で新たに出た用語・概念

このステップで初めて出る語のみ。基本 SQL（`CREATE TABLE`/`SELECT`/`WHERE`/`JOIN` 等）は既知前提。

## 実行計画（query plan）と `EXPLAIN` / `EXPLAIN ANALYZE`

DB はクエリを受けると「どう実行するか」の手順（プラン）をプランナが決める。`EXPLAIN` は
**実行せずに見積りプラン**を、`EXPLAIN ANALYZE` は**実際に実行して実測値**（時間・行数）を出す。
`BUFFERS` を付けると**読んだブロック数**まで見え、仕事量の増減を定量的に追える。

## Seq Scan / Index Scan / Bitmap Scan

- **Seq Scan（順次走査）**: テーブルを先頭から全部読む。索引が無い／絞りが緩いとき。
- **Index Scan**: インデックスを辿って該当行を直接拾う。よく絞れるとき速い。
- **Bitmap Index Scan + Bitmap Heap Scan**: 索引で「読むべきページの地図(ビットマップ)」を作ってから
  まとめて本体を読む中間策。中くらいの件数を返すときに選ばれやすい。

## 選択性（selectivity）／カーディナリティ

条件がどれだけ行を絞るか。値の種類が多い列（`user_id` 約10万種）は**高選択性**で索引が効きやすく、
種類が少ない列（`status` 4種）は**低選択性**で、索引があっても Seq Scan が選ばれやすい。
プランナはこの見積りを**統計情報**（`ANALYZE` が更新）から計算する。

## B-tree インデックス

PostgreSQL の既定インデックス。値が**順序付き**に並ぶため、等価（`=`）・範囲（`<`,`>`,`BETWEEN`）・
前方一致 `LIKE 'x%'` に効く。逆に後方一致 `'%x'` や列を加工した条件には効かない。

## 複合インデックスと先頭列規則（leftmost prefix rule）

`(a, b)` の複合インデックスは、**先頭列 `a` を条件に含むときだけ**使える。`b` 単独の条件では使えない。
電話帳が「姓→名」順に並んでいるので姓から引けるが名だけでは引けないのと同じ。

## カバリングインデックスと Index Only Scan（`INCLUDE`）

クエリが必要とする列がすべてインデックス内に揃っていれば、**テーブル本体を読まず**に索引だけで
答えられる（`Index Only Scan`）。`CREATE INDEX ... (key) INCLUDE (other)` で検索キー以外の列を
索引に同梱できる。実際に本体を読まないためには**可視性マップ**が最新である必要があり、`VACUUM`
が前提になる（`Heap Fetches` の値で本体アクセス量が分かる）。

## 可視性マップ（visibility map）

各ページが「全タプル可視（誰から見ても確定）」かを記録するビットマップ。`VACUUM` が更新する。
これが立っていると Index Only Scan が本体アクセスを省略できる。

## 演算子クラス（operator class）と `text_pattern_ops`

インデックスが「どの順序・どの演算子で使えるか」を決める指定。テキスト列を通常ロケールで作った
B-tree は前方一致 `LIKE 'x%'` に**そのままでは使えない**。`CREATE INDEX ... (col text_pattern_ops)`
にすると文字バイト順で並び、前方一致に効くようになる（C ロケールの DB なら既定でも効く）。

## 式インデックス（expression index）

`CREATE INDEX ... ON t (lower(email))` のように、**列ではなく式の結果**に対して索引する。
`WHERE lower(email) = ...` のように条件側で列を加工するクエリを救う手段。条件の式とインデックスの
式が一致して初めて使われる。

## インデックスのトレードオフ（読み ↔ 書き）

インデックスは検索（読み）を速くする一方、`INSERT`/`UPDATE`/`DELETE` のたびに**索引も更新**する
コストとディスク使用を増やす。「とりあえず全列に貼る」が悪手なのはこのため。効くクエリがあるか
（＝高選択性で実際に使われるか）を `EXPLAIN` で確かめてから貼るのが筋。
