# KNOWLEDGE — このプロジェクトで新たに出てくる用語・概念

`entra-spa-login` でカバー済みの語（OIDC、PKCE、ID/アクセストークン、JWT、リダイレクト URI、テナント、MSAL 等）は繰り返さない。ここでは「**初めてリソースサーバー側を作る**」ことで登場する語に絞る。

## リソースサーバー（保護対象 API）

- **リソースサーバー**：保護されたリソース（API）を提供し、**受け取ったアクセストークンを検証して**可否を判断する側。
  - 前プロジェクトは「クライアント側」しか作っていなかった。今回初めて「トークンを**受け取って検証する**側」を自分で書く。
  - これまで宛先だった Microsoft Graph も、実体はマイクロソフトが運用するリソースサーバー。それを自前で作るのが本プロジェクト。
- **Bearer トークン**：「持っている人＝使ってよい人」とみなす持参人式トークン。HTTP の `Authorization: Bearer <token>` ヘッダで送る。だからこそ漏らさない・必ず検証する、が要。

## Expose an API（自前 API の公開）

- **Expose an API**：自前のアプリ登録を「API（リソースサーバー）」として公開する Entra の設定。
- **Application ID URI**：その API を識別する URI。既定は `api://<appId>`。アクセストークンの宛先（aud）になる。
- **委任スコープ（delegated scope）**：`access_as_user` のように「**ユーザーの代理で**この API を使ってよい範囲」を表す単位。API 側が定義し、クライアントが要求する。
  - スコープを定義 → クライアントが要求 → ユーザーが同意 → トークンの `scp` に乗る、という流れ。
  - （「ユーザーの代理ではなくアプリ自身として」のアプリケーション許可は後続 client-credentials-daemon で対比する。）

## トークン検証（リソースサーバーの責務）

受け取ったアクセストークンについて、リソースサーバーは最低限この 4 点を確かめる（`api/server.js`）：

- **署名**：トークンが改ざんされていないか。Entra の公開鍵集合 **JWKS** で検証する。
  - **JWKS (JSON Web Key Set)**：IdP が公開する署名検証用の公開鍵の集合。エンドポイント例：`https://login.microsoftonline.com/<tenantId>/discovery/v2.0/keys`。鍵はローテーションされるためライブラリが定期取得する。
- **iss（発行者 / issuer）**：信頼する IdP が発行したか。v2 トークンは `https://login.microsoftonline.com/<tenantId>/v2.0`。
- **aud（宛先 / audience）**：このトークンは**自分（この API）宛**か。`api://<API_CLIENT_ID>`。
  - ⚠️ 宛先が違うトークンを受け入れてはいけない。Graph 宛トークンで自前 API を通してしまう、といった取り違えを防ぐ。案5（on-behalf-of）で「aud が違うから転送できない」が効いてくる伏線。
- **scp（スコープ / scope claim）**：要求された操作範囲を含むか（`access_as_user`）。無ければ拒否。
- **トークンバージョン**：Entra のアクセストークンには v1 / v2 がある。本プロジェクトはアプリ登録で `requestedAccessTokenVersion=2` を指定し、v2（iss が `.../v2.0`）に揃えて検証をシンプルにしている。

## 401 と 403 の違い

- **401 Unauthorized**：そもそも**誰だか分からない**（トークンが無い／無効）。→ 認証の失敗。
- **403 Forbidden**：**誰かは分かるが権限（スコープ）が足りない**。→ 認可の失敗。
- 同じ「弾かれる」でも理由が違う。本サンプルでは「トークン無し＝401」「スコープ不足＝403」を出し分けて観察する。

## アプリ登録が 2 つに分かれる

- **クライアントのアプリ登録（SPA）** と **リソースサーバーのアプリ登録（API）** は別物。
  - SPA 側：`requiredResourceAccess` で「API のスコープを使いたい」と宣言する。
  - API 側：`Expose an API` でスコープを定義する。
- 1 つの登録に両方を兼ねさせることも可能だが、本プロジェクトは「クライアント vs リソースサーバー」を体で分けるため**あえて 2 つ**に分ける。

## CORS（おまけだが実務上ほぼ必須）

- SPA（`http://localhost:5173`）と API（`http://localhost:3000`）は**別オリジン**。ブラウザは既定でクロスオリジンの取得を制限する。
- API 側が `Access-Control-Allow-Origin` 等を返して明示的に許可する必要がある。`Authorization` ヘッダ付きの呼び出しは、本リクエストの前に **プリフライト（OPTIONS）** が飛ぶ点にも注意。
- 認証・認可そのものではないが、「SPA から別オリジンの保護 API を呼ぶ」と必ず出会うため最小限だけ扱う。
