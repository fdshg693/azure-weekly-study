# KNOWLEDGE — このプロジェクトで新たに出てくる用語・概念

`entra-spa-login` / `api-protect` / `app-roles-rbac` でカバー済みの語（OIDC、PKCE、ID/アクセストークン、JWT、リダイレクト URI、テナント、MSAL、Authorization Code Flow、リソースサーバー、委任スコープ、JWKS、aud・scp 検証、401・403、CORS、認証と認可、App ロール 等）は繰り返さない。ここでは「**パブリックからコンフィデンシャルへ**」で登場する語に絞る。

## パブリッククライアント vs コンフィデンシャルクライアント ★このプロジェクトの核心

OAuth 2.0 はクライアントを 2 種類に分ける。違いは一言、**「秘密（資格情報）を安全に保持できるか」**。

| | パブリッククライアント | コンフィデンシャルクライアント |
|---|---|---|
| 例 | SPA、モバイル / デスクトップアプリ | サーバーサイド Web アプリ、デーモン |
| 秘密を持てるか | **持てない**（ソースが利用者の手元で丸見え） | **持てる**（サーバー上で秘匿できる） |
| コード横取り対策 | **PKCE**（動的に生成する検証値） | **クライアントシークレット／証明書**（＋任意で PKCE） |
| トークンの置き場所 | ブラウザ（sessionStorage 等） | **サーバー側**（ブラウザに渡さない） |
| これまでの auth | entra-spa-login / api-protect / app-roles-rbac | **本プロジェクト** |

- 「秘密を持てるか」がすべての分岐点。SPA は秘密を埋め込めば即漏洩するので、秘密の代わりに **PKCE** で「コードを横取りされても使えない」状態を作っていた。
- サーバーサイド Web アプリは秘密を秘匿できるので、**クライアントシークレット**でトークン交換を認証できる。

## クライアントシークレット / クライアント資格情報（client credentials）

- **クライアントシークレット**：アプリ登録に紐づく「アプリのパスワード」。token エンドポイントで `client_secret` として提示し、「確かにこのアプリだ」と Entra に証明する。
- 発行時に **一度だけ** 値が表示され、以後は再表示できない（ハッシュで保管される）。漏れたら再発行（ローテーション）する。
- **証明書資格情報**：シークレットの代わりに証明書（秘密鍵で署名した client assertion）を使う方式。より強く、運用の主流。本プロジェクトはまずシークレットで最小に学ぶ（証明書化は案10 security-hardening の主題）。
- ⚠ シークレットは「管理・漏洩・ローテーション」の重荷でもある。これを **消す** のが後続の案7（managed-identity）・案8（workload-identity-federation）。

## token エンドポイントでのクライアント認証

- 認可コードフローの後半、**code → token の交換**は `POST /oauth2/v2.0/token` で行う。
- パブリッククライアントはここに `code_verifier`（PKCE）を付ける。
- **コンフィデンシャルクライアントはここに `client_secret` を付ける**（`grant_type=authorization_code` ＋ `client_id` ＋ `code` ＋ `redirect_uri` ＋ `client_secret`）。
- この交換は **ブラウザを介さないサーバー → Entra の直接通信**。だからシークレットが外に出ない。`server.js` の `exchangeCode()` がこれ。
- シークレットが無い／間違っていると Entra は **`invalid_client`** で拒否する（＝出し入れの観察ポイント）。

## Web プラットフォーム vs SPA プラットフォーム（アプリ登録）

- アプリ登録のリダイレクト URI は「プラットフォーム」ごとに登録する。
  - **SPA プラットフォーム**：PKCE 前提、CORS 対応。秘密は使わない（これまでのプロジェクト）。
  - **Web プラットフォーム**：サーバーサイド Web アプリ用。**クライアントシークレット**での認証を前提とする。
- 本プロジェクトは `az ad app create --web-redirect-uris` で **Web プラットフォーム** に登録する。同じ「リダイレクト URI」でも置き場所（プラットフォーム）でフローの前提が変わる。

## BFF（Backend for Frontend）

- フロントエンドの「前」にバックエンドを一枚置き、**そのバックエンドがトークンを握る**設計。
- 流れ：**ブラウザ → 自分のサーバー（セッションクッキー）→ サーバーが API を代理呼び出し（アクセストークン）→ 結果だけブラウザへ**。
- 利点：**トークンをブラウザに置かない**。XSS でトークンを抜かれない、リフレッシュトークンを安全に保持できる、CORS をブラウザに晒さない。
- 本サンプルの `/api/graph` がこれ：ブラウザは `Authorization` ヘッダを付けず、サーバーが保持するアクセストークンで Graph `/me` を取得して返す。

## セッションと httpOnly クッキー（トークンを置かない代わりの「鍵」）

- トークンをブラウザに置かない代わり、ブラウザには **セッション ID（sid）** だけを **クッキー**で渡す。サーバーは sid からトークン一式を引く。
- **httpOnly クッキー**：JavaScript から読めないクッキー。XSS でも盗み出しにくい。`document.cookie` に出てこない。
- **SameSite=Lax**：別サイトからの埋め込みリクエストにはクッキーを送らないが、トップレベルの画面遷移（Entra からの戻り）では送る。CSRF を抑えつつログイン遷移は通すバランス。
- 対比：SPA はトークンそのものを sessionStorage に置いていた。ここではトークンはサーバー、ブラウザは sid クッキーだけ。

## state と nonce（CSRF / リプレイ対策）

- **state**：ログイン開始時にサーバーが発番し、callback で一致を確認する乱数。**CSRF**（他人に仕込まれた偽のコールバック）を弾く。
- **nonce**：authorize 要求に乗せ、返ってきた **ID トークンの `nonce` クレーム**と一致するか確かめる。**リプレイ**（使い回し）を弾く。
- SPA では MSAL.js が裏でやっていた。今回は自前サーバーなので、自分で発番・照合する（`server.js` の `pending` マップ）。

## リフレッシュトークン（offline_access）

- `offline_access` スコープを要求すると、アクセストークンに加えて **リフレッシュトークン**が返る。アクセストークンが切れたら、これで新しいアクセストークンを取り直せる（再ログイン不要）。
- ⚠ リフレッシュトークンは強力なので保管場所が重要。**コンフィデンシャルクライアントはサーバーに安全に保持できる**（SPA だと寿命や扱いに強い制約がある）。これもパブリック／コンフィデンシャルの実利的な差。
- 本サンプルの `refreshIfNeeded()` がこれ：期限が近ければサーバー内でシークレット付きに更新する。
