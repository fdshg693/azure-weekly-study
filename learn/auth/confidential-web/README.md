# confidential-web — パブリックからコンフィデンシャルへ（サーバーで秘密を持つ / BFF）

`auth` の 4 つ目のプロジェクト（[PLAN.md](../PLAN.md) の案3）。これまでの 3 プロジェクトはすべて **SPA ＝ パブリッククライアント**だった。ソースがブラウザで丸見えなので**秘密を持てず**、Authorization Code Flow の横取り対策に **PKCE** を使い、**トークンをブラウザ**（sessionStorage）に置いていた。

本プロジェクトはその正面の対比。**サーバーサイド Web アプリ ＝ コンフィデンシャルクライアント**で、

- **クライアントシークレット**（アプリのパスワード）を持つ。
- 認可コードフローを **サーバーで完結**させる（code → token の交換を、サーバーが**シークレット付きで**実行）。
- 発行されたトークンは **サーバーに保持**し、**ブラウザには渡さない**。ブラウザが持つのは **`sid` クッキー（httpOnly）だけ**。
- これが **BFF（Backend for Frontend）**：トークンをバックエンドで握り、API もサーバーが代理で呼ぶ。

> 学ぶ概念の整理は [KNOWLEDGE.md](./KNOWLEDGE.md)、フロー図は [MERMAID.md](./MERMAID.md) を参照。

## 目的

- **パブリック vs コンフィデンシャル**クライアントの違い（＝「秘密を安全に持てるか」）を、実物で体感する。
- `entra-spa-login` の「**PKCE で秘密なし**」と、本プロジェクトの「**クライアントシークレットで秘密あり**」を正面から対比する。
- **トークンをブラウザに置かない**設計（BFF）を、devtools で「ブラウザには `sid` クッキーしか無い」ことを見て確かめる。
- サーバーが保持するアクセストークンで **Graph `/me` を代理呼び出し**し、BFF の流れ（ブラウザ → サーバー → API）を見る。

## 前提条件

- **Azure CLI**（`az login` 済み）と、アプリ登録**およびクライアントシークレット発行**ができる権限のある Entra テナント
- **Task**（[taskfile.dev](https://taskfile.dev) のコマンドランナー）と **PowerShell 7+**（`scripts/*.ps1` の実行）
- **Node.js 18+**（サーバーの実行。依存は `jose` のみで `task run` 初回に自動インストール。グローバル `fetch` を使うため 18 以上）
- モダンブラウザ（`http://localhost:3000` で開く）

> ⚠ このプロジェクトは **SPA を配信しない**。「ブラウザに秘密もクライアント設定も渡さない」のが主題なので、フロントは**サーバーがレンダリングする最小 HTML** で、`config.js` の生成手順も無い（sibling プロジェクトとの違い）。

## 実行手順

すべてこの `confidential-web` ディレクトリで実行する。

### 1. アプリ登録＋シークレット発行（ユーザーが一度だけ実行）

`task register` が、**Web プラットフォーム**にリダイレクト URI を登録したアプリを作り、**クライアントシークレット**を発行する（SPA プラットフォームではない点が肝）。シークレットは**この一度だけ**表示される。

```bash
task register     # 出力された CLIENT_ID と CLIENT_SECRET を控える（シークレットは一度きり表示）
task tenant       # テナント ID を表示
```

> このリポジトリの方針どおり、Azure 上にオブジェクトを作る操作は**ユーザーが実行**する（AI は実行しない）。

### 2. 設定の記入

`.env.example` をコピーして `.env` を作り、上で得た値を入れる。**`.env` はシークレットを含むので絶対にコミットしない**（`.gitignore` 済み）。

```pwsh
Copy-Item .env.example .env
# .env を編集：TENANT_ID / CLIENT_ID / CLIENT_SECRET（REDIRECT_URI などは既定で可）
```

### 3. サーバーを起動してログイン

```bash
task run          # 初回は jose を npm install → http://localhost:3000 で待ち受け
```

ブラウザで <http://localhost:3000> を開き、

1. **「ログイン」** … サーバーが Entra の authorize へリダイレクト。サインイン＆同意すると `/auth/callback` に戻り、**サーバーが**コードをトークンに交換する（このときシークレットを使う）。
2. **「自分の情報を見る（/api/me）」** … ID トークンのクレーム（名前など）が返る。**トークン本体は応答に含まれない**。
3. **「Graph 経由で /me を取得（/api/graph）」** … **サーバーが**保持するアクセストークンで Graph を叩いた結果が返る（BFF）。

### 4. 「ブラウザにトークンが無い」ことを確かめる（このプロジェクトの肝）

ブラウザの devtools を開く：

- **Application → Cookies → `http://localhost:3000`**：あるのは **`sid`（HttpOnly）だけ**。トークンは無い。
- **Network**：`/api/me` も `/api/graph` も **`Authorization` ヘッダを送っていない**（ブラウザはトークンを持っていないから付けられない）。クッキー `sid` が自動で付くだけ。
- **トークンはどこ？** → サーバーのコンソールに `[session …] …のトークンをサーバーに保管` というログが出る。トークンは**サーバーのメモリ**にだけ存在する。

`entra-spa-login` で同じ devtools を見ると、トークンが sessionStorage に入っていた。**その違いがパブリック↔コンフィデンシャルの実物。**

### 5. 後片付け

```bash
task unregister   # .env の CLIENT_ID のアプリ登録を削除（シークレット・SP も一緒に消える）
```

## 学習の流れ（設定を「出し入れ」して因果を確かめる）

「スイッチを出し入れして、何が効いているか切り分ける」を**コンフィデンシャル＝シークレット**で行う。

1. **正常系**：ログイン → `/api/me` → `/api/graph` が通ることを確認する。
2. **シークレットを壊す**（核心の対比）：`.env` の `CLIENT_SECRET` を 1 文字変えて `task run` し直し → ログインの**コールバックで失敗**する（token エンドポイントが **`invalid_client`**）。**シークレットこそがコンフィデンシャルの資格情報**だと分かる。元に戻す。
   - 対比：`entra-spa-login`（パブリック）にはそもそもシークレットが無く、PKCE で成立していた。
3. **トークンの所在を見る**：手順 4 のとおり、ブラウザには `sid` クッキーだけ・サーバーにトークン、を devtools で確認する。
4. **リダイレクト URI を壊す**：`.env` の `REDIRECT_URI` を登録と違う値（例：ポートを `3001`）にすると、authorize が **`redirect_uri` 不一致**で止まる。戻り先の完全一致は SPA と同じく厳格。元に戻す。
5. **Web と SPA の登録の違い**（対比・任意）：`register.ps1` は `--web-redirect-uris`（**Web プラットフォーム**）を使っている。これを SPA プラットフォームに登録すると、シークレットでの token 交換が想定されず挙動が変わる。「同じリダイレクト URI でもプラットフォームでフローの前提が変わる」ことを意識する。
6. **リフレッシュを観察**（任意）：アクセストークンの寿命が切れる頃に `/api/graph` を押すと、サーバーがリフレッシュトークンで黙って更新する（コンソールにログ）。**サーバーがリフレッシュトークンを安全に持てる**のもコンフィデンシャルの利点。

## 補足

- **`config.js` が無い／ブラウザに設定を渡さない**：これが意図。SPA ではブラウザにクライアント ID 等を渡していたが、コンフィデンシャルではブラウザは何も知らなくてよい。サーバーが `.env` を直接読む。
- **PKCE は付けていない**：対比を明確にするため。実運用ではコンフィデンシャルでも PKCE を併用するのが推奨。本サンプルは「シークレットが効いている」ことを見せることを優先した。
- **インメモリのセッション**：学習用に単一プロセスのメモリに置いている。実運用では Redis 等の共有ストア＋有効期限管理にする。
- **証明書資格情報**：シークレットの代わりに証明書を使うとより強い。シークレットの「管理・漏洩・ローテーション」の重荷を**消す**のは後続の案7（managed-identity）・案8（workload-identity-federation）、証明書化は案10（security-hardening）の主題。
- 次の一歩は [PLAN.md](../PLAN.md) の案4（client-credentials-daemon）。本プロジェクトで持った「クライアントシークレット」を、今度は**ユーザー不在**でアプリ自身が使う（委任 vs アプリケーション許可）。
