# KNOWLEDGE — このプロジェクトで出てくる用語・概念

`auth` の最初のプロジェクトなので、認証まわりの基礎用語をここに集約する。後続プロジェクトでカバー済みの語は繰り返さない。

## 認証と認可

- **認証 (Authentication)**：「誰か」を確かめること（ログイン）。本プロジェクトの主題。
- **認可 (Authorization)**：「何をしてよいか」を決めること（権限）。本プロジェクトではほぼ扱わない（後続の app-roles-rbac へ）。

## OAuth 2.0 / OpenID Connect (OIDC)

- **OAuth 2.0**：「アクセスを委譲する」ための枠組み。アクセストークン（API への通行証）を発行する。
- **OpenID Connect (OIDC)**：OAuth 2.0 の上に「ログイン（本人確認）」を乗せた拡張。ID トークンを発行する。
- 登場人物：
    - **リソースオーナー** … ユーザー本人
    - **クライアント** … 今回の SPA（ブラウザ上の JS）
    - **認可サーバー / IdP** … ログインを受け持ち、トークンを出す（= Entra ID）

## トークン（どちらも JWT）

- **JWT (JSON Web Token)**：`header.payload.signature` の 3 つを `.` で繋いだ文字列。payload に**クレーム**（属性）が入る。署名があるので改ざんを検知できる。
- **ID トークン**：「このユーザーがログインした」ことを示す**身分証**。`name` / `preferred_username` / `tid`（テナント）などのクレームを持つ。**アプリが消費する**。
- **アクセストークン**：「この API を叩いてよい」という**通行証**。`aud`（宛先 API）/ `scp`（許可された操作）を持つ。**API に渡して使う**。
- 見分けの勘所：ID トークンは「誰か」、アクセストークンは「何にアクセスできるか」。本サンプルでは両方をデコードして `aud` / `scp` の違いで対比する。
- ⚠️ **クライアント（SPA）はアクセストークンの中身を解釈してはいけない**。中身を検証・解釈するのは宛先 API の責務。ID トークンの検証も本来はライブラリ/サーバーが行う。本サンプルのデコードは「中身を覗いて学ぶ」用途に限る。

## クライアントの種別とフロー

- **パブリッククライアント**：ソースが全部見える環境（SPA・モバイル）で動き、**クライアントシークレットを安全に保持できない**。SPA はこれ。
- **コンフィデンシャルクライアント**：サーバー側で動き、シークレットを秘匿できる（後続プロジェクトで登場）。
- **Authorization Code Flow**：まず「認可コード」を受け取り、それをトークンに交換する標準フロー。
- **PKCE (Proof Key for Code Exchange)**：パブリッククライアント向けの保護。`code_verifier`/`code_challenge` の対で「認可コードを横取りされても、検証値を知らなければトークンに交換できない」ようにする。**MSAL.js が自動で行う**ため、コードに PKCE は出てこない。
- **Implicit Flow（非推奨）**：トークンを URL ハッシュで直接受け取る古い方式。漏洩リスクが高く、SPA でも今は **Auth Code + PKCE** を使う。

## リダイレクト URI

- ログイン後にブラウザが**戻ってくる先**の URL。IdP に事前登録した値と**完全一致**（スキーム・ホスト・ポート・パス）しないと拒否される。
- 横取り対策の要なので厳格。学習ステップ2でわざと不一致にして `redirect_uri` エラーを観察する。

## スコープ (scope)

- クライアントが「これを要求します」と宣言する権限の単位。
    - `openid` … OIDC を使う（ID トークンを出してもらう）合図
    - `profile` … 氏名等のプロフィール系クレームを含めてもらう
    - `User.Read` … Microsoft Graph で自分の情報を読むアクセス権（アクセストークン側）

## Entra ID 固有

- **テナント (tenant)**：組織ごとの Entra ID の区画。`tenantId`（GUID）で識別。
- **アプリ登録 (App Registration)**：このアプリを Entra に登録したもの。`clientId`（= appId）で識別。「どのアプリとしてログインするか」の正体。
- **サインイン対象アカウント (sign-in audience)**：誰がログインできるか。
    - `AzureADMyOrg` … 自テナントのみ（single tenant・本サンプルの既定）
    - `AzureADMultipleOrgs` / `common` … 任意の組織テナント（multi tenant）
- **MSAL (Microsoft Authentication Library)**：Microsoft 製の認証ライブラリ。本サンプルはブラウザ版 `@azure/msal-browser` を CDN から読み込み、PKCE・トークンキャッシュ・リダイレクト処理を肩代わりさせる。
- **authority**：MSAL に渡す「認可サーバーの場所」。`https://login.microsoftonline.com/<tenantId>` の形。
