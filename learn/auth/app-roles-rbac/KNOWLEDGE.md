# KNOWLEDGE — このプロジェクトで新たに出てくる用語・概念

`entra-spa-login` / `api-protect` でカバー済みの語（OIDC、PKCE、ID/アクセストークン、JWT、リダイレクト URI、テナント、MSAL、リソースサーバー、Expose an API、委任スコープ、JWKS、aud・scp 検証、401・403、CORS 等）は繰り返さない。ここでは「**認証から認可へ**」進むことで登場する語に絞る。

## 認証（Authentication）と認可（Authorization）

- **認証（AuthN）**：「**あなたは誰か**」を確かめること。ログインがこれ。`api-protect` までの主題。
- **認可（AuthZ）**：「**あなたは何をしてよいか**」を決めること。本プロジェクトの主題。
- この 2 つは独立している。**認証は通っている（誰かは分かる）が、認可で弾かれる（その操作はダメ）** が起こりうる ＝ **403 Forbidden**。
  - 「ログインできた」＝「何でもしてよい」ではない。ここが認可の出発点。

## RBAC（Role-Based Access Control / ロールベースアクセス制御）

- **役割（ロール）に権限を束ね、主体（ユーザー）にはロールを割り当てる**認可の方式。
- 「ユーザーごとに個別の権限」ではなく「ロール経由」にすることで、権限管理が運用しやすくなる（人にロールを足し引きするだけ）。
- 本プロジェクトでは `Tasks.Read`（閲覧）/ `Tasks.Write`（追加）という 2 つのロールで、エンドポイントへのアクセスを束ねる。

## App ロール（App Roles / アプリケーションロール）

- **アプリ（ここでは自前 API）が自分で定義する役割**。アプリ登録の `appRoles` に並べる。
- 各ロールは `value`（トークンに乗る文字列、例 `Tasks.Read`）、`displayName`、`allowedMemberTypes`（誰に割り当てられるか：`User` / `Application`）を持つ。
  - 本プロジェクトは `allowedMemberTypes: ["User"]`＝**ユーザー（人）に割り当てる**ロール。
  - （`Application` に割り当てる「アプリケーションロール」は後続 client-credentials-daemon で対比する。）
- **ロールの割り当て**：ユーザー（や グループ）を、API の**サービスプリンシパル（エンタープライズ アプリ）**に対して割り当てる。
  - 割り当ての実体は Graph の `appRoleAssignment`（`principalId`＝ユーザー、`resourceId`＝API の SP、`appRoleId`＝ロール）。
  - 割り当てられたロールの `value` が、そのユーザーへ発行されるアクセストークンの **`roles` クレーム（配列）**に乗る。
  - 割り当てが 1 つも無いと、`roles` クレーム自体が出ない（＝ロール無し）。

## `scp`（スコープ）と `roles`（ロール）の違い ★このプロジェクトの核心

1 つのアクセストークンに、両方が同時に乗りうる。混同しやすいので明確に分ける：

| | `scp`（scope claim） | `roles`（roles claim） |
|---|---|---|
| 表すもの | **アプリ（クライアント）がユーザーの代理で要求した操作範囲** | **主体（ユーザー）に割り当てられた役割** |
| 誰が決める | クライアントが要求し、**ユーザー／管理者の同意**で確定 | **管理者**がユーザー／グループに**割り当て**る |
| 出どころ | Expose an API の委任スコープ（`access_as_user`） | App ロール（`Tasks.Read` / `Tasks.Write`） |
| 形 | スペース区切りの文字列 | 文字列の配列（v2） |
| 問い | 「このアプリは何をしてよいと言われたか」 | 「この人は何者か」 |

- 直感：**`scp` は「アプリに渡した通行許可」、`roles` は「人に与えた肩書き」**。
- 本サンプルの API は両方を見る：`scp`（access_as_user）が無ければそもそも入口で拒否、`scp` はあっても必要な `roles` が無ければ操作ごとに 403。
- ⚠️ 同じ「権限」でも層が違う。クライアント側でスコープを増やしても `roles` は増えない（ロールは要求するものではなく割り当てられるもの）。

## クレームベース認可（Claims-based Authorization）

- リソースサーバーが、トークン内の**クレーム（`roles` / `scp` / `groups` 等）を読んで可否を判断**する方式。
- API は IdP に問い合わせ直す必要がない。**署名済みトークンに同梱されたクレームを信頼**して、その場で判断できる（だから署名検証が前提）。
- 本サンプルの `requireRole()` がこれ：`payload.roles` を見て、要求ロールが無ければ 403。

## App ロール vs グループクレーム（補足）

- 認可の材料として、Entra は **App ロール**のほかに**グループメンバーシップ（`groups` クレーム）**も運べる。
- App ロールは「**このアプリにとっての役割**」をアプリ自身が定義するため、**意味が安定し移植しやすい**（`Tasks.Write` は常に同じ意味）。
- グループは **ID（GUID）がテナント依存**で、別環境に持っていくと壊れやすく、数が多いとトークン肥大化（オーバーエイジ）も起きる。
- そのため「アプリ内の機能認可」は App ロールが扱いやすい。本プロジェクトは App ロールを採用する。
