# KNOWLEDGE — `simple` で新たに出た用語・概念

> db トピック最初のプロジェクト。マネージド DB 固有の概念に絞る。
> ネットワークの一般概念（パブリック／閉域）は network・storage トピックで既習だが、
> ここではそれを **DB（PaaS）に適用**した形で再確認する。

## マネージド DB（PaaS）と Flexible Server
- **マネージドデータベース**: OS パッチ・DB エンジンの更新・バックアップ・監視などの運用を
  Azure が肩代わりするサービス。利用者は「接続して使う」ことに集中できる代わりに、
  OS への SSH や任意拡張のインストールといった自由は失う（IaaS で自前 DB を立てる vm トピックとの対比）。
- **Azure Database for PostgreSQL — Flexible Server**: PostgreSQL のマネージド提供形態。
  コンピュートとストレージが分離され、SKU（vCore）とストレージを別々に選べる。
  旧 Single Server より柔軟（メンテナンス時間帯の指定・停止/再開・ゾーン配置など）。

## サーバー > データベース > ロール の階層
- **サーバー（Flexible Server）**: 接続先の単位（FQDN を持つ）。1 台の中に複数の
  **論理データベース**を持てる。
- **データベース**: テーブル等を入れる論理的な入れ物（このプロジェクトでは `appdb`）。
  接続時の `dbname` で選ぶ。`postgres` という管理用 DB が既定で存在する。
- **ロール（ユーザー）**: ログイン主体（このプロジェクトでは管理者 `pgadmin`）。
  PostgreSQL ではユーザーとグループをまとめて「ロール」と呼ぶ。GRANT で権限を与える。
  → 認可（DB 内の権限）を出し入れする話は Step 2 で深掘りする。

## 接続文字列の構成要素
- マネージド PostgreSQL に繋ぐのに必要な最小要素:
  - `host`（FQDN, 例: `pg-...-xxxx.postgres.database.azure.com`）/ `port`（5432）
  - `dbname`（論理 DB 名）/ `user`（ロール）/ `password`
  - `sslmode`（後述。既定で `require`）
- 本プロジェクトは `.env`（`PGHOST` / `PGUSER` / `PGPASSWORD` / `PGDATABASE` / `PGSSLMODE`）に
  置き、`connect.py` が読む。`PG*` は psql/psycopg が認識する慣習的な環境変数名でもある。

## パブリックエンドポイント + ファイアウォール規則
- Flexible Server の `publicNetworkAccess: Enabled` は**パブリックな接続経路を持つ**設定。
  ただしそれだけでは繋がらない。**ファイアウォール規則**で**許可した IP レンジ**からのみ通れる。
- 規則が 1 件も無い初期状態＝**どこからも繋がらない**。「経路はあるが許可制」が要点で、
  **マネージド DB はデフォルトで閉じている**と表現される。
- 規則は IP の開始〜終了アドレスで表す（このプロジェクトは自分の 1 IP を `ClientIP` 規則で出し入れ）。
- 対比: vm トピックの **NSG** は「VM が属するサブネットの受信フィルタ」。
  こちらは「マネージド DB サービス自身が持つ IP 許可リスト」で、守る対象も設定場所も違うが、
  **「許可を出し入れすると到達が変わる」**という体験は同型。

## TLS 必須（sslmode=require）
- マネージド PostgreSQL は既定で**通信の暗号化（TLS）を要求**する。
  クライアントは `sslmode=require` 以上で繋ぐ必要がある（平文接続は拒否）。
- ローカルからパブリック経路で繋ぐ本プロジェクトでは特に重要（盗聴対策）。

## Burstable（バースト可能）SKU と課金感覚
- `Standard_B1ms`（Burstable 階層）は普段低い CPU で「クレジット」を貯め、必要時に
  一時的に高い性能を出せる安価なコンピュート。学習・検証向き（vm の B シリーズと同発想）。
- **課金の注意**: マネージド DB は VM の `deallocate`（割り当て解除で課金停止）に当たる
  「気軽な停止」が無い。コンピュートは稼働中ずっと課金されるため、**使い終わったら
  `just destroy`（または明示的なサーバー停止）**で止める。ストレージ／バックアップにも課金がある。
