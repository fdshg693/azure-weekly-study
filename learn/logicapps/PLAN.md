# Logic Apps 学習プラン — 何から作るか

このファイルは logicapps トピックの**プロジェクト設計の目安（ロードマップの正本）**。
このトピックはまだプロジェクトが無いので、ここでは「最初の最小構成 → どう深化させるか」を定める。
方針はリポジトリ共通（[../CLAUDE.md](../CLAUDE.md)）どおり:
「**一般概念／最小構成 → 実装 → 設定を出し入れして因果を確かめる**」「**構築・実行はユーザー自身が行い、AI は Azure 上で実行しない**」。

---

## 1. このトピックで学びたいこと（ゴール像）

Logic Apps は **ノーコード／ローコードのワークフロー統合（iPaaS）** サービス。
「トリガー（起点）→ アクション（処理）→ コネクタ（外部接続）」というイベント駆動の組み立てを、
コードではなく**ワークフロー定義（JSON / デザイナ）**で行う点が func（Azure Functions）との対比軸になる。

主に押さえたい概念:

- **Consumption（従量）と Standard（App Service ベース）の違い**: 課金モデル・実行環境・VNet 統合・ローカル開発可否。
- **トリガーの種類**: Recurrence（スケジュール）/ HTTP（リクエスト）/ コネクタトリガー（イベント駆動）。
- **コネクタ**: ビルトイン vs マネージド、**API 接続（connections）と認証**（接続リソースが別に出来る独特のモデル）。
- **制御フロー**: 条件分岐 / For each / Until / スコープと**実行履歴（run history）での可視化・再実行**。
- **状態とエラー処理**: リトライポリシー、`runAfter`、ステートフル vs ステートレス（Standard）。
- **IaC 化**: ポータルのデザイナで作ったものを **Bicep / ARM のワークフロー定義** に落とす難しさと向き合う。

> 比較の軸: func は「コードで書くサーバーレス」、Logic Apps は「**繋ぐことが主役**のサーバーレス」。
> 同じ「Blob に置いたら処理」を両方で作ると違いが体感できる（→ Step 1）。

## 2. 推奨ロードマップ（やさしい順）

各プロジェクトは `learn/logicapps/{name}/` に置く想定。「**1 プロジェクト = 主役の概念 1〜2 個**」に絞る。
最初は **Consumption** で手軽に一周し、後半で **Standard** に踏み込む。

### Step 1 — `simple`（最小ワークフローを一周）
**主役**: トリガー → アクション → 実行履歴の基本ループ。
- Consumption プランで、**Recurrence トリガー → HTTP アクション**（公開 API を叩く）だけの最小ワークフロー。
- もしくは **Request（HTTP）トリガー → レスポンス返却** で「叩くと動く」を体感。
- **因果を確かめる実験**: トリガー間隔を変える／アクションを 1 つ足す → **run history** で各ステップの入出力・所要時間を見る。
- IaC: ワークフロー定義（`workflow.json` 相当）を Bicep の `Microsoft.Logic/workflows` で表現してみる。

### Step 2 — `connector-auth`（コネクタと API 接続が主役）
**主役**: マネージドコネクタの**接続リソース（connections）と認証**。
- Storage（Blob 作成トリガー）や、すでに学んだ func / Storage トピックのリソースと連携。
- **因果を確かめる実験**: 接続の認証情報（接続文字列 / Managed Identity）を切り替えると、どこに「connection」リソースが生え、
  どの権限で動くかを観察。auth トピックの **Managed Identity / RBAC** をここに接続する。
- ビルトインコネクタとマネージドコネクタで、課金・レイテンシ・VNet 到達性がどう違うかをメモ。

### Step 3 — `control-flow`（分岐・ループ・エラー処理が主役）
**主役**: 条件分岐 / For each / リトライ / `runAfter`。
- 複数アイテムを For each で処理し、一部を意図的に失敗させてリトライポリシーと `runAfter`（失敗時のみ実行）を確認。
- **因果を確かめる実験**: リトライ回数・間隔を変える、`runAfter` の分岐条件を変える → run history で再試行と分岐の挙動が変わるのを見る。

### Step 4 — `standard-local`（Standard プランとローカル開発が主役）
**主役**: Standard プラン（App Service ベース）/ ステートフル vs ステートレス / ローカル実行。
- VS Code の Logic Apps 拡張でローカル開発・デバッグ（func のローカル開発体験に近い）。
- **因果を確かめる実験**: 同じワークフローをステートフル／ステートレスで動かし、run history の残り方・レイテンシ・課金感覚の違いを比較。
- VNet 統合・複数ワークフローの同居など、Consumption では出来ないことを体感。

### Step 5（発展）— `func-vs-logicapps` / イベント駆動連携
- 同じ要件（例: Blob 投入 → 加工 → 通知）を **func と Logic Apps の両方**で作り、コード vs ワークフローの境界を言語化。
- Event Grid / Service Bus と組み合わせ、複数サービスをまたぐイベント駆動オーケストレーションへ。

## 3. 進め方のメモ

- 各プロジェクトに共通構成（`README.md` / `KNOWLEDGE.md` / `justfile` または `Taskfile.yml`）を置く。複雑になったら justfile ではなく Taskfile。
- ポータルで作る誘惑が強いサービスだが、**最終的に Bicep / ARM の定義へ落とす**ことを目標にし、「クリックで出来たもの」と「コード」の対応を毎回確認する。
- トピックの習熟度は、最初のプロジェクトを 1 つ追加したタイミングで `learn/logicapps/CLAUDE.md` に記録する。
- 既存の func / storage / auth トピックの資産（Managed Identity・RBAC・Blob）と積極的に接続すると学習効率が良い。
