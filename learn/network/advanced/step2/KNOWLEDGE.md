# KNOWLEDGE.md (advanced/step2)

このステップで新たに登場した用語や概念を解説します。
（WAF の Detection/Prevention モードそのものは advanced/step1 を、ヘルスプローブ・負荷分散の基礎は basic/step9・10 を参照）

## エッジ（Edge）とエニーキャスト（Anycast）
**エッジ**とは、利用者に物理的に近い場所に置かれた受け口（PoP: Point of Presence）のことです。
リクエストをまず最寄りのエッジで受け止め、そこから実体（オリジン）へ中継します。利点は 2 つ。

1. **近さ＝速さ**：利用者は最寄りの PoP で TLS を確立できるので往復遅延が小さい。
2. **盾になる**：攻撃トラフィックもまずエッジが受けるので、オリジンに直接到達させずに吸収・緩和できる。

**エニーキャスト**は、同じ 1 つの IP アドレスを世界中の複数の PoP から広告（アドバタイズ）し、
利用者のパケットを「ネットワーク的に最も近い PoP」へ自動的に届ける仕組みです。
Front Door のエンドポイント（`*.azurefd.net`）はこのエニーキャストで公開されており、
「1 つの入口アドレスなのに、受けるのは世界中の最寄りエッジ」という構図になります。

## グローバル分散 vs リージョン内分散（basic/step9・10 との対比）
- basic/step9（Load Balancer・L4）・step10（Application Gateway・L7）は、**1 つのリージョン内**で複数の VM に振り分ける仕組みでした。
- 本ステップの Front Door は、**リージョンをまたいだグローバル**なエッジで受け、近いバックエンドへ流します。
- 「どこで受けるか」が *リージョン内のロードバランサ* から *世界中のエッジ* に上がった、と捉えると位置づけが分かりやすいです。

## Azure Front Door（Standard/Premium）
グローバルな **L7（HTTP/HTTPS）エッジ入口**です。主な役割は次の通り。

- **エンドポイント（afdEndpoint）**：エニーキャストで公開される入口ホスト名（`<name>-<hash>.z01.azurefd.net`）。
- **オリジングループ / オリジン**：実バックエンド（本ステップでは VM の公開 FQDN）。ヘルスプローブで生きているオリジンだけに振り分ける。
- **ルート**：どのパス（`/*`）をどのオリジングループへ送るか。本ステップは HTTP でオリジンへ転送（`forwardingProtocol: HttpOnly`）。
- **セキュリティポリシー**：WAF ポリシーをエンドポイントのドメインに紐付ける接着剤。

SKU は **Standard** と **Premium** があり、WAF 機能に差があります。

| | Standard | Premium |
| --- | --- | --- |
| カスタムルール（レート制限・geo フィルタ・IP 制限） | ○ | ○ |
| マネージドルールセット（DRS：OWASP 相当） | × | ○ |
| Private Link でオリジンへ閉域接続 | × | ○ |

本ステップの核心は **レート制限（カスタムルール）** なので、安価な **Standard** で十分です。
（step1 の OWASP マネージドルールのような「攻撃パターン照合」をエッジでやりたい場合は Premium が必要です。）

## レート制限（Rate Limiting）— 体積型攻撃の緩和
WAF の **カスタムルール**の一種（`ruleType: RateLimitRule`）です。
**クライアント IP ごとに「一定時間内のリクエスト数」を数え、しきい値を超えた分をブロック（429 Too Many Requests）** します。

- `rateLimitThreshold`：窓あたりの許容リクエスト数（本ステップは 30）。
- `rateLimitDurationInMinutes`：計数の窓（1 または 5 分）。窓が空けばカウンタはリセットされる。

これが「体積型（フラッディング）攻撃の緩和」の L7 での具体策です。正規利用者の通常アクセスは閾値以下なので影響を受けず、
1 つの送信元からの異常な連打だけが弾かれます。
Front Door のレート制限は **PoP 単位**で計数されるため、実測の 429 が出始める件数は厳密には閾値ぴったりにならないことがあります（学習上は「効いている／いない」の差が見えれば十分）。

> step1 の WAF mode（Detection/Prevention）はここでも同じ意味で効きます。Detection ならレート超過もログだけで通し、Prevention なら 429 で実ブロック。**出し入れするスイッチが「SQLi の照合」から「レート超過の判定」に変わっただけ**です。

## エッジ・フロンティング（オリジンの直アクセス遮断）
エッジ防御は「**オリジンに直接到達させない**（必ずエッジを通させる）」ことで完成します。
オリジンの公開 IP に攻撃者が直接アクセスできてしまうと、せっかくのエッジを迂回されてしまうためです。本ステップでは 2 段で塞ぎます（network 層のみ実装）。

1. **NSG の service tag `AzureFrontDoor.Backend`**：オリジンの NSG で、HTTP(80) の送信元を Front Door のバックエンド向き IP 群だけに限定する。インターネットからの直アクセスは既定 Deny で落ちる。（本ステップで出し入れするスイッチ）
2. **`X-Azure-FDID` ヘッダの検証**（アプリ層・本ステップでは未実装）：service tag だけだと「他テナントの Front Door」も通れてしまうため、本番では自分の Front Door の ID を示すヘッダをオリジン側で検証して、自分のプロファイル経由だけを受け入れます。

> `AzureFrontDoor.Backend` は「Front Door がオリジンへ向けて出ていく IP 群」を表す service tag です。`basic` で使った `VirtualNetwork`/`Internet`/`GatewayManager` と同じ「名前付き IP 集合」の一種です。

## Azure DDoS Protection（本ステップでは概念のみ・未デプロイ）
DDoS（分散サービス妨害）攻撃のうち、特に **体積型（L3/L4 のパケット洪水）** を緩和する仕組みです。Azure には 2 つの層があります。

- **インフラ DDoS 防御（Basic）**：Azure の全公開エンドポイントに**常時・無償**で付く、プラットフォーム規模の防御。Front Door のエッジも、この基盤の上で体積型トラフィックを吸収します。
- **DDoS Network/IP Protection（有償プラン）**：VNet/公開 IP 単位で、適応的なしきい値学習・攻撃メトリクス・アラート・コスト保護などを追加する上位プラン。

本ステップでは**有償の DDoS Protection プランはデプロイしません**（プランは月額が非常に高価で、学習目的での常時起動に向かないため）。
代わりに、**「エッジで受ける（Front Door）＋ L7 レート制限＋オリジン直アクセス遮断」** という、学習環境でも安全に出し入れできる防御の組み合わせを手で確かめます。
L3/L4 の体積型に対しては「Azure 基盤の DDoS 防御が常時働いており、必要に応じて有償プランで強化する」という整理だけ押さえておきます。

## step1（App Gateway WAF）との対比
| 観点 | advanced/step1 (App Gateway WAF) | advanced/step2 (Front Door) |
| --- | --- | --- |
| スコープ | **リージョン内**の入口 | **グローバル・エッジ**の入口 |
| 主眼 | HTTP の中身を検査（SQLi/XSS を弾く） | エッジで受けて**体積型を緩和**（レート制限）＋近接分散 |
| WAF ルール | OWASP マネージドルールセット | レート制限のカスタムルール（Standard） |
| 出し入れスイッチ | WAF mode（Detection⇄Prevention） | WAF mode＋オリジンの直アクセス lock/unlock |

両者は排他ではなく、**Front Door（エッジ）→ App Gateway（リージョン内 WAF）→ バックエンド** と重ねて「多層の入口防御」にできます（PLAN の案4＋案5 を合わせた構図）。
