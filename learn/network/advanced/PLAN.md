# advanced — 次のステップの候補（PLAN）

`basic`（step1〜12）で、**通信制御の独立した軸（許可／経路／名前解決／外向き・内向き）** と、それを Azure の基本リソース（VNet・サブネット・NSG・UDR・NAT GW・Bastion・Private DNS/Endpoint・LB・App Gateway・Firewall・Network Watcher）に対応づけるメンタルモデルを構築した。

`advanced` はその前提のうえで、**basic がスコープ外として切った領域**（ハイブリッド接続・動的ルーティング・広域設計・セキュリティの作り込み・コンテナ／PaaS 固有のネットワーク・観測の発展）へ踏み込む。

## このフェーズの方針（basic との違い）

- **1 ステップ＝1 概念にこだわらない。** basic で基礎を積んだので、関連する複数の概念を 1 ステップでまとめて導入してよい（例：「VPN Gateway ＋ IPsec ＋ BGP」を 1 ステップで扱う）。
- **学習手法は basic と同じ** ―― 「設定を出し入れして、通信の変化から因果を確かめる」。スイッチを足しても、切り分けの考え方は変わらない。
- **まず一般概念、次に Azure 実装。** どの案も「ネットワーク一般としてどういう問題を解くか」を先に置き、Azure のリソースはその実現例として扱う。

## 案一覧

各案に **レベル（★＝basic 直後でも入りやすい／★★＝中／★★★＝発展）**、**関連する basic ステップ**、**1 ステップに込める概念**を併記する。番号は優先度ではなく識別用。

---

### 案1. オンプレ／拠点間をつなぐ ―― VPN Gateway と IPsec、そして BGP

- **レベル**：★★★
- **関連 basic**：step2（ピアリング＝同一クラウド内の VNet 接続）・step3（UDR の静的経路）
- **解く問題（一般）**：物理的に離れた拠点（オンプレ／別クラウド）同士を、公衆インターネット越しに暗号化トンネルで安全につなぐ。ピアリングが「同じ Azure 内の私設配線」だったのに対し、こちらは「外を経由する暗号化トンネル」。
- **Azure 実現**：`GatewaySubnet` に VPN Gateway を置き、Site-to-Site（IPsec/IKE）トンネルを張る。検証用の "オンプレ" は別 VNet ＋ 別 VPN Gateway（VNet-to-VNet）で代用。
- **1 ステップに込める概念**：IPsec トンネル＋ローカル/リモートゲートウェイ＋**BGP による動的経路交換**（basic で扱わなかった BGP をここで導入）。「静的 UDR（step3）」と「BGP で経路が自動伝播する」対比までを 1 ステップで。
- **出し入れ検証**：トンネルを up/down、または BGP を on/off して「経路が消える／復活する」を観測。経路を成立させているのがトンネルと経路交換であることを切り分ける。

### 案2. 動的ルーティングを正面から ―― ルート伝播と経路の優先順位

- **レベル**：★★★
- **関連 basic**：step3（UDR・NVA・有効/無効の切り替え）
- **解く問題（一般）**：手で書いた静的経路（UDR）は規模が増えると破綻する。経路を「広告し合って自動で学習する」仕組み（BGP）と、複数経路があるときの**優先順位（静的 vs 学習、ロンゲストマッチ）**を理解する。
- **Azure 実現**：VPN Gateway／Route Server を使った BGP 学習経路と、UDR の静的経路を同一サブネットで共存させ、どちらが選ばれるかを観測。
- **1 ステップに込める概念**：経路伝播・AS パス的な優先・UDR と学習経路の競合解決をまとめて。
- **備考**：案1 と密接。案1 を「接続の実現」、案2 を「経路選択の理屈」と分担させるか、合体して 1 つの大型ステップにしてもよい。

### 案3. 広域・大規模トポロジ ―― ハブ&スポークから Virtual WAN へ

- **レベル**：★★★
- **関連 basic**：step3（手組みハブ&スポーク）・step11（ハブ集約の Firewall）
- **解く問題（一般）**：拠点・リージョンが増えると、手組みのハブ&スポーク＋UDR は管理しきれない。**トランジット（スポーク同士・拠点同士を中央で中継）** をマネージドに任せる広域設計へ。
- **Azure 実現**：Virtual WAN ＋ Virtual Hub。step3 で UDR と NVA を手組みして実現したスポーク間通信が、マネージドハブで自動化されることを対比。マルチリージョンのハブ間接続も視野に。
- **1 ステップに込める概念**：トランジットルーティング・マネージドハブ・（任意で）リージョン跨ぎ接続をまとめて。
- **出し入れ検証**：ハブ接続を付け外しして、スポーク間/リージョン間の到達性が変わることを確認（step3 の「UDR 出し入れ」のマネージド版）。

### 案4. L7 を「守る」 ―― WAF と TLS 終端の作り込み

- **レベル**：★★
- **関連 basic**：step10（Application Gateway・L7 ルーティング・TLS 終端の概念）
- **解く問題（一般）**：L7 リバースプロキシは「振り分ける」だけでなく「**中身を検査して防ぐ**」場所にもなる。HTTP の中身を見て攻撃パターン（OWASP Top 10 的な SQLi/XSS）を弾く WAF と、入口での TLS 終端＋証明書運用を一体で理解する。
- **Azure 実現**：Application Gateway に WAF（WAF_v2）を有効化し、リスナーに証明書を載せて HTTPS 終端。OWASP ルールセットを Prevention/Detection で切り替え。
- **1 ステップに込める概念**：WAF（L7 検査）＋ TLS 終端＋証明書（Key Vault 連携）をまとめて。step10 が「どこへ振るか」なら、本案は「通す前に検査する」。
- **出し入れ検証**：悪性っぽいリクエスト（`?id=1' OR '1'='1` など）を投げ、WAF を Detection→Prevention に切り替えて「ログだけ／実ブロック」が変わることを観測。step11（FQDN egress 制御）が「外向きの検査」だったのに対し、本案は「内向きの検査」と対比。

### 案5. 可用性とエッジ防御 ―― Front Door／DDoS Protection

- **レベル**：★★
- **関連 basic**：step5・step11（パブリック IP・egress）・step9/10（負荷分散）
- **解く問題（一般）**：公開エンドポイントは攻撃（特に体積型 DDoS）の標的になる。エッジ（ユーザーに近い場所）で受けて、近いバックエンドへ流し、異常トラフィックを吸収・緩和する。
- **Azure 実現**：Azure Front Door（グローバル L7・エニーキャスト入口）＋ DDoS Protection。step9/10 がリージョン内の分散だったのに対し、こちらはグローバルなエッジ分散。
- **1 ステップに込める概念**：エッジ／エニーキャスト・体積型攻撃の緩和・グローバル分散をまとめて。
- **備考**：案4 の WAF はエッジ（Front Door）側にも置けるので、案4 と合わせて「入口防御」のまとまりにできる。

### 案6. コンテナのネットワーク ―― AKS の Pod 通信と CNI

- **レベル**：★★★
- **関連 basic**：step1（VNet/サブネット）・step8（Private Endpoint）・step9（LB）
- **解く問題（一般）**：コンテナは「サブネット内の VM」とは別の住所体系（Pod ごとの IP・オーバーレイ vs フラット）を持つ。L4 LB（step9）が Service として、Private Endpoint（step8）が外部 PaaS 接続として、コンテナ世界でどう現れるかを理解する。
- **Azure 実現**：AKS を Azure CNI（Pod が VNet の IP を持つ）と Overlay の 2 方式で対比。Service（type=LoadBalancer）が step9 の LB に対応し、Ingress が step10 の App Gateway（AGIC）に対応することを確認。
- **1 ステップに込める概念**：Pod ネットワーク・CNI（フラット/オーバーレイ）・Service/Ingress と既存 Azure リソースの対応をまとめて。
- **備考**：basic の総まとめ的な位置づけ。step1・8・9・10 の知識が「コンテナ前提」でどう写像されるかを一気に見る。

### 案7. 観測を運用へ ―― Connection Monitor と継続的なネットワーク診断

- **レベル**：★★
- **関連 basic**：step12（Network Watcher・IP Flow Verify・フローログ）
- **解く問題（一般）**：step12 は「その場で 1 回診断する」だった。実運用では**継続的に**疎通・遅延・経路を監視し、劣化を検知したい。点の観測から線（時系列）の観測へ。
- **Azure 実現**：Connection Monitor で複数経路の到達性・RTT・ホップを継続計測し、フローログ（VNet Flow Logs）＋ Traffic Analytics で集計・可視化。
- **1 ステップに込める概念**：継続監視・遅延/経路の時系列・ログ集計をまとめて。step12 の単発診断ツール群の発展として一気に。

### 案8. PaaS を「提供する」側へ ―― Private Link Service

- **レベル**：★★★
- **関連 basic**：step8（Private Endpoint＝PaaS を「使う」側）・step9（LB）
- **解く問題（一般）**：step8 は Microsoft の PaaS を**消費**する側だった。今度は自分のサービスを、相手の VNet からプライベート IP で**提供**する（プロバイダ/コンシューマモデル）。
- **Azure 実現**：Standard LB の裏に自前サービスを置き、Private Link Service として公開。別 VNet から Private Endpoint で接続し、step8 と「提供/消費」が表裏であることを確認。
- **1 ステップに込める概念**：プロバイダ側モデル・LB との連動・コンシューマ承認フローをまとめて。

## おすすめの進め方

basic からの距離が近い順・依存関係を踏まえると、次の 3 グループに分けて進めるのがおすすめ。

1. **入口防御グループ（★★）**：案4（WAF/TLS）→ 案5（Front Door/DDoS）
   - step9/10 の直接の発展で、基礎からの飛躍が小さく入りやすい。
2. **接続・経路グループ（★★★）**：案1（VPN/IPsec/BGP）＋ 案2（動的ルーティング）→ 案3（Virtual WAN）
   - step2/3 の発展。案1＋案2 は合体して 1 大型ステップにしてもよい。basic がスコープ外と明言した「ハイブリッド・BGP・広域」を正面から扱う中核。
3. **発展・総まとめグループ（★★/★★★）**：案7（観測の運用化）→ 案8（Private Link 提供側）→ 案6（AKS で総ざらい）
   - 既存ステップの知識を運用・提供・コンテナの文脈で写像し直す仕上げ。

> どの案も「まず一般概念 → Azure 実装」「設定を出し入れして因果を確かめる」という basic の学び方を踏襲する。構築・実行はユーザー自身が行う（AI が Azure 上で実行することはない）。
