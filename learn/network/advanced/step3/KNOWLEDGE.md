# KNOWLEDGE.md (advanced/step3)

このステップで新たに登場した用語や概念を解説します。
（VNet・サブネット・NSG・UDR の基礎は basic を、ピアリングは basic/step2、静的経路は basic/step3 を参照）

## ハイブリッド接続と「拠点間 VPN」
**ハイブリッド接続**とは、オンプレミスや別拠点・別クラウドのネットワークを、Azure の VNet と相互接続することです。
本ステップが扱うのはそのうち **Site-to-Site（拠点間）VPN** ―― 物理的に離れた 2 つのネットワークを、
**公衆インターネット越しの暗号化トンネル**で結ぶ方式です。
ピアリング（basic/step2）が「同一クラウドのバックボーン上の私設配線」だったのに対し、
こちらは「外（公衆網）を経由するが、暗号化されているので安全」という違いがあります。

## IPsec / IKE と事前共有鍵（PSK）
- **IPsec**：IP パケットを暗号化・認証してトンネル化するプロトコル群。トンネルの「中身を守る」担当。
- **IKE（Internet Key Exchange）**：トンネルを張る前に、双方で暗号鍵を安全に取り決めるネゴシエーション。
- **PSK（Pre-Shared Key・事前共有鍵）**：両端であらかじめ共有しておく秘密の文字列。
  IKE はこれを使って相手が正規かを確認する。**両方向の接続で同じ値**でないとトンネルが上がらない。
  本ステップでは `sharedKey` パラメータとして渡し、`tunnel-up` での再作成時も同じ PSK を使う。

## VPN Gateway（と GatewaySubnet）
VNet 内に置く、**トンネルの終端を担うマネージドなルータ**です。

- **`GatewaySubnet`**：VPN Gateway 専用のサブネット。**名前は必ず `GatewaySubnet`** で、Azure はここに
  ゲートウェイのインスタンスを配置する。VM を置いてはいけない。**NSG は付けない**（制御通信を壊さないため）。
- **SKU（VpnGw1 など）**：性能・機能の段階。**BGP を使うには VpnGw1 以上**が必要（Basic SKU は BGP 非対応）。
- **RouteBased / PolicyBased**：
  - **RouteBased（ルートベース）**：経路表に基づいて「どのトンネルへ流すか」を決める。**BGP・VNet-to-VNet・
    複数トンネルが使えるのはこちら**。本ステップはこれ。
  - **PolicyBased（ポリシーベース）**：固定の「このアドレス対このアドレス」のセレクタでのみ流す古い方式。BGP 不可。
- **Public IP**：トンネルの「外側エンドポイント」。相手はこの IP に向けて IPsec を張る。

> プロビジョニングに 30〜45 分かかり、時間課金も高い。学習では「作る → 出し入れ → **必ず削除**」を徹底する。

## VNet-to-VNet 接続（"オンプレ" の代用）
2 つの VPN Gateway 同士を IPsec トンネルで結ぶ接続タイプ（`connectionType: Vnet2Vnet`）。
本来オンプレ相手なら、相手側ルータを表す **Local Network Gateway**（相手の公開 IP と背後のプレフィックスを登録した
論理オブジェクト）を作り、`connectionType: IPsec` でつなぎます。本ステップでは実機オンプレが無いので、
**"オンプレ" 役を別 VNet＋別 VPN Gateway で代用**し、VNet-to-VNet で結んでいます（学べる概念は同じ）。
双方向通信のため接続は **2 本**（各ゲートウェイ起点に 1 本ずつ）作ります。

## BGP（Border Gateway Protocol）と ASN
**BGP** は、ルータ同士が「自分が到達できるネットワーク（プレフィックス）はこれ」と**広告し合い、
経路を動的に学習する**ルーティングプロトコルです。手書きの静的経路（UDR）と対になる「動的ルーティング」の代表。

- **ASN（Autonomous System Number・自律システム番号）**：BGP を話すルータ群を識別する番号。
  **VNet-to-VNet の BGP では両端で必ず異なる ASN** にする（同じだと内部 iBGP 扱いになり経路交換が成立しない）。
  本ステップは hub=65515（Azure 既定）、onprem=65501。`65515` など一部は Azure 予約なので自前 ASN には使えない。
- **BGP ピアリングアドレス**：ゲートウェイが BGP を話すための IP。`GatewaySubnet` 内から Azure が自動採番する。
- 仕組み：トンネルが上がると、その上で BGP セッション（state=Connected）が張られ、
  各ゲートウェイが自 VNet のアドレス空間を相手へ広告 → 相手の経路表に「学習経路」として入る。

### 静的 UDR（basic/step3）との対比 ―― このステップの主題
| | 静的 UDR（basic/step3） | BGP（本ステップ） |
| --- | --- | --- |
| 経路の入れ方 | 人が経路表に手書き | ルータ同士が自動で広告・学習 |
| 相手にネットワークを足したとき | **両側で手修正が必要** | **自動で広告され、相手の経路表に現れる** |
| 規模が増えると | 手作業が爆発して破綻 | 自動伝播でスケールする |
| 確認コマンド | （ルートテーブルを見る） | `list-learned-routes` / `list-bgp-peer-status` |

`just add-prefix` で onprem VNet に `10.60.0.0/16` を足すと、**経路を一切手書きしていないのに**
hub 側の `learned-routes` に `10.60.0.0/16` が `origin=EBgp` で現れます。これが BGP の真価です。

## 学習経路（learned routes）の読み方
`az network vnet-gateway list-learned-routes` の各列の意味:

- **network**：学習したプレフィックス（例 `10.50.0.0/16`）。
- **origin**：経路の出どころ。**`EBgp`** なら外部 BGP ピアから学習した動的経路、`Network` は自 VNet 由来など。
- **nextHop**：そのプレフィックスへ送るときの次ホップ（= 対向ゲートウェイの BGP ピア IP）。
- **asPath**：経路が通ってきた ASN の並び（対向の ASN が含まれる）。経路の出所・ループ防止に使う。

`origin=EBgp` で対向プレフィックスが見えること ＝「トンネル＋BGP で経路が動的に入った」ことの動かぬ証拠です。

## `az vm run-command`（公開 IP / SSH なしで VM 内から実行）
VM の Azure エージェント経由で任意のシェルスクリプトを実行する仕組み。**インバウンドの口（公開 IP・SSH 22）を
一切開けずに、VM の "内側" からコマンドを走らせられる**。本ステップでは hub VM の内側から
`curl http://10.50.1.4/`（対向 VM の private IP）を実行し、**トンネル越しの private 通信**だけを検証している
（公開経路を一切作らないので、見ているのが純粋にトンネル経由の到達性だと切り分けられる）。

## トンネル up/down が示すこと
`tunnel-down`（接続削除）→ `tunnel-up`（再作成）で、`test` の 200⇄TIMEOUT と
`learned-routes` の経路出現⇄消失が連動します。これにより、拠点間到達を成立させているのが
**「IPsec トンネルが張れていること」＋「その上で経路が交換されていること」** の 2 点であると切り分けられます。
