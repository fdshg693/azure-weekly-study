# advanced — basic の先へ、より高度なネットワークを手を動かして学ぶ

`basic`（step1〜12）で築いた **「通信制御の独立した軸（許可／経路／名前解決／外向き・内向き）」** と、
それを Azure の基本リソースに対応づけるメンタルモデルを土台に、`basic` が**スコープ外として切った領域**へ踏み込む学習群です。

各ステップの候補と全体方針は [`PLAN.md`](./PLAN.md) に、各ステップの詳細は各 `step*/README.md` にあります。
ここでは **このフェーズで何を、どう学ぶか** をまとめます。

## このフェーズで学べること

`basic` が扱わなかった、次の発展領域を対象にします（`PLAN.md` の案1〜8 に対応）。

- **ハイブリッド接続・動的ルーティング** ―― VPN Gateway / IPsec / BGP、ルート伝播と経路の優先順位（案1・案2）
- **広域・大規模設計** ―― ハブ&スポークから Virtual WAN へ（案3）
- **セキュリティの作り込み** ―― L7 の WAF と TLS 終端、エッジ防御（Front Door）と DDoS の緩和（案4・案5）
- **コンテナ／PaaS 固有のネットワーク** ―― AKS の Pod 通信と CNI、Private Link で「提供する」側（案6・案8）
- **観測の運用化** ―― 単発診断から継続監視（Connection Monitor・Traffic Analytics）へ（案7）

## basic との違い（このフェーズの方針）

- **1 ステップ＝1 概念にこだわらない。** basic で基礎を積んだので、関連する複数の概念を 1 ステップにまとめて導入します（例：「VPN ＋ IPsec ＋ BGP」を 1 ステップで）。
- **まず一般概念、次に Azure 実装。** どのステップも「ネットワーク一般としてどんな問題を解くか」を先に置き、Azure のリソースはその実現例として扱います。
- **学習手法は basic と同じ。** ―― 「設定を出し入れして、通信の変化から因果を確かめる」。扱う対象が高度になっても、切り分けの考え方は変わりません。

## 到達できるレベル

- basic の基本リソースだけでは届かない構成（ハイブリッド・広域・エッジ防御・コンテナ）を、**一般概念と Azure 実装の両面から語れる**
- マネージドサービス（Virtual WAN・Front Door・Bastion 等）と手組み構成を **対比し、責務・コスト・運用の違いを理解して選べる**
- 高度な構成でも、**どのスイッチが通信を成立／遮断させているかを出し入れで切り分けて説明できる**

## 進め方（おすすめの順序）

`PLAN.md` の「おすすめの進め方」に沿って、3 グループに分けて進めます。

1. **入口防御グループ（★★）**：案4（WAF/TLS）→ 案5（Front Door/DDoS）
   - step9/10 の直接の発展で、基礎からの飛躍が小さく入りやすい。
2. **接続・経路グループ（★★★）**：案1（VPN/IPsec/BGP）＋案2（動的ルーティング）→ 案3（Virtual WAN）
   - step2/3 の発展。basic がスコープ外と明言した「ハイブリッド・BGP・広域」を正面から扱う中核。
3. **発展・総まとめグループ（★★/★★★）**：案7（観測の運用化）→ 案8（Private Link 提供側）→ 案6（AKS で総ざらい）

## 学び方の特徴（basic と共通）

- 構築は Bicep、コマンドのまとめは just、Bicep でカバーしない部分は Azure CLI を利用
- 各ステップは新しい用語・概念を `KNOWLEDGE.md` に、構成図を `MERMAID.md` に整理
- 共通する学習手法は **「設定を出し入れして、通信の変化から因果を確かめる」** こと

> 構築・実行はユーザー自身が行います（AI が Azure 上で実行することはありません）。
> コスト注意：Front Door・Application Gateway・VPN Gateway など、このフェーズで扱うリソースには時間/容量課金が伴います。検証が終わったら各ステップの `just cleanup` で必ず削除してください。
