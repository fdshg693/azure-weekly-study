# Step 7 で登場した用語・概念

このステップで**新たに**登場した用語・概念をまとめます。
VNet・サブネット・NSG・VM・プライベート IP・Run Command による疎通確認など、
Step 1〜6 でカバーした内容は前提として含めません。

## ネットワーク全般の概念

### 名前解決（DNS / name resolution）
* DNS は「**名前 → IP アドレスの対応表**」を引く仕組み。`vm-b.corp.internal` のような名前を
  問い合わせると、対応する IP（`10.0.1.5`）が返ってくる。
* これまで（Step1〜6）は宛先を**プライベート IP の直打ち**で指定してきた。IP 直打ちには弱点がある：
  * 人が覚えにくい・取り違えやすい。
  * VM を再作成したり動的割り当てだったりすると **IP が変わって参照が壊れる**。
  * 役割（"アプリのDB"）と数字（10.0.1.5）が結びつかない。
* 名前解決を挟むと、**名前は固定のまま、裏側の IP だけ差し替え**られる。これが「名前で到達する」価値。
* 名前解決そのものは**経路（到達性）とは別物**。名前が引けなくても IP が分かれば通信は届くし、
  逆に名前が引けても経路が無ければ届かない。本ステップはこの 2 つを切り分けて理解する。

### A レコード / FQDN / ゾーン
* **A レコード**：ある名前を IPv4 アドレスに対応づける 1 行（例: `vm-b → 10.0.1.5`）。
* **ゾーン（zone）**：1 つのドメイン名空間（例: `corp.internal`）に属するレコードの集まり。
* **FQDN（完全修飾ドメイン名）**：レコード名とゾーン名をつないだ完全な名前（例: `vm-b.corp.internal`）。
* このステップでは、ゾーン `corp.internal` の中に `vm-a` / `vm-b` / `app` といった A レコードを持つ。

### 自動登録（auto-registration）と手動レコード
* **自動登録**：VM が起動時に**自分のホスト名を自動でゾーンに登録**する。台数が増えても手作業が要らない。
  登録される名前は VM の **computerName（OS のホスト名）**。
* **手動レコード**：人が**任意の別名**を IP に向けて手で作る（例: 用途を表す `app` を特定 VM に向ける）。
  ホスト名と無関係に「分かりやすい名前」を付けられ、向き先も人が決める。
* 両者は併存できる。自動登録で「機械的な実体名」を、手動レコードで「意味のある別名」を持つ、という使い分け。

## Azure 固有の用語（上記概念の具体例）

### Private DNS Zone（プライベート DNS ゾーン）
* **VNet 内だけで通用する独自の DNS ゾーン**。インターネットのパブリック DNS には一切公開されない。
* **グローバルリソース**（`location: 'global'`）。特定リージョンに属さない。
* ゾーン名はドメイン名の形であれば任意（本ステップは学習用に `corp.internal`）。
  既存のパブリックドメインと衝突しない名前にするのが無難。

### VNet リンク（virtual network link）と `registrationEnabled`
* Private DNS Zone は、**VNet と「リンク」して初めてその VNet 内から引ける**ようになる。
  リンクが無い VNet からは、同じゾーンも見えない（名前を引けない）。
* リンクの **`registrationEnabled: true`** で、その VNet 内の VM の**自動登録**が有効になる。
  * 自動登録できる（registration ON の）リンクは、**1 ゾーンにつき 1 つだけ**。
  * `false`（または既定）でも、そのゾーンの**解決**（既存レコードを引く）はできる。
    つまり「**登録する VNet**」と「**解決だけする VNet**」を分けられる。
* 本ステップの `just unlink` / `just link` はこのリンクを削除/再作成して、
  「名前解決が効くのはリンクがあるとき」だと切り分ける。

### VM が Private DNS Zone を引ける理由（Azure 提供 DNS: 168.63.129.16）
* リンク済み VNet 内の VM は、既定で **Azure 提供 DNS（168.63.129.16）** を使う。
  この DNS がリンクされた Private DNS Zone を解決してくれるので、VM 側に特別な DNS 設定は要らない。
* だから VM 上で `getent hosts vm-b.corp.internal` や `ping vm-b.corp.internal` がそのまま通る。

### 静的プライベート IP（このステップでの使い方）
* vm-a=10.0.1.4 / vm-b=10.0.1.5 を**静的**に固定し、「名前 → IP」の対応を分かりやすくした。
  （Azure はサブネット先頭 `.0〜.3` を予約するので `.4` から使う）
* 本来 Private DNS の自動登録は動的 IP でも機能するが、学習では IP を固定した方が
  「app → 10.0.1.5」のような手動レコードの意味が見えやすい。

## このステップの要点
* **名前解決（DNS）= 名前 → IP の対応表**。IP 直打ち（Step1〜6）の弱点（変わる・覚えにくい）を解消する。
* **Private DNS Zone** は VNet 内だけで有効な独自ゾーン。**VNet リンク**があって初めてその VNet から引ける。
* 名前と IP の対応は **自動登録**（VM のホスト名）と **手動レコード**（人が決める別名）の 2 通りで持てる。
  自動登録は `registrationEnabled: true` のリンク（ゾーンにつき 1 つ）で有効になる。
* `unlink` → 名前解決は失敗するが **IP では届く**／`link` → 再び名前で届く。
  この対比で、名前解決を担っているのが Private DNS Zone（とリンク）だと切り分けられる
  （NSG/UDR/NAT GW の出し入れと同じ検証手法）。
* この名前解決の考え方は **候補 E（Private Endpoint / Private Link）** で「サービス名をプライベート IP に
  解決させる」前提になる。
