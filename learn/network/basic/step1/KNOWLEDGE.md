# Step 1 で登場した用語・概念

このステップで新たに登場したネットワークの用語・概念をまとめます。
まず一般的なネットワークの考え方を整理し、Azure 固有の用語はその具体例として補足します。

## ネットワーク全般の概念

### IP アドレスと CIDR 表記
* ネットワーク上の機器を識別するための住所。
* `10.0.0.0/16` のような **CIDR 表記** は、「ネットワークの範囲（アドレス空間）」を表す。
  * `/16` はIPアドレスの先頭16ビットがネットワーク部であることを示し、`10.0.0.0` 〜 `10.0.255.255` の約6.5万個のアドレスを含む。
  * `/24` なら先頭24ビットがネットワーク部で、`10.0.0.0` 〜 `10.0.0.255` の256個のアドレスを含む。
* 数字が小さいほど範囲が広く、大きいほど範囲が狭い。

### プライベート IP とパブリック IP
* **プライベート IP**: 組織内・閉じたネットワーク内でのみ通用する住所（例: `10.x.x.x`）。インターネットからは直接届かない。
* **パブリック IP**: インターネット上で一意な住所。外部からアクセスするために必要。

### サブネット
* 大きなネットワーク（VNet）を用途ごとに分割した小さなネットワークの区画。
* アクセス制御やルーティングをサブネット単位で管理しやすくする。

### ファイアウォール / アクセス制御ルール
* 「どの通信を許可し、どの通信を拒否するか」を定義する仕組み。
* 送信元・宛先・ポート・プロトコルなどの条件でルールを記述する。

### ICMP と ping
* **ICMP**: 通信の到達確認やエラー通知に使われるプロトコル。
* **ping**: ICMP を使って「相手に届くか」を確認する基本的な疎通確認コマンド。

## Azure 固有の用語（上記概念の具体例）

### 仮想ネットワーク (VNet / Virtual Network)
* Azure 上に作るプライベートなネットワーク空間。アドレス空間（CIDR）を持つ。
* Azure で VM をネットワークに参加させるには VNet とサブネットが**必須**。

### サブネット (Subnet)
* VNet 内を分割した区画。VM の NIC はいずれかのサブネットに所属する。

### ネットワークセキュリティグループ (NSG / Network Security Group)
* Azure におけるアクセス制御ルール（ファイアウォール）の実体。
* **受信 (Inbound) / 送信 (Outbound)** のセキュリティルールを持つ。
* **サブネット**または**NIC**に関連付けて適用する。
  * このステップでは **サブネットに対して NSG を関連付け**ている。
* ルールには優先度 (priority) があり、数値が小さいルールほど先に評価される。
* ルールの変更は即座に反映され、デプロイし直さなくても通信の許可・拒否が切り替わる。

### パブリック IP アドレス (Public IP)
* VM に外部（インターネット）からアクセスするために割り当てる Azure リソース。

### ネットワークインターフェース (NIC / Network Interface)
* VM をネットワーク（サブネット）に接続するための仮想的なネットワークカード。
* プライベート IP や（必要なら）パブリック IP、NSG はこの NIC やサブネットに紐づく。

### 仮想マシン (VM / Virtual Machine)
* Azure 上で動かす仮想サーバー。単独では存在できず、VNet・サブネット・NIC を介してネットワークに接続される。

## NSG の設定はなぜこうなっているのか

`main.bicep` の NSG (`nsg-minimal`) には、2つの受信ルールが定義されています。
それぞれの設定項目が「なぜその値なのか」を理解します。

```
ルール1: Allow-ICMP-Inbound   priority 100  Allow  Inbound  Icmp  port:*   src:*  dst:*
ルール2: Allow-SSH-Inbound    priority 110  Allow  Inbound  Tcp   port:22  src:*  dst:*
```

### 各設定項目の意味と意図

| 項目 | 値 | なぜこの値か |
|------|----|------------|
| `direction` | **Inbound（受信）** | 制御したいのは「外から VM へ入ってくる通信」だから。ping も SSH も外部から VM へ届く通信なので Inbound で許可する。（VM から出ていく Outbound は既定で許可されているため、今回は触れていない） |
| `access` | **Allow（許可）** | 既定では受信は拒否されている（後述）。明示的に Allow しない限り通らないため、通したい通信を Allow で開ける。 |
| `protocol` | **Icmp / Tcp** | ping は ICMP、SSH は TCP という別プロトコル。NSG はプロトコル単位でルールを分けるため、2つのルールに分かれている。 |
| `destinationPortRange` | **\*（ICMP）/ 22（SSH）** | SSH は TCP の **22番ポート**で待ち受けるため `22` を指定。ICMP には「ポート」という概念がないため `*`（指定なし）にしている。 |
| `sourcePortRange` | **\*** | 送信元のポートは通信のたびに OS が動的に決める（任意）。固定できないため `*`（すべて）にする。 |
| `sourceAddressPrefix` | **\*（どこからでも）** | 学習用に、どの IP からのアクセスも許可している。本番では自分の IP やオフィスの IP に絞るのが安全。 |
| `destinationAddressPrefix` | **\*** | 宛先（VM 側）を限定しないという意味。このサブネット内のリソース宛なら対象になる。 |
| `priority` | **100 / 110** | 数値が小さいルールほど先に評価される（後述）。 |

### なぜ ICMP と SSH の2つを許可するのか
* **ICMP**: このステップの主目的である ping（疎通確認）を成立させるため。`deny-ping` / `allow-ping` で切り替える対象もこのルール。
* **SSH (TCP 22)**: VM に**ログインして操作できる経路**を確保するため。ICMP を Deny にしても SSH は別ルールなので生き続け、VM にアクセスできなくなる事故を防げる。

### priority（優先度）はなぜ 100 / 110 なのか
* NSG のルールは **priority の小さい順に評価**され、最初に一致したルールで「許可／拒否」が決まる（以降のルールは見ない）。
* 設定可能な範囲は **100〜4096**。100 から始めているのは最小値付近で、かつ後からルールを差し込めるよう **10 ずつ空けて**いる（100 と 110 の間に 101〜109 を追加できる）。
* ICMP(100) を SSH(110) より先に置いているが、両者は宛先（プロトコル）が異なるため評価順による衝突はない。「複数ルールがある時は番号順に見る」という仕組みを体感するための並びと考えてよい。

### 明示的に Allow しないと通らない理由（既定ルール）
* NSG には、利用者が書いたルールとは別に、**既定のルール (Default rules)** が裏側に存在する。
  * `AllowVnetInBound` (65000): VNet 内部どうしの通信は許可
  * `AllowAzureLoadBalancerInBound` (65001): Azure のロードバランサーからは許可
  * **`DenyAllInBound` (65500): 上記以外のすべての受信を拒否**
* これらの priority は 65000 番台で、利用者ルール（100〜4096）より**後に評価**される。
* つまり、インターネットからの ICMP/SSH は最後の `DenyAllInBound` に行き着いて**拒否される**のが既定動作。だからこそ、priority 100/110 の Allow ルールを**先に**置いて明示的に通している。
* `deny-ping` で ICMP の Allow ルールを Deny に変えると、ICMP は既定の `DenyAllInBound` を待たずに priority 100 の段階で拒否される。

## ping はどのように VM へ到達するのか（通信経路）

`just test-ping` は、あなたのローカル PC から VM のパブリック IP へ ICMP (ping) を送ります。
このパケットが VM に届くまでには、以下の経路をたどります。各段階で「何がそれを可能にしているか」を対応づけて理解します。

```
[ローカルPC]
   │  ① 宛先 = VM のパブリック IP (例: 20.x.x.x) へ ICMP を送信
   ▼
[インターネット]
   │  ② Azure のパブリック IP までルーティングされる
   ▼
[パブリック IP リソース (pip-vm-minimal)]
   │  ③ NIC に紐づくパブリック IP がパケットを受け取り、
   │     宛先をプライベート IP (10.0.0.x) へ変換 (DNAT) して内部へ渡す
   ▼
[サブネット (subnet-default / 10.0.0.0/24) の境界]
   │  ④ サブネットに適用された NSG が受信ルールを評価
   │     → Allow-ICMP-Inbound (priority 100) に一致 → 通過を許可
   ▼
[NIC (nic-vm-minimal)]
   │  ⑤ プライベート IP を持つ NIC がパケットを受け取る
   ▼
[VM (vm-minimal)]
      ⑥ OS が ICMP Echo Request を受信し、Echo Reply を返す
         （応答は逆の経路をたどってローカル PC へ戻る）
```

### 各段階で何がそれを可能にしているか

| 段階 | 可能にしているリソース / 設定 | 役割 |
|------|------------------------------|------|
| ① 宛先の指定 | **パブリック IP (`pip-vm-minimal`)** | インターネット上で VM を指し示す一意の住所を提供する |
| ② インターネット経由の到達 | パブリック IP（Standard SKU・静的割り当て） | Azure のエッジまでパケットを届ける |
| ③ 外→内へのアドレス変換 | **パブリック IP と NIC の関連付け** | パブリック IP 宛のパケットを、VM のプライベート IP へ転送する |
| ④ 通信の許可判定 | **NSG (`nsg-minimal`) のサブネット適用** | 受信ルールを評価。`Allow-ICMP-Inbound` があるため ICMP が通る |
| ⑤ ネットワークへの接続 | **NIC (`nic-vm-minimal`) とサブネットの所属** | VM を VNet/サブネットに参加させ、プライベート IP で受信する |
| ⑥ 応答の生成 | **VM (`vm-minimal`) の OS** | ICMP Echo Request に対して Echo Reply を返す |

### このことから分かること
* **パブリック IP が無ければ** ①〜③が成立せず、インターネットからは VM に到達できない（Step 2 の VM2 がこれに該当）。
* **NSG のルールが鍵**: `just deny-ping` で `Allow-ICMP-Inbound` を Deny に変えると、④の段階でパケットが破棄され、ping がタイムアウトする。VM や OS は一切変更していないのに通信が止まるのは、サブネット境界の NSG がブロックしているため。
* **SSH (TCP 22) も同様の経路**をたどる。NSG に `Allow-SSH-Inbound` があるため、ping を Deny にしても SSH では接続できる（プロトコル単位でルールが独立しているため）。

## NIC とは何か、なぜ経路に登場するのか

経路図を見ると「パブリック IP」と「NIC」が別々の段階（③と⑤）に現れるため、
**NIC が経路上に2つあるように見えますが、実際には NIC は1つだけ**です。
ここを正しく理解するのがこのステップの要点のひとつです。

### なぜ NIC が必要なのか
* VM は**ネットワークに直接つながらない**。VM と VNet/サブネットの間には必ず NIC（仮想ネットワークカード）が挟まる。
* NIC は VM の「ネットワークへの唯一の接続口」であり、以下の情報を**まとめて保持**している。
  * 所属するサブネット（＝プライベート IP の出どころ）
  * プライベート IP アドレス（`10.0.0.x` / 動的割り当て）
  * （任意で）パブリック IP アドレスとの関連付け

### パブリック IP と プライベート IP は「同じ1つの NIC」に乗っている
`main.bicep` の NIC を見ると、1つの `ipConfigurations`（`ipconfig1`）の中に、
**パブリック IP** と **サブネット（＝プライベート IP）** が両方ぶら下がっています。

```
NIC (nic-vm-minimal)
└─ ipConfigurations[0] = ipconfig1
     ├─ publicIPAddress  → pip-vm-minimal (20.x.x.x)   ← 外向きの顔
     └─ subnet           → subnet-default (10.0.0.x)   ← 内向きの顔
```

つまり、

* **パブリック IP** は独立した「箱」ではなく、**この NIC に紐づいた属性**にすぎない。
* 経路図の③（パブリック IP）と⑤（NIC）は、**物理的には同じ1つの NIC を指している**。
  * ③ = NIC の「外向きの顔（パブリック IP）」でパケットを受け取る段階
  * ⑤ = NIC の「内向きの顔（プライベート IP）」で VM へ渡す段階
  * 1枚の NIC が外側と内側の2つの住所を持っているため、図では2段階に分けて描いている。

### まとめると経路上の「箱」は1つ
```
インターネット
   │ 宛先: パブリック IP (NIC の外向きの顔)
   ▼
┌───────────── NIC (nic-vm-minimal) ─────────────┐
│  publicIP 20.x.x.x ──(同一NIC内)── privateIP 10.0.0.x │
└────────────────────────────────────────────────┘
   │ プライベート IP で受信
   ▼
  VM (vm-minimal)
```

* NIC が**2つあるわけではなく**、1つの NIC が「パブリック IP」と「プライベート IP」という2つの住所を持っているだけ。
* このため、パブリック IP を NIC から外す（関連付けない）と、その NIC は内向きの顔（プライベート IP）だけになり、インターネットから到達できなくなる。これが Step 2 の VM2 の構成。

## このステップの要点
* VM をデプロイするには VNet とサブネットが前提として必要。
* アクセス制御 (NSG) は**サブネットに適用**されており、ルール変更だけで通信の許可・拒否を切り替えられる。
* ローカル PC → パブリック IP → (NSG 判定) → NIC → VM という経路を通り、各リソースが経路上の異なる役割を担っている。
