# Step 4 で登場した用語・概念

このステップで**新たに**登場した用語・概念をまとめます。
VNet・サブネット・NSG・VM・パブリック IP の有無による隔離・送信元限定（最小権限）など、
Step 1〜3 でカバーした内容は前提として含めません。

## ネットワーク全般の概念

### 踏み台（bastion host / jump box）
* インターネットなど信頼できない側に**唯一公開するホスト**を 1 台に絞り、保護対象（内部のサーバ）へは**この 1 台を経由してだけ**到達させる構成。
* 目的は **攻撃対象領域（attack surface）の縮小**：公開する入口が 1 つなら、監視・更新・アクセス制限をそこに集中できる。内部ホストは公開しないので、インターネットから直接は狙えない。
* 踏み台は「中継して転送する」のではなく、**そこで一度ログインし、そこから次へ接続する**点が、Step3 の NVA（パケットを素通しで転送するルーター）との違い。
  * NVA: ネットワーク層でパケットを**フォワード**する（ログイン不要、L3 の中継）。
  * 踏み台: アプリ層で **SSH 接続を張り直す**（ログインを伴う、人間/鍵の中継点）。

### 多段 SSH と SSH ProxyJump（`ssh -J`）
* 手前のホスト（踏み台）に SSH でつなぎ、**その接続の上を通して**さらに奥のホストへ SSH する方式を多段 SSH という。
* 素朴にやると「踏み台にログイン → 踏み台で `ssh 次のホスト`」と 2 回手で打つことになる。
* OpenSSH の **ProxyJump（`ssh -J 踏み台 最終ホスト`）** は、これを 1 コマンドにまとめる。踏み台に対しては「最終ホストへの TCP 接続を中継する」役割だけをさせ、**SSH の認証・暗号化はローカル PC と最終ホストの間で直接（エンドツーエンド）**行う。
* このとき**秘密鍵はローカル PC から外に出ない**。踏み台には鍵を置かなくてよい（踏み台が乗っ取られても秘密鍵は盗まれない）。これが ProxyJump の安全上の利点。
  * 対比: 古いやり方の「踏み台に秘密鍵をコピーする」「エージェント転送（`-A`）で踏み台に鍵を使わせる」は、踏み台が侵害された場合に鍵が悪用されうるため、可能なら ProxyJump を使うのが安全。
* 踏み台側に必要な設定は実はほとんど無い：sshd が動いていてログインでき、**TCP フォワーディングが許可**されていれば動く。`ssh -J` は踏み台の sshd に「最終ホスト:22 への TCP 接続を中継して」と頼む（direct-tcpip チャネルを開く）ため、踏み台の **`AllowTcpForwarding`** が有効である必要がある（Ubuntu の既定は `yes`）。
  * 一方 **最終ホスト（private VM）側には特別な設定は不要**：private VM は中継をせず、普通の SSH 接続を受けるだけだから。だから `AllowTcpForwarding` を気にするのは**踏み台だけ**でよい。
  * 古い `ProxyCommand` 方式と違い、踏み台に `nc`（netcat）を入れる必要もない。OpenSSH 標準機能だけで完結する。

### 公開鍵認証（SSH key authentication）
* SSH のログインを**パスワードではなく鍵ペア**（秘密鍵=手元、公開鍵=サーバ）で行う方式。
* 公開鍵をサーバの `~/.ssh/authorized_keys` に置き、接続時に「対応する秘密鍵を持っていること」を暗号的に証明する。秘密鍵自体はネットワークに流れない。
* 踏み台のように外部に開くホストでは、総当たり攻撃に弱い**パスワード認証を無効化**し、公開鍵認証だけにするのが定石。
* 多段 SSH では、**同じ公開鍵を各ホップのサーバに入れておけば、1 本の秘密鍵で全ホップを認証**できる（本ステップは踏み台・private VM の両方に同じ公開鍵を入れている）。

### 入口の最小公開と内部の最小権限（二層で絞る）
* **入口（踏み台）**: インターネットに開くのは踏み台だけ。さらにその踏み台への SSH 元を**自分の IP だけ**に絞る＝「公開はするが、開ける相手は最小限」。
* **内部（private VM）**: 公開しない（パブリック IP なし）。加えて、SSH を受け付ける送信元を**踏み台のいるサブネットだけ**に絞る。
* 「公開 IP を持たない」＝直接は届かない、という Step2/3 の隔離に、**「経路上も踏み台からだけ」**という NSG の制限を重ねている。成功した接続が確実に踏み台経由であることを保証できる。

## Azure 固有の用語（上記概念の具体例）

### 公開ゾーン／非公開ゾーンをサブネットで分ける
* 1 つの VNet を **2 つのサブネット**に分け、役割で隔離している。
  * `subnet-bastion`（10.0.0.0/24）= 踏み台（パブリック IP あり）を置く公開ゾーン。
  * `subnet-private`（10.0.1.0/24）= 保護対象（パブリック IP なし）を置く非公開ゾーン。
* サブネットごとに NSG を付け替えられるため、「公開ゾーンは自分の IP からの SSH を受ける」「非公開ゾーンは踏み台サブネットからの SSH だけ受ける」と、ゾーンごとに別ポリシーを当てられる。
* 同一 VNet 内なのでサブネット間はシステムルートで到達可能。**到達できるかどうかは NSG が決めている**点が要点（Step3 で学んだ「経路（ルーティング）」と「許可（NSG）」の分離がここでも効く）。

### NSG で SSH の送信元を「自分のグローバル IP」に固定する
* `nsg-bastion` の `Allow-SSH-From-MyIp` は `sourceAddressPrefix` を `<あなたのIP>/32` にしている。`/32` は「その 1 アドレスだけ」。
* グローバル IP は回線によって変わるため、`just deploy` 時に `https://api.ipify.org` から取得して埋め込み、変わったら `just update-myip` で貼り替える運用にしている。
* Step2 では「保護対象 VM の送信元をサブネットに絞った」が、Step4 では**入口そのものの送信元を 1 IP に絞る**。最小権限を「公開する入口」に適用するのが踏み台の肝。

### private VM に踏み台サブネットからの SSH だけを許可する（`sourceAddressPrefix: '10.0.0.0/24'`）
* `nsg-private` の `Allow-SSH-From-Bastion` は、SSH の送信元を踏み台サブネット `10.0.0.0/24` に限定している。
* ProxyJump で踏み台を経由すると、private VM から見た **TCP 接続の送信元は踏み台の IP（10.0.0.x）** になる。だからこのルールでジャンプが通る。
  * （Step3 の NVA は NAT せず元の送信元 IP を保っていたが、踏み台は SSH 接続を**張り直す**ため、private VM から見える送信元は踏み台になる。ここが NVA 中継との違い。）
* `just lock-private` でこのルールを消すと、踏み台からも private VM へ SSH できなくなり、`just test-jump` が失敗する。`just unlock-private` で復活する。「入れていたのは NSG の許可があったから」だと確認できる。

### Bicep での公開鍵認証の指定（`linuxConfiguration`）
* VM の `osProfile.linuxConfiguration` で以下を指定し、公開鍵認証のみにしている。
  * `disablePasswordAuthentication: true` … パスワードログインを禁止。
  * `ssh.publicKeys[].keyData` … サーバに登録する公開鍵（`bastion_key.pub` の中身）。
  * `ssh.publicKeys[].path` … 公開鍵を書き込む先（`/home/azureuser/.ssh/authorized_keys`）。
* `adminPassword` は不要になる（Step1〜3 では `newGuid()` を使っていたが、Step4 は鍵認証なので渡していない）。
* 同じ公開鍵を踏み台・private VM の両方に入れているため、ローカルの 1 本の秘密鍵で両ホップを認証できる。

### Bicep の cloud-init（`customData`）で踏み台の sshd 設定を変える
* 踏み台 VM の `osProfile.customData` に **cloud-init**（`base64()` でエンコードした `#cloud-config`）を渡し、起動時に sshd の設定を投入している。
  * `write_files` で `/etc/ssh/sshd_config.d/10-allow-tcp-forwarding.conf` に `AllowTcpForwarding yes` を書き、`runcmd` で `systemctl restart ssh` して反映。
  * Ubuntu 22.04 の `sshd_config` は冒頭で `Include /etc/ssh/sshd_config.d/*.conf` を読むため、ドロップインファイルを置くだけで設定が効く（本体ファイルを直接編集しない安全なやり方）。
* `AllowTcpForwarding` の既定値は `yes` なので挙動上は変わらないが、「`ssh -J` の中継に必要な設定」という意図を構成として明示するために固定している。**この設定は踏み台にだけ適用**（private VM は中継しないため不要）。

### Azure Bastion（マネージドの踏み台サービス）※補足
* 今回は SSH と安価な VM で「踏み台」という**概念そのもの**を手で組んだが、Azure には同名のマネージドサービス **Azure Bastion** がある。
* 専用サブネット `AzureBastionSubnet`（/26 以上）が必須で、対象 VM に**パブリック IP を付けずに**ブラウザや `az` から接続できる。踏み台 VM 自体の OS 管理や SSH 公開を自分で持たなくてよいのが利点。
* 一方で時間課金（常時稼働コスト）があり、`ssh -J` のような多段 SSH の手触りは薄い。**「踏み台＝概念」「Azure Bastion＝それを Azure が肩代わりするマネージド実装の一つ」**という関係で捉えるとよい。

## なぜ private VM に「踏み台越しでだけ」入れるのか（経路の全体像）

`ローカル PC` から `vm-private (10.0.1.x)` への SSH を、`ssh -J azureuser@<踏み台> azureuser@<private>` で行う例。

```
[ローカル PC]  (秘密鍵 bastion_key はここから出ない)
   │ ① 踏み台へ SSH(22)
   │    → nsg-bastion: 送信元が「自分のグローバル IP」か判定 ⇒ 一致なら許可
   ▼
[vm-bastion 10.0.0.4 / public IP あり]
   │ ② ProxyJump: 踏み台は「private VM:22 への TCP 接続」を中継するだけ
   │    （踏み台はログインの中継点。SSH 認証はこの上を通って PC↔private で直接行う）
   ▼
[subnet 内のルーティング]
   │ ③ 同一 VNet 内なのでシステムルートで subnet-private へ到達可能
   ▼
[nsg-private の受信判定]
   │ ④ 接続の送信元は踏み台の IP(10.0.0.x) ＝ 踏み台サブネット
   │    → Allow-SSH-From-Bastion (src 10.0.0.0/24) に一致 ⇒ 許可
   ▼
[vm-private 10.0.1.x / public IP なし]
   ⑤ PC が持つ秘密鍵で認証成立（公開鍵が authorized_keys にある）
      → ログイン成功。秘密鍵は終始ローカルにあり、踏み台には渡っていない。
```

### 各段階で何がそれを可能にしているか

| 段階 | 可能にしている設定 | これが無いと |
|------|-------------------|-------------|
| ① 踏み台への到達 | **nsg-bastion の Allow-SSH-From-MyIp**（自分の IP /32） | 自分の IP からでも届かない（IP がズレると `update-myip` が必要） |
| ② 中継 | `ssh -J`（ProxyJump）＋ **踏み台の `AllowTcpForwarding yes`**（cloud-init で明示） | 踏み台に入って手で `ssh` し直す必要があり、鍵の扱いも面倒・危険になりがち。踏み台で forwarding が無効だとトンネルが張れずジャンプ自体が失敗する |
| ③ 内部到達 | 同一 VNet のシステムルート | サブネット間の経路が無く届かない |
| ④ private への受信許可 | **nsg-private の Allow-SSH-From-Bastion**（src 10.0.0.0/24） | 踏み台からでも SSH が拒否される（`lock-private` で再現できる） |
| ⑤ 認証 | 両 VM に入れた**同じ公開鍵** + ローカルの秘密鍵 | 鍵が一致せずログインできない |

## このステップの要点
* **踏み台**は「インターネットに開くホストを 1 台に絞り、内部はそこ経由でだけ触る」構成。攻撃対象領域を小さくするのが狙い。
* **`ssh -J`（ProxyJump）** なら多段 SSH を 1 コマンドで行え、**秘密鍵はローカルから出ない**（踏み台に鍵を置かない＝安全）。
* 踏み台は NVA と違い、パケットを素通しで転送するのではなく **SSH 接続を張り直す**。だから private VM から見た送信元は踏み台の IP になり、`nsg-private` は踏み台サブネットを許可すれば通る。
* 守りは二層：**入口（踏み台）は自分の IP だけに、内部（private VM）は踏み台サブネットだけに**送信元を絞る。
* private VM に**パブリック IP を付けない**ことで、`test-direct-fail`（直接は失敗）と `test-jump`（踏み台経由は成功）の対比により、到達経路が確実に踏み台経由であることを示せる。
* `just lock-private` / `unlock-private` で NSG の許可を出し入れすると通信が止まる／復活することから、「入れていたのは NSG の許可があったから」だと確認できる。
* Azure には同概念のマネージド実装 **Azure Bastion** もある（本ステップは概念を手で理解するために自前の踏み台 VM を採用）。
