# KNOWLEDGE.md (Step12)

このステップで新たに登場した用語や概念について解説します。
（NSG・既定ルール・ping/SSH 疎通の基礎は Step1/2/4、UDR・経路の基礎は Step3 を参照）

## Network Watcher
リージョン単位で提供される、ネットワークの**監視・診断・ログ**の集合サービスです。サブスクリプション内で
リージョンごとに自動で有効化され（`NetworkWatcher_<region>`）、IP Flow Verify・接続トラブルシュート・
Next Hop・NSG フローログ・パケットキャプチャなどの機能を束ねています。これまで ping や SSH で
**体感**していた疎通を、**判定・経路・記録**として観測するための入り口です。

## IP Flow Verify
「送信元 IP:ポート／宛先 IP:ポート／プロトコル／方向」を指定すると、対象 NIC に適用された **NSG が
その通信を許可するか拒否するか**、そして **どのセキュリティルールが効いたか（ruleName）**を返します。
特徴は **実際にパケットを流さない**こと——構成（NSG ルール）の評価結果だけを返すので、安全に
「通るはず？ なぜ？」を確認できます。許可は明示ルール（例: `Allow-SSH-From-Vnet`）で、拒否は多くの場合
既定ルール（`DefaultRule_DenyAllInBound`）で起きる、という構図がそのまま読み取れます。

## 接続トラブルシュート（connectivity test / test-connectivity）
送信元 VM 内の **Network Watcher Agent** が実際に宛先へパケットを出し、**到達可否
（connectionStatus: Reachable / Unreachable）・経路上のホップ・遅延・ドロップ箇所**を返します。
IP Flow Verify が「構成上どう評価されるか（流さない）」を見るのに対し、こちらは「**実際に届くか**
（流す）」を経路つきで見ます。両者を併せると、「構成では許可なのに届かない」「構成で拒否だから届かない」
といった切り分けができます。

## Network Watcher Agent（VM 拡張機能）
接続トラブルシュートやパケットキャプチャのように、**VM の内側から実際に通信を出す/観測する**機能は、
対象 VM にこのエージェント拡張（Linux なら `NetworkWatcherAgentLinux`）が必要です。
一方、IP Flow Verify や Next Hop は NSG・ルートの**評価**なので、エージェントが無くても動きます
（本ステップでは両 VM にまとめて入れています）。

## Next Hop（経路診断）
指定した送信元 VM から、ある宛先 IP へ向かうパケットが **次にどこへ転送されるか**を返します。
返る `nextHopType` の例:
- `VnetLocal` … 同じ VNet 内（ローカル配送）
- `Internet` … インターネットへ直接
- `VirtualAppliance` … UDR で NVA/Firewall に曲げられている（Step3/11 の構成ならこれが現れる）
- `VnetPeering` … ピアリング相手の VNet へ
「経路を成立/変更させているのがピアリングなのか UDR なのか」を、推測ではなく**判定**として確認できます。

## NSG フローログ（NSG Flow Logs）
NSG を通過した **許可/拒否トラフィック（5 タプル＋許可/拒否の別）**を、指定した Storage アカウントへ
JSON として**記録**する機能です。ping のようにその場で消える確認と違い、**後から見返せる通信ログ**が
残ります。`insights-logs-networksecuritygroupflowevent` というコンテナに、時刻・NSG 単位で blob が
書き込まれます（反映には数分のラグがあります）。

## トラフィック分析（Traffic Analytics・参考）
本ステップでは扱いませんが、NSG フローログを **Log Analytics ワークスペース**に集約すると、
上位の通信元/宛先・許可/拒否の傾向・地理情報などを**可視化・集計**できます。生のフローログ（blob）を
人が読む代わりに、ダッシュボードで俯瞰するための上位機能だと押さえておきます。

## 「流さずに判定」と「流して観測」の使い分け
- **構成が正しいかを確かめたい** → IP Flow Verify / Next Hop（パケットを流さず、NSG・ルートの評価を見る）
- **実際に届くか・どこで落ちるかを確かめたい** → 接続トラブルシュート（パケットを流し、経路を見る）
- **後から傾向を見返したい・記録を残したい** → NSG フローログ
Step1〜11 で「リソースを出し入れして因果を確かめる」手法はそのままに、確かめる手段を
「体感（ping）」から「観測（診断・記録）」へ引き上げるのが本ステップの位置づけです。
