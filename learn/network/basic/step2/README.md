# Step 2: VNet ピアリングによるネットワーク間通信

このステップでは、2つの独立した仮想ネットワーク (VNet) を作成し、それらを VNet ピアリングで接続して、異なる VNet 間にある仮想マシン (VM) がプライベート IP アドレスで通信できることを確認します。

## 目的
* 複数の VNet を作成し、それぞれにサブネットと VM を配置する構成を学ぶ。
* VNet ピアリング機能を使用して、異なる VNet を接続する方法を学ぶ。
* ピアリングによって、VM 同士がプライベート IP を使って安全に通信できることを確認する。

## 前提条件
* [Azure CLI](https://learn.microsoft.com/ja-jp/cli/azure/install-azure-cli) (事前に `az login` でログイン済みであること)
* [Just](https://github.com/casey/just)

## 構成されるリソース
`main.bicep` ファイルにより、以下のリソースがデプロイされます。
* **リソースグループ**: `rg-network-learn-peering` (東日本リージョン)
* **VNet 1**: `vnet-1` (10.1.0.0/16) / サブネット: `subnet-1` (10.1.0.0/24)
* **VNet 2**: `vnet-2` (10.2.0.0/16) / サブネット: `subnet-2` (10.2.0.0/24)
* **ネットワークセキュリティグループ (NSG)**: 各サブネットに適用 (ICMP, SSH 許可)
  * `nsg-vnet1` (vm1 側): パブリック IP を持つ「入口」のため、送信元は `*`（どこからでも許可）。
  * `nsg-vnet2` (vm2 側): 送信元を **`subnet-1` (10.1.0.0/24) のみに限定**（最小権限）。vm2 が受け取るのは vm1 からの通信だけのため、これによりピアリング経由の通信であることを NSG レベルでも担保しています。
* **仮想マシン**: `vm-1` (VNet 1内、パブリック IP あり), `vm-2` (VNet 2内、**パブリック IP なし**)
  * ※ `vm-2` には意図的にパブリック IP を付与していません。これにより、インターネットからの直接アクセスが不可能であり、VM1 からの ping 応答が間違いなくプライベートネットワーク（VNet ピアリング）経由の通信であることを担保しています。
* **VNet ピアリング**: `vnet-1` と `vnet-2` を相互に接続

---

## なぜピアリングが必要なのか（同一 VNet 内の異なるサブネットとの違い）

このステップの肝は「**別々のネットワーク（VNet）同士は、デフォルトでは互いに通信できない**」という点です。これを理解するために、対照的なケースとして「2つの VM が**同じ VNet 内の異なるサブネット**にいる場合」を整理しておきます。

* **同一 VNet・異なるサブネットの場合（ピアリング不要）**
  * 1つのネットワーク（VNet）はアドレス空間（例: `10.0.0.0/16`）を持ち、サブネットはその内側を区切ったものに過ぎません。
  * 同じネットワーク内であれば、サブネットをまたぐ経路は**自動的に用意されており**、VM 同士は最初からプライベート IP で通信できます（別途の接続設定は不要）。
  * したがって、もし vm1 と vm2 を 1つの VNet の subnet-A / subnet-B に置いていれば、ピアリングを一切設定しなくても疎通します（通信可否を最終的に左右するのは NSG などのアクセス制御だけ）。

* **別々の VNet の場合（このステップ＝ピアリングが必要）**
  * VNet が分かれていると、それぞれが独立したネットワークとして**互いに隔離**されており、サブネット間のような自動経路は存在しません。
  * そのため、明示的に「2つの VNet をつなぐ」設定（＝VNet ピアリング）を行って、初めて相互にプライベート通信ができるようになります。

言い換えると、ピアリングとは「**別々の VNet を、あたかも経路がつながっているかのように振る舞わせる**」仕組みであり、同一 VNet 内のサブネット同士が最初から持っている『つながり』を、ネットワークをまたいで人為的に作り出すものだと捉えると分かりやすいです。

## 実行手順

コマンドはすべてこの `step2` ディレクトリで実行してください。

### 1. リソースのデプロイ
以下のコマンドで、Azure 上にリソースを作成します。VM とピアリングの作成が含まれるため、完了まで数分かかります。
```bash
just deploy
```

### 2. 疎通確認（VNet ピアリングのテスト）
デプロイが完了したら、VM1 から VM2 のプライベート IP に対して ping を実行し、VNet ピアリング経由で通信が成功することを確認します。
(Azure VM Run Command を使用して、VM1 の内部からコマンドを実行します)
```bash
just test-peering
```
> VNet ピアリングが正しく設定されていれば、VM2 (10.2.0.x) からの ping 応答が返ってきます。

### 3. インターネットからの隔離の確認（失敗テスト）
念のため、あなたのローカルPCから VM2 のプライベート IP に対して ping を実行し、**通信が失敗すること** を確認します。
```bash
just test-local-fail
```
> VM2 にはパブリック IP がなくインターネットに公開されていないため、ローカル PC からの ping はタイムアウト（または到達不可）となります。これにより、先ほど成功した VM1 からの ping が、インターネット経由ではなく確実に VNet ピアリング経由のプライベート通信であることが担保されます。

### 4. リソースの削除 (クリーンアップ)
テストが完了したら、以下のコマンドでリソースグループごと削除します。
```bash
just destroy
```
