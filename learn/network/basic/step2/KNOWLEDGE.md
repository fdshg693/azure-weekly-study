# Step 2 で登場した用語・概念

このステップで**新たに**登場した用語・概念をまとめます。
VNet・サブネット・NSG・VM など Step 1 でカバーした内容は含めません。

## ネットワーク全般の概念

### ネットワーク間接続 (ネットワークの相互接続)
* 独立した複数のネットワークを接続し、互いの機器がプライベート IP で通信できるようにする考え方。
* インターネットを経由せず、閉じた経路で安全に通信できる点が利点。
* 接続する際は、双方のアドレス空間（CIDR）が**重複していない**ことが前提となる。
  * 重複していると、どちらのネットワークの宛先か判別できずルーティングできない。
  * このステップでは `10.1.0.0/16` と `10.2.0.0/16` のように重複しない範囲を使用している。

### 同一ネットワーク内（サブネット間）と、別ネットワーク間（VNet 間）の違い
* 1つのネットワークはアドレス空間を持ち、**サブネット**はその内側を区切った単位に過ぎない。
  * 同一ネットワーク内のサブネット同士は、**自動的に経路が用意されている**ため、特別な接続設定なしに最初から相互通信できる（通信可否を左右するのはアクセス制御＝NSG などだけ）。
* 一方、**別々のネットワーク**は互いに隔離されており、サブネット間のような自動経路は存在しない。
  * 別ネットワーク同士を通信させるには、明示的に「2つのネットワークをつなぐ」設定（ネットワーク間接続）が必要になる。
* つまりネットワーク間接続とは、別ネットワーク間に対して「同一ネットワーク内のサブネット同士が最初から持っている『つながり』」を人為的に作り出す仕組みと捉えると理解しやすい。
  * このステップで2つの VM を別々の VNet に置いているのは、まさにこの「自動ではつながらない」状況をあえて作り、接続設定（ピアリング）の役割を浮き彫りにするため。

### プライベート通信の担保（経路の確認）
* 「通信が確かにプライベート経路を通っているか」を確認するには、
  公開経路（インターネット）からのアクセスが**失敗する**ことを併せて確認するとよい。
* 一方の機器をインターネットに公開しない（パブリック IP を付けない）ことで、
  成功した通信がプライベート経路経由であることを論理的に保証できる。

### 最小権限（送信元の限定）
* アクセス制御ルールは「許可する／しない」だけでなく、**「どこからの通信を許可するか（送信元）」まで絞る**とより安全になる。
* 必要な相手からの通信だけを通し、それ以外をすべて遮断する考え方を**最小権限**という。
* 送信元を限定すると、想定外の経路から届いた通信は受け付けられない。
  → 「成功した通信が想定どおりの相手・経路から来たものである」ことを、より強く保証できる。

## Azure 固有の用語（上記概念の具体例）

### VNet ピアリング (Virtual Network Peering)
* 2つの VNet を接続し、相互にプライベート IP で通信できるようにする Azure の機能。
* 接続は**双方向**に設定する必要がある（VNet1→VNet2 と VNet2→VNet1 の両方）。
* ピアリングする VNet 同士はアドレス空間が重複していてはならない。
* インターネットを経由せず、Azure のバックボーンネットワーク内で通信が完結する。

### NSG の送信元限定（`sourceAddressPrefix`）
* Step 1 では NSG の `sourceAddressPrefix` を `*`（どこからでも許可）にしていたが、Step 2 の **nsg2（vm2 を保護）では送信元を `10.1.0.0/24`（subnet-1）に限定**している。
* vm2 が受け取るべき通信は vm1（subnet-1）からの ping だけなので、それ以外の送信元を NSG の段階で拒否する。
* `sourceAddressPrefix` には CIDR（`10.1.0.0/24` など）のほか、`VirtualNetwork` / `Internet` / `AzureLoadBalancer` といった**サービスタグ**も指定できる。今回は最も狭い subnet-1 の範囲を指定している。
* これにより、「vm2 への疎通成功が確実にピアリング経由（subnet-1 から）である」ことを、パブリック IP 不在に加えて **NSG レベルでも二重に担保**している。
* 一方、vm1 を保護する nsg1 はパブリック IP を持つ「入口」のため `*` のまま（外部からのアクセスを受ける想定）。**役割に応じて送信元の絞り方を変える**のがポイント。

### Azure VM Run Command
* VM に SSH などで直接ログインせずに、Azure 経由で VM 内部のコマンドを実行する機能。
* このステップでは、パブリック IP を持つ VM1 の**内部**から VM2 へ ping を実行し、
  VNet ピアリング経由の疎通を確認するために使用する。
* パブリック IP を持たない VM に対しても、VM の内部から操作を行える点が便利。

## VNet ピアリングの設定はどうなっていて、なぜそうしているのか

`main.bicep` では、ピアリングを**2つのリソース**として定義しています。
`vnet1 → vnet2` 向きと `vnet2 → vnet1` 向きの2つです。

```
peering1to2  parent: vnet1  remoteVirtualNetwork: vnet2   (vnet1 から見た vnet2 への接続)
peering2to1  parent: vnet2  remoteVirtualNetwork: vnet1   (vnet2 から見た vnet1 への接続)
```

### なぜ2つ定義するのか（ピアリングは片方向ずつ）
* Azure の VNet ピアリングは、**各 VNet 側にそれぞれ1つずつ**設定する必要がある。
* `vnet1` 側に「vnet2 とつながる」設定を書いても、それだけでは vnet1 から vnet2 を**見にいける**だけ。
* `vnet2` 側にも「vnet1 とつながる」設定を書いて初めて、**双方向の通信が成立**する。
* どちらか一方だけだとピアリングの状態が未接続（`Initiated`）のままになり、通信できない。両方そろって初めて接続済み（`Connected`）になる。
* `parent` でどちらの VNet 側の設定かを指定し、`remoteVirtualNetwork` で相手の VNet を指している。1to2 と 2to1 で parent と remote が入れ替わっているのはこのため。

### 各プロパティの意味と、なぜこの値にしているか

| プロパティ | 設定値 | なぜこの値か |
|-----------|--------|------------|
| `remoteVirtualNetwork` | 相手の VNet の ID | 「どの VNet と接続するか」を指定する必須項目。1to2 は vnet2、2to1 は vnet1。 |
| `allowVirtualNetworkAccess` | **true** | ピアリング先 VNet との通信を許可する。これが今回の目的（VM 間のプライベート通信）そのものなので true。false にすると経路は張られても通信が遮断される。 |
| `allowForwardedTraffic` | **false** | 「相手 VNet から*転送されてきた*（発信元が相手 VNet 外の）通信」を受け入れるか。今回は VM 同士の直接通信だけで、間に転送する機器（NVA など）が無いため不要。 |
| `allowGatewayTransit` | **false** | 自分側の VNet ゲートウェイを相手に貸し出すか。オンプレ接続用ゲートウェイは無いので false。 |
| `useRemoteGateways` | **false** | 相手側のゲートウェイを自分の出口として使うか。相手にゲートウェイが無く、その必要も無いため false。 |

### 設計のポイント
* 今回は「同じ Azure 内の2つの VNet を直結し、VM 同士をプライベート IP で通信させる」最小構成。
  そのため、**通信を許可する `allowVirtualNetworkAccess` だけを true** にし、
  転送やゲートウェイ共有に関する3つのオプション（forwarded / gatewayTransit / useRemoteGateways）はすべて false にしている。
* これらを false にしておくことで、「ピアリングした2つの VNet の直接通信」以外の経路が開かない、最小権限の構成になる。
* アドレス空間が重複しない（`10.1.0.0/16` と `10.2.0.0/16`）ことがピアリング成立の前提。重複していると作成自体が失敗する。

## このステップの要点
* VNet ピアリングにより、別々の VNet にある VM 同士がプライベート IP で通信できる。
* 一方の VM にパブリック IP を付けないことで、成功した ping が確実にピアリング経由（プライベート通信）であることを担保している。
* さらに nsg2 で送信元を subnet-1 に限定する（最小権限）ことで、ピアリング経由であることを NSG レベルでも裏付けている。
* VM 内部からの操作には Azure VM Run Command を利用できる。
