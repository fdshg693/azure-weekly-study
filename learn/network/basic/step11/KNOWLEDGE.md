# KNOWLEDGE.md (Step11)

このステップで新たに登場した用語や概念について解説します。
（ハブ&スポーク・ピアリング・UDR・NVA の基礎は Step3、egress / SNAT・`defaultOutboundAccess` の基礎は Step5 を参照）

## Azure Firewall
ハブに置くマネージドな**ステートフル・ファイアウォール**です。Step3 で自前 VM として手組みした NVA
（OS の `ip_forward` でパケットを素通しするだけ・自分で運用が必要）を置き換えるもので、Microsoft が
冗長性・スケール・パッチ適用を引き受けます。通信の中身（宛先 FQDN、ポート、プロトコルなど）を見て
**許可/拒否（検査・制御）**できる点が、単なる中継（NVA）や出口集約（NAT Gateway）との決定的な違いです。
本ステップでは Standard SKU を使用します（下位の Basic、上位の Premium もあります）。

## AzureFirewallSubnet（専用サブネット）
Azure Firewall は、**予約名 `AzureFirewallSubnet`**・**/26 以上**の専用サブネットに配置する必要があります。
Application Gateway（Step10）が専用サブネットを要求したのと同じく、マネージド機器が内部でアドレスを
消費するためです。このサブネットには NSG や UDR を付けません。

## egress の中央集約（強制トンネリング / forced tunneling）
各スポークが個別にインターネットへ出るのではなく、**すべての外向き通信をハブの 1 か所
（Firewall）に必ず経由させる**設計です。本ステップではスポークの UDR で `0.0.0.0/0`（=あらゆる
宛先＝デフォルトルート）の next hop を Firewall のプライベート IP に向けることで実現します。
Step3 の UDR は「特定スポーク宛て」を NVA に向けましたが、ここでは「全外向き」を Firewall に
集約する点が異なります。出口を 1 か所に絞ることで、検査・ログ・制御をまとめて行えます。

## デフォルトルート（0.0.0.0/0）
「他のどのルートにも一致しない宛先」=実質的に「VNet 外（インターネット側）すべて」を指すルートです。
これを Firewall に向けることで、スポークの外向き通信をまるごと Firewall 経由に矯正できます。
Azure の既定では `0.0.0.0/0` はインターネットへ直接向きますが、UDR で上書きできます。

## ファイアウォールポリシー（Firewall Policy）とルールコレクショングループ
Azure Firewall 本体と「何を許可/拒否するか（ルール）」は分離されています。**ファイアウォールポリシー**に
ルールを定義し、それを Firewall に適用します。ポリシーは複数の Firewall で共有でき、ルールは
**ルールコレクショングループ → ルールコレクション → ルール**という階層で優先度づけして管理します。

## アプリケーションルールと FQDN フィルタリング
**宛先の FQDN（ドメイン名）**で許可/拒否を判断するルールです。「どの IP アドレスか」ではなく
「どのドメインへ向かう通信か」で制御するのが特徴で、CDN のように IP が変動する宛先にも対応できます。
- HTTPS では **SNI（Server Name Indication、接続先サーバ名）** を見て一致判定します。
- HTTP では **Host ヘッダ**を見ます。
- 許可リスト（`targetFqdns`）に無い宛先は既定で拒否されます。

本ステップでは `api.ipify.org` と `ifconfig.me` だけを許可し、それ以外（例: `www.bing.com`）は
遮断されることを確認します。

## ネットワークルール（参考）
アプリケーションルールが「FQDN・HTTP(S)」を対象にするのに対し、**ネットワークルール**は
送信元/宛先 IP・ポート・プロトコル（L3/L4）で制御します。任意の TCP/UDP を IP ベースで通したい
場合に使います。本ステップでは FQDN 制御（アプリケーションルール）に焦点を当てています。

## Firewall による SNAT
出口を通った外向き通信の送信元アドレスは、**Firewall の公開 IP に集約（SNAT）**されます。
外部のサーバから見た送信元はこの 1 つの IP になります。Step5 の NAT Gateway も同じ「出口集約」を
行いますが、NAT Gateway は中身を検査しない SNAT 専用機器であるのに対し、Firewall は
**集約と検査・制御を兼ねる**点が違います。

## DNAT（概念）
本ステップでは扱いませんが、Azure Firewall は外向き（egress）だけでなく、**外から内への着信**を
公開 IP の特定ポートで受けて内部の VM へ転送する **DNAT（宛先 NAT）**も行えます。Firewall は
「出口の検査」と「入口の受け口」の両方を 1 か所で担えるマネージド境界機器だと押さえておきます。
