# KNOWLEDGE.md (Step10)

このステップで新たに登場した用語や概念について解説します。
（L4 負荷分散・バックエンドプール・ヘルスプローブの基礎は Step9 を参照）

## Azure Application Gateway (L7 負荷分散)
OSI 参照モデルの第7層（アプリケーション層、HTTP/HTTPS）で動作するロードバランサ兼リバースプロキシです。
L4 の Load Balancer が「IP アドレス・ポート番号」しか見ないのに対し、Application Gateway は **HTTP リクエストの中身**（URL パス、ホスト名、ヘッダなど）を読み取って振り分け先を決められます。
これにより、パスベース／ホストベースのルーティング、TLS 終端、URL 書き換え、Cookie ベースのセッション維持、WAF（Web Application Firewall、上位 SKU）といった、L4 にはできない高度な機能を提供します。

## リバースプロキシ (Reverse Proxy)
クライアントからのリクエストを受け取り、自身が代理（プロキシ）となって裏側のバックエンドへ中継するサーバです。
クライアントは Application Gateway としか通信せず、バックエンドの台数・IP・構成は隠蔽されます。Application Gateway は本質的にマネージドなリバースプロキシです。

## 専用サブネット (Dedicated Subnet)
Application Gateway は、**他のリソースを置けない専用サブネット**に配置する必要があります。
Gateway のインスタンスがスケール時に IP を消費するため、十分なアドレス空間（一般的に /24 程度）を確保しておきます。本ステップでは `subnet-appgw` を専用に割り当てています。

## リスナー (Listener)
クライアントからの接続を待ち受ける「耳」です。フロントエンド IP・ポート・プロトコル（HTTP/HTTPS）の組み合わせで定義します。
HTTPS リスナーには証明書を紐付けることで、Gateway 上で TLS を終端（復号）できます。本ステップでは HTTP/80 のリスナーのみを構成しています。

## URL パスマップ (URL Path Map) とパスルール (Path Rule)
受け取ったリクエストの **URL パス** を見て、どのバックエンドプールへ送るかを決めるルールの集合です。
- **パスルール**: `/api/*` のようなパターンに一致したら指定のプールへ送る、という個別ルール。
- **既定のプール (defaultBackendAddressPool)**: どのパスルールにも一致しなかったリクエスト（本ステップでは `/` や `/web/`）の送り先。

本ステップでは「`/api/*` → `api-pool`、それ以外 → `web-pool`」というパスマップを構成しています。

## ルーティング規則 (Request Routing Rule)
リスナー・パスマップ（またはバックエンドプール）・HTTP 設定を結びつけ、「この入口で受けたら、このルールで、このバックエンドへ送る」という対応関係を確定させるものです。
パスベースルーティングを行う場合は `ruleType: 'PathBasedRouting'` とし、リスナーと URL パスマップを紐付けます。

## バックエンド HTTP 設定 (Backend HTTP Settings)
Gateway がバックエンドへリクエストを転送する際の挙動（宛先ポート、プロトコル、タイムアウト、Cookie ベースアフィニティの有無など）を定義します。
L4 の「負荷分散規則」に相当しますが、HTTP レイヤならではの設定項目を持つ点が異なります。

## TLS 終端 (TLS Termination) — 概念
Gateway 上で HTTPS を復号する処理です。中身（パスやヘッダ）を読めるからこそ L7 ルーティングが成立します。
復号した後、バックエンドへは HTTP で送る（オフロード）か、再暗号化して HTTPS で送る（エンドツーエンド TLS）かを選べます。本ステップでは HTTP のみの構成のため終端は行いませんが、L7 機器の重要な役割として押さえておきます。
