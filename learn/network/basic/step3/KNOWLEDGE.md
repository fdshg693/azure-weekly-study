# Step 3 で登場した用語・概念

このステップで**新たに**登場した用語・概念をまとめます。
VNet・サブネット・NSG・VM・ピアリングなど Step 1 / Step 2 でカバーした内容は含めません。

## ネットワーク全般の概念

### ルーティングと経路表（ルートテーブル）
* 機器は、宛先 IP を見て「次にどこへ送るか（next hop）」を**経路表**に従って決めている。
* 経路表には「この宛先範囲（CIDR）への通信は、この next hop へ」という対応が並ぶ。
* 宛先に一致する経路が複数あるときは、**より具体的な（プレフィックスが長い）経路**が優先される（最長一致）。
* 該当する経路が無い宛先は、最後にデフォルトルート（`0.0.0.0/0`）へ送られる。閉じたネットワークではこれが「外（インターネット）」を指し、プライベート宛てだと届かず破棄される。

### ルーティングは「推移しない」
* A↔B、B↔C がつながっていても、それだけでは **A↔C は通信できない**。
* A が C への経路を知らなければ、A はそもそも C 宛てのパケットをどこへ送ればよいか分からない。
* A↔C を通すには、(1) A に「C 宛ては B 経由」という経路を教え、(2) 中継役 B が**転送**を行い、(3) 戻りの経路（C→A）も同様に用意する、の 3 つがそろう必要がある。

### NVA（中継ノード / ルーター）と IP フォワーディング
* 自分宛てでないパケットを受け取り、経路表に従って**転送する**役割の機器を中継ノード（ルーター）という。
* 一般的な端末は「自分宛て以外のパケットは破棄する」のが既定動作。中継させるには **IP フォワーディングを有効化**する必要がある。
* 転送は宛先・送信元 IP を**書き換えない**（NAT しない）のが基本。受け手から見た送信元は「元の発信元」のまま。

### ステートフルなアクセス制御と「行き／戻り」両方向
* 通信は「行き（要求）」と「戻り（応答）」の両方がそろって初めて成立する。
* 行きだけ許可しても、戻りの経路や許可が無ければ通信は完結しない。
* 中継経路では、**行き・戻りの両方**に経路（UDR）と許可（NSG）が必要になる。

## Azure 固有の用語（上記概念の具体例）

### Hub-Spoke（ハブ&スポーク）構成
* 中央の **ハブ VNet** に共有機能（今回は中継役の NVA）を置き、各業務ネットワークを **スポーク VNet** として周囲に配置し、ハブとだけピアリングする構成。
* スポーク同士は直接ピアリングせず、通信はハブを経由させる。共有機能（ファイアウォール・NVA・ゲートウェイなど）をハブに集約できるのが利点。

### ルートテーブル（Route Table）と UDR（User Defined Route / ユーザー定義ルート）
* Azure のサブネットには既定で**システムルート**が入っており、自 VNet 内やピアリング先の範囲への経路は自動で用意される。
* それを**上書き／追加**するのがルートテーブル（UDR）。サブネットに関連付けて適用する。
* 今回は「相手スポーク宛て（10.x/16）は NVA(10.0.0.4) へ」という UDR を各スポークに付けることで、本来は届かない spoke 間通信を NVA 経由に誘導している。

### nextHopType（次の転送先の種類）
* UDR の `nextHopType` は next hop の種類を表す。主な値:
  * `VirtualAppliance`: 指定した IP（NVA など）へ転送する。**今回使うのはこれ**。`nextHopIpAddress` に NVA の IP を指定する。
  * `VirtualNetworkGateway` / `Internet` / `VnetLocal` / `None` など。
* `VirtualAppliance` を選ぶことで「この宛先は NVA に処理させる」という意図を経路として表現している。

### NVA における IP フォワーディングの「二段構え」
NVA を中継役として機能させるには、**2 か所**で転送を許可する必要がある。片方だけでは通らない。

| 層 | 設定 | 役割 |
|----|------|------|
| Azure ネットワーク層 | NIC の **`enableIPForwarding = true`** | 「自分宛てでないパケットを送受信してよい NIC」であることを Azure に伝える。これが false だと Azure 側でパケットが破棄される。 |
| OS 層 | **`net.ipv4.ip_forward = 1`**（cloud-init で設定） | Linux カーネルが受け取ったパケットを転送するようにする。既定は 0（転送しない）。 |

* どちらか一方だけでは転送されない。**両方そろって初めて**パケットが NVA を通過する。

### ピアリングの `allowForwardedTraffic`（Step2 の伏線回収）
* Step2 では `allowForwardedTraffic = false` にしていた。これは「相手 VNet から**転送されてきた**（発信元が相手 VNet の外の）通信を受け入れるか」の設定。
* Step3 では NVA が転送を行うため、**ハブ → 各スポーク方向のピアリングで `true` が必須**になる。
  * 例: NVA が spoke1(10.1.x) のパケットを spoke2 へ転送するとき、spoke2 から見ると「送信元がハブの外（10.1.x）」＝転送トラフィック。`peering-hub-to-spoke2` の `allowForwardedTraffic = true` で初めて spoke2 がこれを受け入れる。
* 一方、スポーク → ハブ方向では転送トラフィックが流れないため `false` のまま（最小権限）。**役割に応じて許可の向きを変える**のが Step2 から続く考え方。

## なぜ spoke 間通信が成立するのか（経路の全体像）

`vm-spoke1 (10.1.0.4)` から `vm-spoke2 (10.2.0.4)` への ping を例に、何が経路を成立させているかを追う。

```
[vm-spoke1 10.1.0.4]
   │ ① 宛先 10.2.0.4 への経路を探す
   │    → 直接ピアリングが無いので spoke2 の経路は無い
   │    → UDR「10.2.0.0/16 → NVA(10.0.0.4)」に一致 ⇒ next hop = NVA
   ▼
[peering spoke1 ⇄ hub]
   │ ② ハブへ入る（送信元 10.1 は spoke1 自身の範囲なので通常トラフィック）
   ▼
[NVA nic-nva 10.0.0.4]
   │ ③ 自分宛てでないパケットを受信
   │    → enableIPForwarding=true（Azure）＋ ip_forward=1（OS）で転送可能
   │    → ハブはサブネットに UDR が無く、システムルートで spoke2(10.2/16) を知っている
   ▼
[peering hub ⇄ spoke2]
   │ ④ spoke2 へ転送。送信元 10.1 はハブの外 ⇒ 転送トラフィック
   │    → peering-hub-to-spoke2 の allowForwardedTraffic=true が必須
   ▼
[NSG nsg-spoke2 の受信判定]
   │ ⑤ 送信元は元のまま 10.1.0.4（NVA は NAT しない）
   │    → Allow-ICMP-From-Spoke1 (src 10.1.0.0/24) に一致 ⇒ 許可
   ▼
[vm-spoke2 10.2.0.4]
      ⑥ ICMP Echo Request を受信し、応答を返す
         （戻りは rt-spoke2 の UDR「10.1/16 → NVA」で、同じ経路を逆向きにたどる）
```

### 各段階で何がそれを可能にしているか

| 段階 | 可能にしている設定 | これが無いと |
|------|-------------------|-------------|
| ① 行きの経路 | **rt-spoke1 の UDR**（10.2/16 → NVA） | spoke2 の経路が無く、宛先不明で破棄される |
| ② ハブへの到達 | hub↔spoke1 ピアリング（`allowVirtualNetworkAccess`） | spoke1 から NVA に届かない |
| ③ 転送 | NVA の **enableIPForwarding + ip_forward** | NVA がパケットを破棄する（自分宛てでないため） |
| ④ spoke2 への転送受け入れ | **peering-hub-to-spoke2 の allowForwardedTraffic=true** | 転送トラフィックが spoke2 に拒否される |
| ⑤ 受信許可 | **nsg-spoke2 の Allow-ICMP-From-Spoke1** | 10.1 は spoke2 の VNet 範囲外なので既定では拒否される |
| ⑥ 戻りの経路 | **rt-spoke2 の UDR** + hub↔spoke1 の forwarded 許可 | 応答が spoke1 へ戻れず、ping が完結しない |

## なぜスポークでは NSG ルールを明示的に書く必要があるのか

* NSG の既定ルール `AllowVnetInBound` / `AllowVnetOutBound` は、送信元・宛先が **`VirtualNetwork` サービスタグ**に含まれる通信を許可する。
* この `VirtualNetwork` タグには「自 VNet の範囲 + **ピアリング先**の範囲」が含まれる。
* **spoke1 は hub としかピアリングしていない**ため、spoke1 の `VirtualNetwork` タグは `10.1/16 + 10.0/16` のみ。**spoke2(10.2/16) は含まれない**。
  * → spoke2 宛て/spoke2 からの通信は既定ルールでカバーされず、**明示的な NSG ルールが必要**になる（`Allow-ICMP-To-Spoke2-Outbound` など）。
* 一方 **ハブは spoke1・spoke2 の両方とピアリング**しているため、ハブの `VirtualNetwork` タグは `10.0+10.1+10.2`。spoke1↔spoke2 の転送は既定ルールで通り、`nsg-nva` には管理用の最小ルールしか書いていない。
* 「ピアリング関係が NSG の既定許可範囲（VirtualNetwork タグ）を決めている」という点が、Hub-Spoke 特有の落とし穴。

## このステップの要点
* ピアリングは**推移しない**。hub↔spoke を 2 本張っても spoke 同士は通信できない。
* **UDR（ルートテーブル）** で「相手 spoke 宛ては NVA 経由」と経路を上書きし、**NVA**（IP フォワーディング有効な VM）が中継することで spoke 間通信が成立する。
* NVA の転送には **NIC の enableIPForwarding（Azure 層）と OS の ip_forward（OS 層）の二段構え**が必要。
* 転送トラフィックを受け入れる側のピアリングで **`allowForwardedTraffic = true`** が必須。
* スポークは相手スポークをピアリングしていない＝`VirtualNetwork` タグに含まないため、spoke 間通信には **NSG ルールを明示的に**書く必要がある。
* `just disable-route` / `enable-route` で UDR を出し入れすると、通信が止まる／復活することから「経路を成立させているのは UDR」だと確認できる。
