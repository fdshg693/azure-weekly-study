# log 学習プラン — ロギング／可観測性をゼロから積む

このファイルは log トピックの**プロジェクト設計の目安（ロードマップの正本）**。
方針はリポジトリ共通（[../CLAUDE.md](../CLAUDE.md)）どおり:
「**一般概念／最小構成 → 実装 → 設定を出し入れして因果を確かめる**」「**構築・実行はユーザー自身、AI は Azure 上で実行しない**」。

## 1. このトピックの立ち位置（前提）

- **新規トピック**。既存の他トピック（auth / func / storage / network / foundry / k8s）には、
  ログ周りの設定・学習は**ほぼ存在しない**前提で、ここでゼロから積み上げる。
- そのため最初は「アプリのログがどこへ行き、どう検索でき、どう因果を追えるか」という**一般概念から**入る。
  特定 Azure サービスの機能網羅ではなく、**ログのライフサイクル（出す → 集める → 問い合わせる → 気づく）**を主役にする。
- 他トピックとの接続点（後述）が多いトピックなので、各 Step で「他トピックのどの体験に繋がるか」を意識する。
  - 例: k8s の [PLAN.md](../k8s/PLAN.md) Step 4 `observability`（Container Insights / Prometheus）は、本トピックの土台があると理解が早い。

## 2. このトピックで扱う主要概念（全体地図）

ログ／可観測性の登場人物を、おおまかに 4 層で捉える。

| 層 | 概念 | Azure での主役 |
|---|---|---|
| 出す（生成） | 構造化ログ、ログレベル、トレース／メトリクス／ログの 3 本柱、相関 ID | アプリ SDK（OpenTelemetry）、`stdout` |
| 集める（収集） | テレメトリの送信、診断設定、プラットフォームログ vs アプリログ、サンプリング | Application Insights、Diagnostic Settings |
| 貯める／問い合わせる | Log Analytics ワークスペース、テーブル（`traces`/`requests`/`exceptions` …）、KQL、保持期間・課金 | Log Analytics Workspace、KQL |
| 気づく／可視化 | アラート、アクショングループ、ダッシュボード、ワークブック、分散トレース | Azure Monitor Alerts、Workbooks |

押さえておきたい区別:

- **3 本柱**: ログ（離散イベント）／メトリクス（数値時系列）／トレース（リクエストの経路）。混同しない。
- **アプリログ vs プラットフォームログ**: 自分のコードが出すログと、Azure リソースが出す診断ログ（リソースログ）／アクティビティログ（コントロールプレーン監査）は出どころも経路も別。
- **Application Insights は今はワークスペースベース**: App Insights の実体は Log Analytics ワークスペース上のテーブル。両者は別物ではなく地続き、という感覚を早めに持つ。
- **収集 SDK は OpenTelemetry が現行**: 旧 Application Insights SDK ではなく、`azure-monitor-opentelemetry`（OpenTelemetry ベース）を使う。ベンダーニュートラルな OTel の概念を主、Azure を送信先（エクスポーター）と捉える。

## 3. 推奨ロードマップ（やさしい順）

各プロジェクトは `learn/log/{name}/` に置く想定。「**1 プロジェクト = 主役の概念 1〜2 個**」に絞り、
前のプロジェクトの成果物（ワークスペース・アプリ）を流用して差分を最小化する。
技術スタックは他トピックに合わせ、Bicep（ワークスペース等の調達）＋ Python or Node（アプリ）＋ just / Taskfile ＋ Azure CLI を基本とする。

### Step 1 — `app-insights-basic`（最小構成：ログを出して、届いたのを見る）
**主役**: 構造化ログとログレベル、Application Insights へのテレメトリ送信の最小ループ。
- Bicep で **Log Analytics ワークスペース + Application Insights（ワークスペースベース）** を 1 つだけ作る。
- 小さな Python or Node アプリに `azure-monitor-opentelemetry` を入れ、接続文字列だけで自動計装。
  ログ（trace）・例外（exception）・リクエスト（request）が**そのまま App Insights に届く**ことを確認。
- **因果を確かめる実験**:
  - ログレベルを `INFO`→`WARNING` に上げると、収集される行が減る／消える。
  - わざと例外を投げると `exceptions` テーブルにスタックトレース付きで現れる。
  - ログを「文字列連結」から「**構造化（属性付き）**」に変えると、後段で属性キーで絞り込めるようになる前振り。

### Step 2 — `kql-queries`（貯めたログに問い合わせる）
**主役**: Log Analytics ワークスペースのデータモデルと **KQL**。
- Step 1 のワークスペース／アプリを**流用**（新規インフラなし）。アプリで色々なログ・例外・負荷パターンを生成。
- KQL の基本（`where` / `project` / `summarize` / `bin()` / `render`）と、テレメトリのテーブル構造
  （`traces` / `requests` / `exceptions` / `dependencies`）を往復。
- **因果を確かめる実験**:
  - 同じデータを「件数集計」「時系列ビン集計」「失敗率」など問い合わせ方を変えて、見え方が変わるのを体感。
  - Application Insights の `traces` と、ワークスペース直クエリが**同じ実体**であることを確認（App Insights ≒ LA テーブル）。
  - クエリ結果を保存し、後の Step（アラート・ワークブック）の材料にする。

### Step 3 — `diagnostic-settings`（アプリ以外＝プラットフォームログを集める）
**主役**: 診断設定（リソースログ）と、アプリログ／プラットフォームログ／アクティビティログの違い。
- 既存トピックのリソースを 1 つ題材にする（例: storage トピックの Storage、または auth/func で使う Key Vault）。
- そのリソースに **Diagnostic Settings** を付け、リソースログを Step 1 のワークスペースへ流す。
- **因果を確かめる実験**:
  - リソースを操作（Blob 読み書き／Key Vault のシークレット取得など）→ 対応するリソースログが現れる。
  - 操作を**拒否される**条件（権限なし等、auth トピックの体験と接続）を作り、ログ上で許可／拒否がどう記録されるか比較。
  - コントロールプレーンの**アクティビティログ**（誰がリソースを作った・消した）が、データプレーンのリソースログとは別レーンであることを確認。

### Step 4 — `alerts`（ログから自動で気づく）
**主役**: ログ検索アラート（KQL ベース）／メトリクスアラートと**アクショングループ**。
- Step 1〜3 のワークスペースを流用し、**KQL クエリの結果がしきい値を超えたら発火**するアラートルールを Bicep / az で作る。
- アクショングループ（通知先。学習用途ではメール or Webhook）を紐付ける。
- **因果を確かめる実験**:
  - アプリでわざとエラーを量産 → エラー数アラートが**発火**、収まると**解決**に戻るライフサイクルを観察。
  - しきい値・評価頻度・時間窓を変えると、発火タイミングと頻度がどう変わるかを出し入れして確かめる。

### Step 5 — `distributed-tracing`（複数サービスをまたいで追う）
**主役**: 分散トレースと相関 ID、OpenTelemetry の trace/span。
- 2 つの小サービス（フロント → API、または API → DB/外部呼び出し）を用意し、OTel の自動／手動計装で繋ぐ。
- **因果を確かめる実験**:
  - 1 リクエストが複数サービスを横断する様子を**アプリケーションマップ／トランザクション検索**で 1 本の trace として可視化。
  - 相関 ID を**切る**と trace が分断され、繋ぐと 1 本になる、を対比して相関の仕組みを理解。
  - 依存関係（`dependencies` テーブル）でどの外部呼び出しが遅いかを特定。

### Step 6（発展）— `container-insights` / `cost-sampling` / `streaming`
- **`container-insights`**: k8s トピックの AKS にコンテナ監視を有効化し、Pod ログ・メトリクスを本トピックの土台（ワークスペース・KQL・アラート）で扱う。k8s [PLAN.md](../k8s/PLAN.md) Step 4 と合流。
- **`cost-sampling`**: サンプリング・保持期間・テーブルプラン（Basic/Analytics）・取り込み課金を出し入れし、**可観測性とコストのトレードオフ**を体感。
- **`streaming`**: 診断設定の送信先を Log Analytics 以外（Storage へのアーカイブ、Event Hub へのストリーミング）にも広げ、用途別に経路を選ぶ感覚を持つ。

## 4. 進め方のメモ

- **Step 1 のワークスペース＋App Insights を以降ずっと流用**するのが基本。新規 Bicep が要るのは Step 1 と、題材リソースを追加する Step 3、アラートを足す Step 4 くらいに留める。
- 共通方針どおり、各プロジェクトには共通構成（`README.md` / `KNOWLEDGE.md` / `justfile` または `Taskfile.yml`）を置く。KQL クエリ集は `queries/*.kql` のように分離して可読性を保つ。
- **OpenTelemetry を主、Azure を送信先と捉える**スタンスで KNOWLEDGE.md を書くと、ベンダー非依存の概念（trace/span/メトリクス）と Azure 固有（App Insights / LA / 診断設定）が切り分けやすい。
- トピックの習熟度・概念の習得状況は別途 `learn/log/CLAUDE.md`（未作成）に記録する想定。**最初のプロジェクト（Step 1）を 1 つ追加したタイミングで作成**し、併せて `learn/CLAUDE.md` にも log トピックの概要行を追加する。
- 課金注意: Log Analytics は**取り込み量・保持期間で課金**される。学習後は不要なリソースを片付ける／保持を短くするところまでを各 Step の締めにする。
