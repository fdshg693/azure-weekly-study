# メッセージアプリ V2.0

複数のユーザーがメッセージを互いに送りあえる。
**V2 では「パスワード認証（＋メール検証）」と「友達リスト」を追加する。**

> V1（認証なし・メッセージ送受信のみ）の仕様・設計は `versions/v1/` を参照。
> 本ドキュメントは **V1 からの差分** に集中し、変更のない部分（4 サービス構成・キャッシュ戦略の基本・
> Cosmos のパーティション設計）は繰り返さない。

## V2 で増えること（V1 からの差分）

| 領域 | V1 | V2 |
| --- | --- | --- |
| 認証 | なし（`X-User` を client が自己申告） | **パスワード認証 + メール検証**。BFF が **セッショントークン(JWT)** を検証し、下流へ信頼済み `X-User` を注入 |
| ユーザー作成 | login 時に username を upsert | **サインアップ**でメール/パスワードを登録（パスワードはハッシュ化）。**メールのリンクを踏んで検証**して初めてログイン可 |
| 友達 | なし | **友達リスト**：任意のユーザーを自分のリストに追加・削除・閲覧できる |
| 新 Azure サービス | — | **Azure Communication Services (Email)**：検証メールを実送信（ローカルはモック） |

> **V2 のスコープ外（将来）**：友達リストは「リスト機能のみ」。
> 「友達同士でのみメッセージ送信可」にする制限は **V2 では実装しない**（メッセージは V1 同様、誰にでも送れる）。

## 機能仕様（V2 追加分）

### 認証（パスワード認証 + メール検証）
- **サインアップ**：`email` / `username` / `password` を登録。
  - パスワードは平文保存しない（ハッシュ化して保存）。
  - 登録直後は **未検証(unverified)** 状態。検証メールのリンクを踏むまでログインできない。
- **メール検証**：
  - **ローカル**：実際のメール送信は行わず、**検証リンクをコンソール／ファイルに出力**して手で踏む（パスワード検証だけを体験する）。
  - **実アプリ（Azure）**：ACS Email で **本物の検証メールを送信** し、ユーザーがリンクを踏んで検証する。
- **ログイン**：`email` + `password` を検証し、**検証済みなら JWT（セッショントークン）を発行**。
  - 以降のリクエストは `Authorization: Bearer <token>` を付ける。
  - **BFF がトークンを検証**し、本人(`username`)を下流へ `X-User` として転送する
    （V1 のように client が `X-User` を詐称できなくなる、が今回の肝）。
- **ログアウト**：トークンを破棄するだけ（ステートレス）。

### 友達リスト
- 自分の友達リストに **任意のユーザーを追加 / 削除 / 一覧** できる。
- **一方向・自己完結**：A が B を追加/削除しても影響を受けるのは **A 自身のリストだけ**で、B のリストは変わらない
  （相手の承認・相互フレンド・申請フローは V2 スコープ外）。
- 友達リストは Redis（`friends:{owner}`）にキャッシュし、**owner 自身の追加/削除時にその owner のキャッシュを無効化**する。
  - これは「自分の操作 → 自分のキャッシュを更新」で、V1 の送信者が自分の `conv` キャッシュを更新したのと**同じパターン**。
  - V1 の陳腐化は「自分の操作が**他人**のキャッシュ（受信者の会話）に影響するのに、それを無効化しなかった」から起きた。
    友達リストは一方向・自己完結で**他人のキャッシュに影響しない**ため、そもそも陳腐化の構図が生じない。

## 技術スタック（V2 差分）

V1 の 4 サービス構成はそのまま。V2 で足すもの：

- **Azure Communication Services (Email)**：検証メールの送信（書き込み = Functions から呼ぶ）。
- **JWT**：セッショントークン。発行は読み取り API（FastAPI / login）、検証は BFF。
- 設定追加（すべて環境変数 / `.env`）：`JWT_SECRET`・`JWT_TTL_SECONDS`・`ACS_CONNECTION_STRING`・
  `ACS_SENDER_ADDRESS`・`EMAIL_MODE`(`local`|`acs`)・`APP_BASE_URL`（メール内リンクの組み立て用）。

> 認証情報（`JWT_SECRET` / `ACS_CONNECTION_STRING`）は本来 Key Vault に置きたい。
> V2 では App Settings / `.env` に留め、**Key Vault は将来バージョンの学習テーマ**とする。

詳しい設計は [PLAN.md](./PLAN.md)、図は [MERMAID.md](./MERMAID.md)、用語は [KNOWLEDGE.md](./KNOWLEDGE.md)。

### この設計で体験する学習ポイント
- **パスワードはハッシュで保存**し、平文を扱わない（漏洩時の被害を抑える）。
- **メール検証フロー**：トークン付きリンク → クリックで検証、という所有確認の仕組み。
- **ステートレスな JWT セッション**：サーバーにセッションを持たず署名検証で本人確認。
- **信頼境界の移動**：本人確認を client 申告（V1）から **BFF でのトークン検証**（V2）へ移す。
- **キャッシュ影響範囲の理解**：友達リストは自己完結（自分の操作=自分のキャッシュ）なので陳腐化しない。V1 の陳腐化は「自分の操作が他人のキャッシュに影響する」場合に限って起きた、という違いを区別する。

## ローカルで動かす

前提・起動手順の大枠は V1（`versions/v1/README.md`）と同じ。**V2 の差分のみ**を示す。

```pwsh
# 0) 設定生成（JWT_SECRET など V2 の項目も含めて雛形を作る）
task env-init

# 1) 依存起動・各サービス起動は V1 と同じ（Cosmos Emulator / Redis / Azurite）
task local-up
task api / task functions / task bff

# 2) ブラウザで開く
task open
```

### 認証フローを体験する（ローカル）
```pwsh
# サインアップ → 検証リンクが出力される → 踏む → ログイン、までを一括で流す
task seed-auth
```
1. サインアップ（`email`/`username`/`password`）。
2. **ローカルでは検証リンクがコンソール（または `.verify-links/`）に出力される**。
3. そのリンクを開く（`GET /api/verify?token=...`）→ メール検証済みになる。
4. ログイン → JWT が返る → 以降のメッセージ送受信ができる。
5. **未検証のままログインを試す**と弾かれることも確認する（検証ゲートの体験）。

### 友達リストを体験する
1. alice でログイン → `task users` 等で bob を見つける。
2. bob を友達追加 → 友達リストに出る。
3. 追加直後にリスト取得 → **即時に反映される**（自分の操作で自分のキャッシュを無効化するため。他人が関わらないので陳腐化の余地がない）。
4. 削除 → これも即時に反映される。

## Azure へデプロイ（任意・実リソース作成）

> ガードレール上、デプロイは明示操作。V1 のコスト（App Service / Redis / Cosmos）に加え、
> **V2 では ACS（Email）** のリソース・ドメイン設定が増える。学習後は `task destroy` で後片付けする。

V1 の手順に対する差分：
- `task deploy` が **ACS（Communication Services + Email ドメイン）** も作成する。
- `EMAIL_MODE=acs` で **本物の検証メールが届く** → リンクを踏んで検証する一連を Azure 上で体験する。
- メール送信元アドレス（`ACS_SENDER_ADDRESS`）は Azure 管理ドメイン（`*.azurecomm.net`）で開始し、
  独自ドメイン検証は任意（学習の深掘りポイント）。

### デプロイ後に体験すること
- 実メールを受信 → リンク → 検証 → ログイン、の本番相当フローを確認。
- Cosmos の `users` に `passwordHash` / `emailVerified` が入り、`friends` コンテナに友達関係が入ることを確認。
- 不正・期限切れトークンや、改ざんした JWT が **BFF で 401 になる**ことを確認（信頼境界の体験）。
