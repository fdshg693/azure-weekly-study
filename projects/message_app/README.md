# メッセージアプリ V3.0

複数のユーザーがメッセージを互いに送りあえる。
**V3 では「知り合い／友達」の二段階の人間関係を本格化し、友達同士に限ってメッセージ送信を許可する。**
あわせて、これまで学習目的で意図的に残していた**キャッシュ陳腐化を解消**し、
「リソースを変える操作は、そのリソースが寄与する全キャッシュを必ず無効化する」一般的な構造に統一する。

> V1（認証なし・メッセージ送受信のみ）・V2（パスワード認証＋メール検証・友達リスト）の仕様・設計は
> `versions/v1/` `versions/v2/` を参照。
> 本ドキュメントは **V2 からの差分** に集中し、変更のない部分（4 サービス構成・認証/JWT・Cosmos パーティション設計）は繰り返さない。

## V3 で増えること（V2 からの差分）

| 領域 | V2 | V3 |
| --- | --- | --- |
| 人間関係 | **友達リスト**（一方向・任意のユーザーを自分のリストに追加/削除/閲覧） | **知り合い（一方向）と 友達（相互マッチ）の二段階**に再設計 |
| 用語・命名 | 「友達」= 自分が認識している相手（一方向） | その一方向の関係を **「知り合い」** に改名。**「友達」は相互に知り合い登録した関係**だけを指す（ドキュメント・コードの命名を全面的に変更） |
| 関係の可視化 | 自分の知り合い一覧のみ | **自分を知り合い登録している人**（受信側＝inbound）も見られる |
| メッセージ送信 | 誰にでも送れる（V2 まで制限なし） | **友達（相互マッチ）同士のみ送信可**。知り合いなだけ（片方向）では送れない |
| キャッシュ | 学習目的で **送信時に受信者のキャッシュをあえて無効化しない**（陳腐化を体験） | **リソース変更時に古くなる全キャッシュを必ず無効化**する一般構造へ統一（旧機能・新機能とも） |

> **V2 で「スコープ外（将来）」と書いた制限が V3 の主役**：
> V2 README には「『友達同士でのみメッセージ送信可』にする制限は V2 では実装しない」と明記していた。
> V3 はまさにこれを実装する。そして「相互マッチで友達になる」設計により、
> **自分の操作が他人のキャッシュに影響する**構図が初めて生じる → 正しいキャッシュ無効化が必須になる。

## 用語の整理（V3 の肝・最初に読む）

| 用語 | 意味 | 方向 |
| --- | --- | --- |
| **知り合い（acquaintance）** | 自分が「この人を知っている」と登録した相手。V2 までの「友達」と同じ一方向の関係 | 一方向（A→B） |
| **自分を知り合い登録している人（inbound）** | 逆向き。自分を知り合いに入れている人たち。自分からは見えるが、こちらは何もしていない | 一方向（B→A） |
| **友達（friend）** | **相互に知り合い登録**が成立した関係（A→B かつ B→A）。両者が互いを知り合いにして初めて成立 | 双方向（相互マッチ） |

- 友達は別テーブルに保存しない。**「自分の知り合い」と「自分を知り合い登録している人」の積集合**として導出する。
- **メッセージは友達同士のみ**。片方向の知り合いだけでは送れない（送ると 403）。

## 機能仕様（V3 追加・変更分）

### 知り合い（一方向。旧「友達」の改名）
- 自分の知り合いリストに **任意のユーザーを追加 / 削除 / 一覧** できる（操作自体は V2 の友達リストと同じ）。
- **一方向**：A が B を知り合いにしても、B のリストには何も作らない。
- ただし V2 と違い、A の追加は **B の「自分を知り合い登録している人」ビュー**に現れる（後述のキャッシュ無効化が効く）。

### 自分を知り合い登録している人（inbound・新規）
- 自分を知り合いに入れている人の一覧を見られる（逆引き）。
- ここに出ている相手を**自分も知り合い登録すると友達が成立**する、という導線になる。

### 友達（相互マッチ・新規の概念）
- A→B と B→A の知り合い登録が**両方そろった時点で友達**になる。
- 友達一覧は導出値（積集合）。一覧取得は read-through キャッシュ。
- どちらかが知り合いを解除すると友達も解消される。

### メッセージ送信（友達ゲートを追加）
- 送信前に **送信者と宛先が友達（相互マッチ）か**を確認する。友達でなければ **403**。
- これにより「知り合いになっただけ／一方的にフォローしただけ」では会話を始められない。

### キャッシュ無効化の正常化（V3 の二本目の柱）
- **方針**：リソース（会話・知り合い関係）を変える書き込みは、**そのリソースが寄与する全キャッシュ**を無効化する。
  「誰のキャッシュに影響するか」を**操作の影響範囲**から機械的に導く。
- **メッセージ送信**：V2 まで「送信者のキャッシュだけ更新し受信者は触らない」＝陳腐化を意図的に残していた。
  V3 は **送信者・受信者の双方の会話キャッシュを無効化**する（受信者も即座に新着を見る）。
- **知り合いの追加/削除**：自分の知り合いキャッシュに加え、相手の「inbound」キャッシュ、
  そして相互マッチが成立/解消する場合は **双方の友達キャッシュ**も無効化する。
- 詳細な無効化表は [PLAN.md](./PLAN.md) を参照。

## 技術スタック（V3 差分）

- 新しい Azure サービスの追加は**なし**（V2 の 4 サービス構成 + ACS のまま）。V3 はアプリ層の設計変更が主。
- データモデル：Cosmos コンテナ `friends` を **`acquaintances`** に改名。1 登録で**発信／受信の 2 ミラー文書**を書く
  dual-write（`direction`=`out`/`in` ＋ `peer`）にして、inbound・友達・友達ゲートを単一パーティション化する。
- キャッシュキーを再設計（`acq:{owner}` / `acqby:{target}` / `friends:{user}` / `conv:{viewer}:{pair}`）。
- 命名変更はドキュメントだけでなく**コードにも反映**する（コンテナ名・関数名・ルート・キャッシュキー）。
  対応表は [PLAN.md](./PLAN.md) の「命名の移行表」を参照。

詳しい設計は [PLAN.md](./PLAN.md)、図は [MERMAID.md](./MERMAID.md)、用語は [KNOWLEDGE.md](./KNOWLEDGE.md)。

### この設計で体験する学習ポイント
- **一方向の関係（知り合い）と双方向の関係（友達）の区別**：友達を**導出値**（積集合）として表し、
  別テーブルに持たない設計の利点（整合性が崩れにくい）を体験する。
- **逆引きインデックス（dual-write / fan-out on write）**：「自分を知り合い登録している人」や友達ゲートは頻出なので、
  登録時に発信・受信の 2 文書を書いて**全ビューを単一パーティション化**する。読み取りの速さと引き換えに
  書き込み増幅・多パーティション整合（単一トランザクション不可）の責任を負う、という設計トレードオフを体験する。
- **権限ゲート＝関係に基づく認可**：認証（誰か）の先にある**認可（何をしてよいか）**を、
  「友達か？」という関係チェックで体験する（友達でなければ送信不可）。
- **キャッシュ無効化の一般原則**：V2 で「自己完結だから陳腐化しない」と説明した関係が、
  V3 で相互マッチを導入した瞬間に**他人のキャッシュへ影響する**ようになる。
  「操作の影響範囲＝無効化すべきキャッシュ」という対応を、旧機能（会話）と新機能（関係）の両方で貫く。

## ローカルで動かす

前提・起動手順の大枠は V1/V2 と同じ。**V3 の差分のみ**を示す。

```pwsh
# 0) 設定生成（V3 で新しい環境変数の追加はなし。V2 のまま）
task env-init

# 1) 依存起動・各サービス起動は V2 と同じ（Cosmos Emulator / Redis / Azurite）
task local-up
task api / task functions / task bff

# 2) ブラウザで開く
task open
```

### 知り合い → 友達 → メッセージ を体験する（ローカル）
```pwsh
# サインアップ/検証/ログイン済みの alice・bob を用意（V2 の seed-auth を踏襲）
task seed-auth
```
1. alice でログイン → bob を**知り合い**に追加する。
2. この時点では **片方向**（alice→bob のみ）。alice から bob にメッセージを送ろうとすると **403**（まだ友達でない）。
3. bob でログイン → bob の「**自分を知り合い登録している人**」に alice が出ていることを確認。
4. bob が alice を**知り合い**に追加 → **相互マッチ成立 → 友達**になる。
5. alice・bob 双方の**友達一覧に相手が即座に出る**（相互マッチ時に双方の友達キャッシュを無効化するため）。
6. これで alice ↔ bob はメッセージ送受信できる。

### キャッシュ無効化の正常化を体験する
1. alice → bob にメッセージ送信。
2. **bob 側で会話を開くと、TTL を待たずに新着が見える**（V2 の陳腐化が解消されている）。
   - V2 では同じ操作で「bob は TTL 切れまで新着が見えない」陳腐化が起きていた。`versions/v2/` と挙動を比べる。
3. どちらかが知り合いを解除 → 友達一覧から相手が**即座に消える**（双方の友達キャッシュを無効化）。

## Azure へデプロイ（任意・実リソース作成）

> ガードレール上、デプロイは明示操作。**V3 で新規 Azure リソースの追加はない**（V2 の構成のまま）。
> 変更はアプリのコード（命名・関係ロジック・キャッシュ無効化）と、Cosmos コンテナ名（`acquaintances`）のみ。

V2 の手順に対する差分：
- 初回デプロイ後、Cosmos に **`acquaintances` コンテナ**が作られることを確認する（旧 `friends` からの移行は KNOWLEDGE.md 参照）。
- 友達でない相手への送信が **403** になること、相互マッチ後に送信できることを確認（関係に基づく認可の体験）。
- メッセージ送信後、**受信者側でも即座に新着が見える**こと（キャッシュ無効化の正常化）を本番相当で確認する。
