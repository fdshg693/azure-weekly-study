# KNOWLEDGE — このプロジェクトで新たに出た用語・概念 V3.0

V1（BFF / CQRS 的分離 / パーティションキー / read-through / TTL / stale cache 等）・
V2（パスワードハッシュ / メール検証 / JWT / 信頼境界 / クロスパーティション・クエリ / 冪等な書き込み 等）で
説明済みの語は `versions/v1/KNOWLEDGE.md` `versions/v2/KNOWLEDGE.md` を参照し、ここでは繰り返さない。
**V3 で新たに出る「二段階の人間関係」「関係に基づく認可」「キャッシュ無効化の一般原則」**だけを記す。

## 人間関係のモデリング

- **一方向の関係 vs 双方向の関係（知り合い / 友達）**
  「知り合い（acquaintance）」は `A→B` の**一方向**：A が B を知っていても B は何もしていない。
  「友達（friend）」は `A→B` かつ `B→A` がそろった**双方向（相互マッチ）**。
  片方向の登録を双方向の合意の素材として使い、両方そろった時だけ上位の関係（友達）に昇格させる、という設計。
  （SNS の「フォロー」と「相互フォロー＝友達」、あるいは招待の往復で握手が成立するモデルと同じ発想。）

- **導出値として持つ（友達を別テーブルに保存しない）**
  友達は「知り合い」と「自分を知り合い登録している人」の**積集合**として毎回導出する。
  - 長所：友達という状態を別に持たないので、**二重更新による不整合が起きない**
    （知り合いを消した瞬間、友達からも自動的に外れる）。
  - 短所：読み取りのたびに積集合を計算する → そこを**キャッシュ**で補う。
  - 反対の設計（相互成立時に `friendships` を実体化）は、読み取りは速いが**整合性維持の責任が増える**。
    学習では「導出＋キャッシュ」を採り、実体化は将来テーマとする。

- **逆引きビュー（inbound）と dual-write（fan-out on write）**
  「自分を知り合い登録している人」は関係を**逆向きに**たどる読み取り。素朴に 1 登録=1 文書だと、
  逆向きはパーティションキー外の検索＝**クロスパーティション**になる（V2 の email/token 逆引きと同じコスト構造）。
  inbound・友達導出・友達ゲートは**頻出**なので、本 V3 は最初から**逆引きインデックスを実体化**する：
  「A が B を登録」した時に**発信エッジ（owner=A, direction=out）と受信エッジ（owner=B, direction=in）の 2 文書**を書く。
  - これで `owner=me` の**単一パーティション**だけで「知り合い／inbound／友達」が全部出る。
  - 代償は**書き込み増幅（1 登録=2 文書）**。「読み取りを安くするために書き込み時に手間をかける（fan-out on write）」という定番トレードオフ。
  - 「逆引き＝必ず高い」ではなく、**インデックス設計次第で逆向きも単一パーティションにできる**例。

## 認可（authorization）

- **認証(authn)と認可(authz)の区別**
  - 認証＝「誰か」を確かめる（V2 の JWT 検証）。
  - 認可＝「その人がこの操作をしてよいか」を確かめる（V3 の友達ゲート）。
  ログインできること（認証済み）と、特定の相手に送ってよいこと（認可）は**別の層**。
  V3 はここを「友達か？」という**関係に基づく認可（relationship-based access control）**で実装する。

- **関係チェックを 1 パーティションのポイントリードで行う**
  相互マッチの確認は、送信者 A の partition で `out__A__B`（A→B）と `in__A__B`（B→A のミラー）を
  **2 回ポイントリード**するだけ。dual-write で受信エッジを A 側にもミラーしているため、
  B の partition に触れずに**単一パーティション**で友達判定できる（ミラーが無ければ 2 パーティションが要る）。

- **多パーティション dual-write の整合（単一トランザクションの限界）**
  発信／受信の 2 文書は **owner=A と owner=B の別パーティション**にある。
  Cosmos の **TransactionalBatch は単一パーティション限定**なので、この 2 書き込みを 1 トランザクションにはできない。
  → 片側だけ成功して落ちると**孤立エッジ**が残りうる。対策は **決定的 id の冪等 upsert/delete**（再実行で収束）＋
  必要なら **Change Feed や定期リコンサイル**で対向エッジを補完。「単一パーティション読み取りの速さ」と
  「多パーティション整合の責任」を**トレードする**のが dual-write の本質。

## キャッシュ無効化の一般原則

- **「リソース変更 → そのリソースが寄与する全キャッシュを無効化」**
  キャッシュは「あるリソースの読み取り結果のコピー」。元リソースを書き換えたら、**それを材料にした全キャッシュ**が古くなる。
  だから書き込みの最後に、影響を受けるキャッシュキーを**漏れなく**消す（または作り直す）。
  - 手順：①この書き込みはどのリソースを変えるか → ②そのリソースを読むキャッシュは誰のどのキーか → ③それを全部無効化。

- **「影響範囲」が自分に閉じるか他人に及ぶか**
  V2 までは「友達リストは自己完結だから自分のキャッシュだけ無効化すれば済む」と整理した。
  V3 で**相互マッチ**を入れると、A の知り合い追加は **B の inbound／友達ビュー**に影響する。
  → 影響範囲が**他人に及ぶ**ので、A の操作で **B 側のキャッシュも**無効化する必要が出る。
  「自己完結か否か」は固定の性質ではなく、**機能設計（一方向か相互か）によって変わる**と理解する。

- **意図的な陳腐化は学習用の例外だった**
  V1/V2 の「送信時に受信者キャッシュを無効化しない」は、**陳腐化を体験させるための意図的な手抜き**だった。
  本番の一般原則は「古くなるキャッシュは必ず消す」。V3 でこの例外を撤廃し、原則に統一する。
  （陳腐化を再現したいときは `versions/v2/` の挙動と比較する。）

## データ移行（メモ）

- **コンテナ改名（`friends` → `acquaintances`）**
  Cosmos はコンテナのインプレース改名ができない。実運用なら新コンテナを作って**移行コピー**するが、
  本学習ではローカル Emulator を作り直す／新規 `acquaintances` を作る運用とし、旧データ移行はスコープ外。
