# KNOWLEDGE — このプロジェクトで新たに出た用語・概念 V2.0

V1 で説明済みの語（BFF / CQRS 的分離 / パーティションキー / read-through / TTL / stale cache /
Functions v2 モデル / Azurite 等）は `versions/v1/KNOWLEDGE.md` を参照し、ここでは繰り返さない。
**V2 で新たに出る認証・メール・友達リスト関連**だけを記す。

## 認証・パスワード

- **パスワードハッシュ（bcrypt / argon2）**
  パスワードを平文や可逆暗号で保存せず、**一方向ハッシュ**で保存する。ソルト付き・計算コストを持たせ、
  漏洩時の総当たりを難しくする。検証は「入力をハッシュして保存値と比較」。

- **タイミング安全な比較**
  ハッシュ／トークン比較を文字単位で早期 return すると、応答時間差から情報が漏れうる。
  一定時間で比較する関数（constant-time compare）を使う。

- **メール検証（所有確認）フロー**
  サインアップ直後は「未検証」。ランダムな**検証トークン付きリンク**をメール送信し、
  クリックされて初めてメールアドレスの**所有**を確認できる。検証済みになるまでログインさせない（検証ゲート）。

- **JWT（JSON Web Token）/ ステートレスセッション**
  `header.payload.signature` の署名付きトークン。サーバーにセッションを持たず、**署名検証だけで本人確認**できる。
  - 長所：スケールしやすい（共有状態が要らない）。
  - 短所：**サーバー側から即時失効しにくい**（exp まで有効）。即時ログアウトが要るなら Redis セッション等を併用。
  - 本 V2 では HMAC（`JWT_SECRET`）で署名、`exp` で失効。

- **信頼境界（trust boundary）と BFF でのトークン検証**
  「どこから先を信頼するか」の線引き。V1 は client 申告の `X-User` を信頼していた（詐称可能）。
  V2 は **BFF が JWT を検証**して本人を確定し、内部の下流へ `X-User` を注入する。
  → 信頼境界を「ブラウザ」から「BFF」へ移すことで、client は身元を詐称できなくなる。

- **Bearer トークン / `Authorization` ヘッダ**
  `Authorization: Bearer <token>` でトークンを送る標準的な作法。BFF がこれを取り出して検証する。

## Azure Communication Services (Email)

- **ACS（Azure Communication Services）**
  メール / SMS / 通話などの通信機能を API で使える Azure サービス。V2 では **Email** だけ使う。

- **ACS Email / 送信元ドメイン**
  メール送信には検証済みの送信元ドメインが要る。学習では **Azure 管理ドメイン（`*.azurecomm.net`）** を使えば
  すぐ送れる。独自ドメインは DNS（SPF / DKIM）検証が必要で、これは深掘りの任意テーマ。

- **接続文字列での認証**
  ACS リソースの `ACS_CONNECTION_STRING` で SDK が認証する。本来は秘密情報なので Key Vault 管理が望ましい
  （V2 では App Settings / `.env` に留め、Key Vault は将来テーマ）。

- **メール送信モードの切替（local / acs）**
  ローカルでは外部送信せず**検証リンクをコンソール/ファイルに出力**してフローだけ体験し、
  Azure では実送信する。環境変数 `EMAIL_MODE` で同一コードを切り替える。

## データモデル / クエリ

- **クロスパーティション・クエリ**
  パーティションキー以外（email / token）で検索すると、全パーティションを横断する。
  単一パーティション・クエリより高コスト。小規模学習では許容するが、頻出キーは
  **ルックアップ用の設計（別コンテナ、または `id` をその値にする）**で単一パーティション化するのが定石。

- **冪等な書き込み（友達追加）**
  `id = owner__friend` を決め打ちして upsert すると、同じ追加を何度実行しても結果が同じ（重複行が増えない）。

## キャッシュ（陳腐化が「起きる/起きない」の境目）

- **陳腐化が起きる条件 = 自分の操作が“他人のキャッシュ”に影響するか**
  V1 のメッセージは、送信者の操作が**受信者**の会話ビューに影響する。送信者は自分のキャッシュを更新するが
  受信者のキャッシュを無効化しなかった → 受信者が陳腐化を見た（V1 の教材）。
  - つまり V1 でも「自分の操作 → 自分のキャッシュ更新」はやっていた。やらなかったのは**他人の**キャッシュの無効化。
  - 友達リスト（`friends:{owner}`）は **一方向・自己完結**：A の追加/削除は A 自身のリストにしか影響しない。
    自分のキャッシュを無効化すれば済み、**他人のキャッシュが陳腐化する構図が無い**。
  → 「キャッシュ無効化の良し悪し」ではなく、**操作の影響範囲（自分だけか／他人に及ぶか）**が陳腐化の有無を決める、と理解する。
